    Estrutura do Projeto¶

    O Wolfgang segue uma estrutura Go padrão com separação clara entre binários (cmd/), bibliotecas internas (pkg/) e extensões externas (plugins/, sdk/).

    Visão Geral dos Diretórios¶

    wolfgang/
├── cmd/                    # Binários executáveis
│   ├── server/             # Servidor C2 (API, listeners, task dispatch)
│   ├── client/             # CLI do operador
│   ├── implant/            # Implant compilável com config embarcada
│   ├── implant-sim/        # Simulador de implant para testes
│   └── stager/             # Stager para download+execução do implant
├── pkg/                    # Bibliotecas compartilhadas
│   ├── agent/              # Gerenciamento de agents (registro, sessions, keys)
│   ├── api/                # REST API + WebSocket (go-chi router)
│   ├── auth/               # JWT, RBAC, rate limiting, bcrypt
│   ├── config/             # Parsing de configuração YAML
│   ├── crypto/             # X25519 ECDH, AES-256-GCM, HKDF
│   ├── generate/           # Build system do implant (compilação, polimorfismo)
│   ├── module/             # Interface Module + módulos built-in
│   ├── moduleloader/       # Carregamento dinâmico de modules em runtime
│   ├── peparser/           # Parser PE para reflective loading (Windows)
│   ├── plugin/             # Plugin manager (lado server)
│   ├── pluginsdk/          # SDK público para plugins externos (gRPC)
│   ├── storage/            # Persistência SQLite (agents, tasks, logs, exfil)
│   ├── task/               # Task manager (dispatch, status tracking)
│   ├── transport/          # Protocolos de transporte (HTTPS, DNS, WS, DoH, pipe)
│   └── winapi/             # Wrappers para Windows API (syscalls, tokens, memory)
├── plugins/                # Plugins externos de exemplo
│   └── example-smb/        # Listener SMB via pluginsdk
├── sdk/                    # SDK C para modules dinâmicos (DLL/SO)
│   ├── module.h            # Header ABI C-compatible
│   └── example/hello.c     # Module de exemplo em C
├── test/integration/       # Testes E2E (build tag: integration)
├── configs/                # Arquivos de configuração de exemplo
├── scripts/                # Scripts auxiliares
└── docs/                   # Documentação (MkDocs)

    Regras de Boundary¶

    O projeto segue regras estritas sobre quem pode importar quem.

    Regra 1: cmd/ importa pkg/, nunca o contrário¶

    Packages em pkg/ nunca devem importar de cmd/. O fluxo de dependência é unidirecional.

    Regra 2: pkg/transport não importa pkg/api¶

    O transport layer opera abaixo da API. O cmd/server conecta ambos.

    Regra 3: pkg/crypto é um package folha¶

    Zero dependências internas -- apenas standard library. Qualquer package pode importar crypto.

    Regra 4: pkg/module não importa pkg/transport¶

    Modules operam sobre dados in-memory. O transport é responsabilidade do implant main loop em cmd/implant.

    Regra 5: pkg/pluginsdk é independente¶

    O SDK pode ser publicado separadamente. Ele não importa nenhum outro package pkg/ do Wolfgang, apenas dependências externas (go-plugin, grpc, protobuf).

    Regra 6: pkg/winapi é condicional¶

    Código em pkg/winapi usa build tags //go:build windows. Outros packages importam via interfaces ou build tags próprias.

    Convenções Go do Projeto¶

    Convenção Exemplo
    Receivers curtos func (r *Registry) GetModule(...)
    Interfaces pelo consumidor Module definido em pkg/module, implementado em subpackages
    Errors como sentinelas var ErrModuleNotFound = errors.New("module not found")
    Context como 1o param Execute(ctx context.Context, args ModuleArgs)
    Construtores New* NewRegistry(), NewBuilder(), NewExecutor()
    Zero-value funcional ModuleArgs{} é válido (Raw e Parsed ficam vazios)
    Platform filtering Module.OS() retorna slice de GOOS suportados
