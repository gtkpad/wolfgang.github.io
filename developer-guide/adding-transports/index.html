    Criando Transports¶

    Transports são os canais de comunicação entre o implant e o server. O Wolfgang usa um protocolo de envelope binário independente do transport, o que permite adicionar novos canais sem alterar a lógica de negócio.

    Arquitetura do Transport Layer¶

    O transport layer é composto por três partes:

    1. Envelope (pkg/transport/envelope.go) -- formato binário para todas as mensagens
    2. Listener (lado server) -- recebe conexões de agents e despacha tasks
    3. Client (lado implant) -- envia check-ins e recebe tasks do server

    Formato do Envelope¶

    Toda comunicação C2 usa o Envelope definido em pkg/transport/envelope.go:

    ┌─────────────────────────────────────────────────────────┐
│                    Header (26 bytes)                     │
├──────────────┬──────────┬─────────┬─────────────────────┤
│ agent_id     │ msg_type │ seq     │ payload_len         │
│ (16 bytes)   │ (2 bytes)│(4 bytes)│ (4 bytes)           │
├──────────────┴──────────┴─────────┴─────────────────────┤
│              Payload (AES-256-GCM encrypted)             │
│              (payload_len bytes)                          │
└─────────────────────────────────────────────────────────┘

    Tipos de mensagem:

    Valor Constante Direção Descrição
    1 MsgRegistration Implant -> Server Registro inicial do agent
    2 MsgCheckIn Implant -> Server Heartbeat periódico
    3 MsgTaskResponse Implant -> Server Resultado de task executada
    4 MsgTaskDispatch Server -> Implant Tasks pendentes para execução

    O encoding/decoding é feito por transport.Encode() e transport.Decode(), que aplicam AES-256-GCM automaticamente.

    Componentes de um Transport¶

    Lado Server: Listener¶

    O listener precisa:

    1. Aceitar conexões/requisições de agents
    2. Extrair bytes do envelope da requisição (dependente do protocolo)
    3. Chamar transport.Decode() para decriptar
    4. Interagir com AgentManager e TaskManager
    5. Chamar transport.Encode() para a resposta
    6. Embalar os bytes no formato do protocolo

    Lado Implant: Client¶

    O client precisa:

    1. Chamar transport.Encode() com o envelope de check-in
    2. Embalar os bytes no formato do protocolo
    3. Enviar ao server
    4. Extrair bytes do envelope da resposta
    5. Chamar transport.Decode() para decriptar tasks

    Passo 2: Implementar o Listener¶

    Skeleton para um listener:

    package mytransport

import (
    "context"
    "log/slog"

    "github.com/user/wolfgang/pkg/agent"
    "github.com/user/wolfgang/pkg/storage"
    "github.com/user/wolfgang/pkg/transport"
)

type Listener struct {
    name     string
    bindAddr string
    port     int
    agentMgr *agent.Manager
    store    *storage.Store
    // campos específicos do protocolo
}

type ListenerConfig struct {
    Name     string
    BindAddr string
    Port     int
    AgentMgr *agent.Manager
    Store    *storage.Store
    // opções específicas do protocolo
}

func NewListener(cfg ListenerConfig) *Listener {
    return &Listener{
        name:     cfg.Name,
        bindAddr: cfg.BindAddr,
        port:     cfg.Port,
        agentMgr: cfg.AgentMgr,
        store:    cfg.Store,
    }
}

func (l *Listener) Name() string { return l.name }

func (l *Listener) Start() error {
    // 1. Bind no endereço configurado
    // 2. Iniciar accept loop em goroutine
    // 3. Retornar nil se o bind foi bem-sucedido
    return nil
}

func (l *Listener) Stop() error {
    // Graceful shutdown
    return nil
}

func (l *Listener) handleMessage(raw []byte) {
    // 1. Validar tamanho mínimo do envelope
    if len(raw) < transport.HeaderSize {
        slog.Info("envelope too small", "size", len(raw))
        return
    }

    // 2. Extrair agent ID do header (primeiros 16 bytes)
    agentID := hex.EncodeToString(raw[0:transport.AgentIDSize])

    // 3. Obter session key
    sessionKey, err := l.agentMgr.GetSessionKey(agentID)
    if err != nil {
        slog.Info("session key not found", "agent_id", agentID)
        return
    }

    // 4. Verificar replay window
    rw, err := l.agentMgr.GetReplayWindow(agentID)
    if err != nil {
        return
    }

    // 5. Decode do envelope (decriptação automática)
    env, err := transport.Decode(raw, sessionKey)
    if err != nil {
        slog.Info("decode failed", "error", err)
        return
    }

    // 6. Verificar replay
    if !rw.Check(env.Seq) {
        slog.Info("replay check failed", "seq", env.Seq)
        return
    }

    // 7. Processar conforme MsgType
    switch env.MsgType {
    case transport.MsgCheckIn:
        l.handleCheckIn(agentID, sessionKey)
    case transport.MsgTaskResponse:
        l.handleTaskResponse(agentID, env.Payload)
    }
}

func (l *Listener) handleCheckIn(agentID string, sessionKey []byte) {
    tasks, err := l.agentMgr.HandleCheckIn(agentID)
    if err != nil || len(tasks) == 0 {
        return

import (
    "github.com/user/wolfgang/pkg/transport"

func doMyTransportCheckIn(session *agentSession, seq *uint32) ([]byte, error) {
    agentIDBytes := parseAgentID(session.AgentID)

        AgentID: agentIDBytes,
        MsgType: transport.MsgCheckIn,
        Seq:     *seq,

    Transports Existentes como Referência¶

    HTTPS (pkg/transport/http/)¶

    O transport mais completo. Usa malleable profiles para disfarçar o tráfego:

    • Listener: listener.go -- HTTP server com rotas dinâmicas baseadas no profile
    • Profile: profile.go -- YAML que define URIs, headers, body templates
    • Profiles prontos: profiles/ -- google-search, slack, teams, winupdate

    Dados C2 são embarcados em parâmetros HTTP (?q={{data_base64}}) e extraídos via template matching.

    DNS (pkg/transport/dns/)¶

    Usa registros DNS TXT e CNAME para transportar dados:

    • Queries DNS codificam dados em subdomínios
    • Respostas usam registros TXT com dados base64

    WebSocket (pkg/transport/ws/)¶

    Conexão persistente bidirecional:

    • Upgrade HTTP -> WebSocket
    • Frames binários com envelope completo

    Named Pipes (pkg/transport/pipe/)¶

    Para comunicação lateral em redes Windows:

    • Usa \\.\pipe\ no Windows
    • Relay entre agents para pivoting

    Malleable Profiles¶

    Transports HTTP-based podem usar malleable profiles para customizar o tráfego. Os profiles são definidos em YAML:

    name: "custom-profile"
client:
  get:
    uris: ["/api/v2/data", "/feed"]
    headers:
      User-Agent: "CustomApp/1.0"
    params:
      token: "{{data_base64}}"
  post:
    uris: ["/api/v2/submit"]
    headers:
      Content-Type: "application/json"
    body: '{"data":"{{data_base64}}"}'
server:
  headers:
    Content-Type: "application/json"
  body: '{"result":"{{data_base64}}"}'

    O placeholder {{data_base64}} é substituído pelo envelope encodado em base64.

    Diagrama de Fluxo Completo¶

    sequenceDiagram
    participant I as Implant
    participant T as Transport Layer
    participant E as Envelope
    participant L as Listener
    participant AM as Agent Manager
    participant TM as Task Manager

    Note over I: Check-in cycle

    I->>E: Encode(check-in envelope, session_key)
    E-->>I: encrypted bytes
    I->>T: Embalar no formato do protocolo
    T->>L: Enviar via rede

    L->>T: Extrair bytes do protocolo
    T->>E: Decode(raw, session_key)
    E-->>L: Envelope{AgentID, MsgCheckIn, Payload}
    L->>AM: HandleCheckIn(agentID)
    AM-->>L: pending tasks

    alt Tem tasks pendentes
        L->>E: Encode(dispatch envelope, session_key)
        E-->>L: encrypted bytes
        L->>T: Embalar resposta
        T->>I: Resposta via rede
        I->>E: Decode(response, session_key)
        E-->>I: tasks[]
        I->>I: Execute modules
        I->>E: Encode(result envelope, session_key)
        I->>T: Enviar resultado
        T->>L: Receber resultado
        L->>TM: UpdateTaskStatus()
    end
