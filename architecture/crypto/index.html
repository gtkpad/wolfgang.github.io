    Criptografia¶

    O subsistema criptografico do Wolfgang fornece confidencialidade, integridade e autenticacao para toda comunicacao entre server e implants. O design prioriza simplicidade auditavel sobre features exoticas: tres primitivas bem estabelecidas (X25519, HKDF-SHA256, AES-256-GCM) compoem o pipeline completo.

    Visao Geral do Pipeline¶

    ECDH(ephemeral_priv, server_pub)
         |
         v
    shared_secret (32 bytes)
         |
         v
HKDF-SHA256(shared_secret, salt=agent_id, info="wolfgang-session-key")
         |
         v
    session_key (32 bytes)     <-- usada para toda a vida da sessao
         |
         +---> HKDF-SHA256(session_key, salt=seq, info="wolfgang-nonce")
         |              |
         |              v
         |         nonce (12 bytes)
         |              |
         v              v
    AES-256-GCM(session_key, nonce, plaintext)
         |
         v
    ciphertext + auth_tag (16 bytes)

    X25519 ECDH Key Exchange¶

    O Wolfgang usa X25519 (Curve25519) para Elliptic Curve Diffie-Hellman. X25519 foi escolhido sobre NIST P-256 por:

    • Constant-time por design — imune a timing side-channels sem codigo especial
    • Chave de 32 bytes (vs 65 bytes comprimidos para P-256)
    • Performance superior — ~2x mais rapido que P-256 em software
    • Go stdlib — crypto/ecdh tem suporte nativo desde Go 1.20

    Keypairs¶

    O Wolfgang usa dois tipos de keypairs:

    // pkg/crypto/keys.go

// Keypair do SERVER — estatico, persiste entre restarts
// Armazenado em ~/.wolfgang/server.key (permissao 0600)
func GenerateStaticKeypair() (*Keypair, error) {
    return generateKeypair()
}

// Keypair do IMPLANT — efemero, gerado a cada registro
// Descartado imediatamente apos derivar shared secret
func GenerateEphemeralKeypair() (*Keypair, error) {
    return generateKeypair()
}

func generateKeypair() (*Keypair, error) {
    priv, err := ecdh.X25519().GenerateKey(rand.Reader)
    return &Keypair{Private: priv, Public: priv.PublicKey()}, nil
}

    Server keypair e persistido no disco para que implants existentes continuem funcionando apos restart do server:

    func LoadOrGenerateKeypair(path string) (*Keypair, error) {
    if kp, err := LoadKeypair(path); err == nil {
        return kp, nil // carrega existente
    }
    kp, _ := GenerateStaticKeypair()
    os.MkdirAll(filepath.Dir(path), 0700)
    SaveKeypair(kp, path)
    return kp, nil
}

    Server Key Pinning¶

    A chave publica do server e embutida no implant em build time via -ldflags:

    go build -ldflags "-X main.buildServerPublicKey=<hex_encoded_pubkey>" ./cmd/implant

    No registro, o implant verifica que a chave publica recebida do server corresponde a chave pinada:

    // cmd/implant/register.go
if cfg.ServerPublicKey != "" {
    if regResp.ServerPublicKey != cfg.ServerPublicKey {
        return nil, fmt.Errorf("server public key mismatch")
    }
}

    Isso previne MITM mesmo sem PKI/CA. Um atacante precisaria da chave privada do server para se passar por ele.

    Forward Secrecy¶

    A chave privada efemera do implant e gerada no stack da funcao Register(). Quando a funcao retorna, o garbage collector de Go eventualmente sobrescreve a memoria. Mesmo que alguem obtenha a chave privada estatica do server depois, nao consegue decifrar sessoes passadas porque a chave efemera do implant ja nao existe.

    HKDF-SHA256 — Derivacao de Chaves¶

    HKDF (HMAC-based Key Derivation Function) e usado em dois pontos:

    1. Derivacao da Session Key¶

    // pkg/crypto/session.go
func DeriveSessionKey(sharedSecret []byte, agentID []byte) ([]byte, error) {
    // salt = agentID — vincula a chave a esta sessao especifica
    // info = "wolfgang-session-key" — domain separation
    hkdfReader := hkdf.New(sha256.New, sharedSecret, agentID, []byte("wolfgang-session-key"))
    key := make([]byte, 32) // AES-256 = 32 bytes
    io.ReadFull(hkdfReader, key)
    return key, nil
}

    O salt e o agentID (16 bytes aleatorios gerados pelo server). Isso garante que dois implants com a mesma chave efemera (impossivel na pratica, mas criptograficamente correto) produzam session keys diferentes.

    O info e a string "wolfgang-session-key", que faz domain separation — garante que a saida do HKDF e diferente de qualquer outra derivacao que use os mesmos inputs.

    2. Derivacao de Nonce¶

    func DeriveNonce(sessionKey []byte, seq uint32) ([]byte, error) {
    // salt = seq como 4 bytes big-endian
    salt := []byte{
        byte(seq >> 24), byte(seq >> 16), byte(seq >> 8), byte(seq),
    }
    // info = "wolfgang-nonce" — domain separation
    hkdfReader := hkdf.New(sha256.New, sessionKey, salt, []byte("wolfgang-nonce"))
    nonce := make([]byte, 12) // AES-GCM nonce = 12 bytes
    io.ReadFull(hkdfReader, nonce)
    return nonce, nil
}

    Nonce Derivation — Por que nao transmitir?¶

    A decisao mais importante do subsistema cripto: o nonce nunca e transmitido no wire. Ambos os lados derivam o nonce de HKDF(sessionKey, seq).

    Vantagens desta abordagem¶

    1. Economia de 12 bytes por mensagem. Em DNS transport, onde cada byte e precioso (labels de 63 chars), isso e significativo
    2. Replay protection implicita. Se um atacante reenvia uma mensagem com seq=42, o server ja tem seq=42 na replay window e rejeita. Mas mesmo sem a replay window, o AES-GCM falharia para seq diferente do original porque o nonce seria diferente
    3. Sem nonce reutilizacao acidental. A unica forma de reutilizar um nonce e reutilizar um seq com a mesma session key. O seq e monotonicamente crescente, e o implant aborta quando seq > MaxUint32 (exaustao de sequence space)

    Exaustao de Sequence Space¶

    Com uint32, existem ~4.3 bilhoes de mensagens possiveis por sessao. Com check-in a cada 5 segundos e ~2 mensagens por ciclo, isso equivale a ~340 anos. Mesmo assim, o codigo detecta exaustao:

    // cmd/implant/checkin.go
if seq > math.MaxUint32 {
    return fmt.Errorf("crypto: sequence exhausted, session re-key required")
}

    // pkg/crypto/session.go
var ErrSequenceExhausted = errors.New("crypto: sequence number space exhausted, re-key required")

    AES-256-GCM — Criptografia Autenticada¶

    AES-256-GCM fornece confidencialidade (AES-256 em modo CTR) e integridade (Galois MAC de 16 bytes) em uma unica operacao.

    // pkg/crypto/aead.go
func Encrypt(sessionKey []byte, seq uint32, plaintext []byte) ([]byte, error) {
    block, _ := aes.NewCipher(sessionKey)  // AES-256 (key = 32 bytes)
    gcm, _ := cipher.NewGCM(block)          // GCM mode
    nonce, _ := DeriveNonce(sessionKey, seq) // 12 bytes derivados
    ciphertext := gcm.Seal(nil, nonce, plaintext, nil)
    return ciphertext, nil
    // ciphertext = encrypted_data || auth_tag (16 bytes)
}

func Decrypt(sessionKey []byte, seq uint32, ciphertext []byte) ([]byte, error) {
    block, _ := aes.NewCipher(sessionKey)
    gcm, _ := cipher.NewGCM(block)
    nonce, _ := DeriveNonce(sessionKey, seq)
    plaintext, err := gcm.Open(nil, nonce, ciphertext, nil)
    // Se auth tag nao bate: "decryption failed (tampered or wrong key)"
    return plaintext, err
}

    Por que AES-256-GCM e nao ChaCha20-Poly1305?¶

    • Go stdlib tem implementacao AES-GCM acelerada por hardware (AES-NI) em x86/x64
    • ChaCha20 e melhor em ARM sem AES-NI, mas a maioria dos targets C2 sao x86
    • AES-256 e FIPS-approved (relevante para engagements em ambientes regulados)
    • Ambas sao AEAD com seguranca equivalente para este uso

    Rotacao de Session Key¶

    A session key e derivada uma vez durante o registro ECDH e usada para toda a vida da sessao. Nao ha rotacao automatica de session key no Wolfgang atual.

    Por que isso e aceitavel:

    • A session key tem 256 bits de entropia (derivada via HKDF de shared secret + agentID)
    • O nonce e unico por mensagem (derivado do seq monotonicamente crescente)
    • Com 4.3B mensagens possiveis, exaustao e impossivel na pratica
    • Se o server reinicia, as sessions sao restauradas do SQLite (session_key column em agents)

    Se rotacao fosse necessaria, o mecanismo seria: server envia task especial rekey com nova ephemeral pubkey, implant deriva novo shared secret e session key. Isso nao esta implementado porque a analise de risco nao justifica a complexidade adicional.

    Persistencia de Session Keys¶

    Session keys sao armazenadas no SQLite (coluna session_key na tabela agents) em formato hex-encoded. Na inicializacao do server, o Agent Manager restaura todas as sessions:

    // cmd/server/main.go
agentMgr := agent.NewManager(store, keypair)
if n := agentMgr.RestoreFromDB(); n > 0 {
    fmt.Printf("[+] Restored %d agent session(s) from database\n", n)
}

    Isso permite que o server reinicie sem perder as sessoes ativas. Implants que fazem check-in apos o restart continuam funcionando normalmente.

    Resumo das Primitivas¶

    Primitiva Parametros Uso
    X25519 ECDH Curve25519, 32B keys Key exchange (efemero + estatico)
    HKDF-SHA256 SHA-256, 32B output Session key + nonce derivation
    AES-256-GCM 256-bit key, 12B nonce, 16B tag Criptografia autenticada de payloads
    SHA-256 256-bit digest Hash de refresh tokens, verificacao de plugins
    bcrypt Cost default (10) Hash de senhas de operadores
