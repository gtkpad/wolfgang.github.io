    Protocolo¶

    O Wolfgang usa um protocolo binario customizado chamado Envelope. Cada mensagem entre implant e server segue o mesmo formato independente do transport (HTTPS, DNS, WebSocket, Named Pipe). Isso significa que trocar de transport nao exige nenhuma mudanca no payload — apenas o framing externo muda.

    Formato do Envelope¶

    Cada envelope consiste de um header fixo de 26 bytes seguido de um payload cifrado com AES-256-GCM.

    packet-beta
    0-127: "agent_id (16 bytes) — UUID do agent em binario"
    128-143: "msg_type (2 bytes) — Tipo de mensagem, big-endian uint16"
    144-175: "seq (4 bytes) — Sequence number, big-endian uint32"
    176-207: "payload_len (4 bytes) — Tamanho do payload cifrado, big-endian"
    208-463: "encrypted_payload (variavel) — AES-256-GCM ciphertext + auth tag"

    Detalhamento byte a byte¶

    Offset Tamanho Campo Tipo Descricao
    0 16 agent_id [16]byte Identificador unico do agent (128-bit, gerado no registro)
    16 2 msg_type uint16 BE Tipo de mensagem (ver tabela abaixo)
    18 4 seq uint32 BE Sequence number (incrementa a cada mensagem)
    22 4 payload_len uint32 BE Tamanho do payload cifrado em bytes
    26 N payload []byte Ciphertext AES-256-GCM (inclui auth tag de 16 bytes)

    O header e sempre plaintext. Isso e intencional — o agent_id precisa ser lido pelo listener para buscar a session key correta antes de decifrar. O seq e lido para derivar o nonce e verificar replay.

    Estrutura em Go¶

    // pkg/transport/envelope.go
const (
    HeaderSize  = 26 // agent_id(16) + msg_type(2) + seq(4) + payload_len(4)
    AgentIDSize = 16
)

type MsgType uint16

const (
    MsgRegistration MsgType = 1 // Implant -> Server: registro inicial
    MsgCheckIn      MsgType = 2 // Implant -> Server: check-in periodico
    MsgTaskResponse MsgType = 3 // Implant -> Server: resultado de task
    MsgTaskDispatch MsgType = 4 // Server -> Implant: tasks pendentes
)

type Envelope struct {
    AgentID [AgentIDSize]byte
    MsgType MsgType
    Seq     uint32
    Payload []byte // plaintext (antes de Encode) ou ciphertext (no wire)
}

    Tipos de Mensagem¶

    Valor Constante Direcao Descricao
    1 MsgRegistration Implant -> Server Registro inicial com ECDH pubkey + metadados
    2 MsgCheckIn Implant -> Server Check-in periodico (heartbeat)
    3 MsgTaskResponse Implant -> Server Resultado de execucao de task
    4 MsgTaskDispatch Server -> Implant Despacho de tasks pendentes

    O registro (tipo 1) e o unico fluxo que nao usa o formato envelope — ele usa JSON puro sobre HTTPS porque a session key ainda nao existe nesse ponto. Todos os outros tipos usam o formato envelope com criptografia.

    Encode e Decode¶

    Encode (Implant -> Wire)¶

    func Encode(env *Envelope, sessionKey []byte) ([]byte, error) {
    // 1. Cifra o payload com AES-256-GCM
    //    Nonce derivado de HKDF(sessionKey, seq) — nunca transmitido
    encrypted, err := crypto.Encrypt(sessionKey, env.Seq, env.Payload)

    // 2. Monta o buffer: header (26B) + payload cifrado
    buf := make([]byte, HeaderSize+len(encrypted))

    // 3. Escreve header em big-endian
    copy(buf[0:16], env.AgentID[:])
    binary.BigEndian.PutUint16(buf[16:18], uint16(env.MsgType))
    binary.BigEndian.PutUint32(buf[18:22], env.Seq)
    binary.BigEndian.PutUint32(buf[22:26], uint32(len(encrypted)))

    // 4. Copia payload cifrado
    copy(buf[HeaderSize:], encrypted)
    return buf, nil
}

    O ponto critico: o nonce nao e incluido no wire format. Ambos os lados derivam o nonce de HKDF(sessionKey, seq). Isso economiza 12 bytes por mensagem e fornece replay protection implicita — um seq repetido geraria o mesmo nonce, que o AES-GCM rejeitaria (ou seria pego pela replay window).

    Decode (Wire -> Server)¶

    func Decode(data []byte, sessionKey []byte) (*Envelope, error) {
    // 1. Valida tamanho minimo
    if len(data) < HeaderSize { return nil, err }

    // 2. Le header fields
    agentID  := data[0:16]
    msgType  := binary.BigEndian.Uint16(data[16:18])
    seq      := binary.BigEndian.Uint32(data[18:22])
    payloadLen := binary.BigEndian.Uint32(data[22:26])

    // 3. Extrai e decifra payload
    encrypted := data[HeaderSize : HeaderSize+payloadLen]
    plaintext, err := crypto.Decrypt(sessionKey, seq, encrypted)
    // Nonce derivado internamente de (sessionKey, seq)

    return &Envelope{AgentID, MsgType(msgType), seq, plaintext}, nil
}

    Replay Protection¶

    O Wolfgang implementa uma sliding window de 64 mensagens para detectar replays. Isso e critico porque um atacante que captura trafego C2 poderia reenviar um check-in para obter tasks pendentes.

    Implementacao da Replay Window¶

    // pkg/transport/replay.go
const WindowSize = 64

type ReplayWindow struct {
    floor  uint32 // menor seq aceito
    bitmap uint64 // bitmap de 64 bits para seqs acima do floor
}

    A replay window usa um uint64 como bitmap, onde cada bit representa um sequence number relativo ao floor. O algoritmo:

    1. Seq abaixo do floor: rejeitado (muito antigo)
    2. Seq dentro da janela: verifica bit no bitmap. Se ja visto, rejeita
    3. Seq acima da janela: avanca o floor, zera bits obsoletos, aceita
    func (w *ReplayWindow) Check(seq uint32) bool {
    if seq < w.floor {
        return false // abaixo do floor — muito antigo
    }

    offset := seq - w.floor
    if offset >= WindowSize {
        // seq esta alem da janela — avanca
        shift := offset - WindowSize + 1
        w.bitmap >>= shift
        w.floor += shift
        offset = seq - w.floor
    }

    mask := uint64(1) << offset
    if w.bitmap&mask != 0 {
        return false // ja visto
    }

    w.bitmap |= mask
    return true
}

    A replay window e persistivel — pode ser serializada para 12 bytes (floor:4 + bitmap:8) e armazenada no banco para sobreviver restarts do server.

    Por que 64 mensagens?¶

    O tamanho da janela (64) foi escolhido porque:

    • Cabe exatamente em um uint64, permitindo operacoes de bit sem alocacao
    • E suficientemente grande para tolerar reordenacao de pacotes em transports UDP (DNS)
    • E pequeno o suficiente para detectar replays em tempo razoavel
    • A maioria dos agents envia 1-10 mensagens por check-in cycle

    Framing por Transport¶

    O envelope e identico em todos os transports. O que muda e como os bytes do envelope sao transportados:

    HTTPS — Malleable Profile¶

    O envelope e codificado em base64 e embutido em parametros HTTP que imitam trafego legitimo:

    Direcao Metodo Framing
    Check-in (implant -> server) GET /search?q=BASE64 Envelope em query param q
    Task response (implant -> server) POST /complete/search Envelope em body form-encoded q=BASE64
    Task dispatch (server -> implant) Response body Envelope em <body>BASE64</body>

    O profile default imita Google Search:

    # Headers imitam browser real
User-Agent: "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
Accept: "text/html,application/xhtml+xml"
# Response parece pagina HTML
Content-Type: "text/html; charset=UTF-8"
Server: "gws"

    DNS — Base32 Subdomains¶

    Dados sao codificados em base32 nos labels de subdomain DNS:

    Direcao Query Type Formato
    Check-in A query <agent_short>.c2.example.com
    Data upload CNAME query <chunk_id>.<seq>.<base32_data>.<agent_short>.c2.example.com
    Task dispatch TXT record Base32 do envelope cifrado nos TXT records
    Ack sem tasks A record IP 1.0.0.0 (sem tasks) ou 1.0.0.1 (com tasks)

    Chunking: DNS labels tem limite de 63 caracteres e queries de 253. Dados maiores sao divididos em chunks com chunk_id. O bit mais alto do chunk_id marca o chunk final (chunkHighBit = 0x80).

    WebSocket — JSON Frames¶

    Conexao persistente bidirecional com message loop:

    Direcao Type Payload
    Registration register JSON com metadados + pubkey
    Check-in checkin (sem payload — implica pelo type)
    Task dispatch task_dispatch data: []byte (envelope binario)
    Task response task_response data: []byte (envelope binario)
    Keepalive ping/pong (sem payload)
    Ack ack (sem payload — confirmacao)

    WebSocket tem a vantagem de server push: o server pode enviar tasks proativamente sem esperar check-in, via PushTasks().

    Named Pipe — Raw Bytes¶

    Comunicacao local entre implants no mesmo host ou rede:

    Direcao Formato
    Qualquer Raw envelope bytes (sem encoding adicional)

    O Relay le raw bytes do pipe, faz Forward para o upstream, e escreve a resposta de volta. Zero overhead de encoding.

    Seguranca do Protocolo¶

    O que e protegido¶

    • Confidencialidade: Payload cifrado com AES-256-GCM (chave de 256 bits)
    • Integridade: GCM authentication tag (16 bytes) detecta tamper
    • Replay: Sliding window de 64 mensagens + nonce derivado do seq
    • Forward secrecy: Chave efemera do implant descartada apos ECDH

    O que NAO e protegido¶

    • Header plaintext: agent_id, msg_type, seq, payload_len sao visiveis no wire. Isso e necessario para o server buscar a session key antes de decifrar
    • Traffic analysis: O tamanho das mensagens revela atividade (mitigado parcialmente pelo profile HTTP)
    • Metadata DNS: Labels de subdomain sao visiveis para resolvers intermediarios

    Por que nao TLS puro?¶

    O Wolfgang usa TLS para o transport HTTPS, mas o protocolo envelope adiciona uma camada propria de criptografia dentro do TLS. Razoes:

    1. Transport-agnostico: DNS e Named Pipe nao tem TLS
    2. Key pinning real: A chave X25519 do server e embutida no implant em build time. TLS certificate pinning e mais fragil
    3. Replay protection: TLS nao protege contra replay a nivel de aplicacao
    4. Defense in depth: Mesmo se TLS for interceptado (proxy corporativo, MITM), o payload permanece cifrado
