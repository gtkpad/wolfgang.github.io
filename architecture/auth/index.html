    Autenticacao e Autorizacao¶

    O sistema de autenticacao do Wolfgang protege a API do teamserver contra acesso nao autorizado. Opera em multiplas camadas: bcrypt para senhas, JWT para sessoes, refresh token rotation com deteccao de roubo, RBAC para permissoes, rate limiting por IP, e TOFU para validacao de certificados TLS.

    Fluxo de Autenticacao Completo¶

    sequenceDiagram
    participant C as Client CLI
    participant RL as Rate Limiter
    participant API as API Server
    participant Auth as AuthService
    participant JWT as JWTService
    participant DB as SQLite

    C->>API: POST /api/v1/auth/login
    {username, password}
    API->>RL: Allow(remoteIP)?

    alt Rate limited
        RL-->>API: false
        API-->>C: 429 Too Many Requests
    else Allowed
        RL-->>API: true
        API->>Auth: Login(username, password)
        Auth->>DB: GetOperatorByUsername(username)
        DB-->>Auth: Operator{id, password_hash, role}

        Auth->>Auth: bcrypt.Compare(password_hash, password)

        alt Credenciais invalidas
            Auth-->>API: ErrInvalidCredentials
            API->>RL: RecordFailure(remoteIP)
            Note over RL: failures++ — lockout apos N falhas
            API-->>C: 401 Unauthorized
        else Credenciais validas
            API->>RL: RecordSuccess(remoteIP)
            Auth->>JWT: GenerateAccessToken(operatorID, role)
            JWT-->>Auth: accessToken (JWT, 1h TTL)
            Auth->>JWT: GenerateRefreshToken(operatorID)
            JWT-->>Auth: refreshToken (JWT, 24h TTL)
            Auth->>Auth: SHA256(refreshToken)
            Auth->>DB: StoreRefreshToken(operatorID, hash, expiresAt)
            Auth-->>API: TokenPair{access, refresh}
            API-->>C: 200 {access_token, refresh_token}
        end
    end

    Note over C: Usa access_token em todas as requests
    C->>API: GET /api/v1/agents
    Authorization: Bearer {access_token}
    API->>JWT: ValidateAccessToken(token)
    JWT-->>API: Claims{operatorID, role}
    API->>API: RBAC check: role allowed?
    API-->>C: 200 [agents...]

    Note over C: Access token expira apos 1h
    C->>API: POST /api/v1/auth/refresh
    {refresh_token}
    API->>Auth: RefreshTokens(refreshToken)
    Auth->>Auth: tokenHash = SHA256(refreshToken)
    Auth->>DB: GetRefreshToken(tokenHash)

    alt Token revogado (possivel roubo)
        DB-->>Auth: RefreshToken{revoked: true}
        Auth->>DB: RevokeAllTokens(operatorID)
        Note over Auth: DETECCAO DE ROUBO:
    Revoga TODOS os tokens do operador
        Auth-->>API: ErrTokenRevoked
        API-->>C: 401 Token revoked
    else Token valido
        Auth->>DB: RevokeRefreshToken(oldHash)
        Note over Auth: Old token invalidado imediatamente
        Auth->>JWT: GenerateAccessToken + GenerateRefreshToken
        Auth->>DB: StoreRefreshToken(newHash)
        Auth-->>API: new TokenPair
        API-->>C: 200 {new_access_token, new_refresh_token}
    end

    JWT — JSON Web Tokens¶

    O Wolfgang usa JWT com HMAC-SHA256 para tokens de sessao.

    JWTService¶

    // pkg/auth/jwt.go
type JWTService struct {
    secret          []byte        // 32 bytes random, gerado no boot
    accessTokenTTL  time.Duration // 1 hora
    refreshTokenTTL time.Duration // 24 horas
}

func NewJWTService() (*JWTService, error) {
    secret := make([]byte, 32)
    rand.Read(secret) // crypto/rand
    return &JWTService{
        secret:          secret,
        accessTokenTTL:  1 * time.Hour,
        refreshTokenTTL: 24 * time.Hour,
    }, nil
}

    O secret e gerado a cada boot do server. Isso significa que todos os tokens existentes sao automaticamente invalidados quando o server reinicia. Isso e intencional — em um C2, sessoes de operador nao precisam sobreviver restarts.

    Claims¶

    type Claims struct {
    OperatorID string `json:"operator_id"`
    Role       string `json:"role"`
    jwt.RegisteredClaims // ExpiresAt, IssuedAt, ID (JTI), Subject
}

    Cada token inclui um JTI (JWT ID) unico de 16 bytes random para evitar colisoes de hash no storage de refresh tokens.

    Access vs Refresh Token¶

    Aspecto Access Token Refresh Token
    TTL 1 hora 24 horas
    Role claim Role real (admin/operator) "refresh" (fixo)
    Armazenado no DB? Nao Sim (hash SHA-256)
    Uso Authorization header em API calls Endpoint /auth/refresh apenas

    Deteccao de Roubo de Token¶

    O mecanismo de refresh token rotation implementa deteccao de roubo:

    // pkg/auth/service.go
func (s *AuthService) RefreshTokens(refreshToken string) (*TokenPair, error) {
    tokenHash := hashToken(refreshToken)
    stored, _ := s.store.GetRefreshToken(tokenHash)

    // Se o token ja foi revogado e esta sendo reutilizado:
    // ALGUEM ROUBOU o token (o legitimo ja usou e obteve um novo)
    if stored.Revoked {
        s.store.RevokeAllTokens(stored.OperatorID) // nuclear option
        return nil, ErrTokenRevoked
    }

    // Revoga o token atual imediatamente
    s.store.RevokeRefreshToken(tokenHash)

    // Gera novo par
    return s.generateAndStoreTokens(op.ID, op.Role)
}

    Como funciona a deteccao:

    1. Operador A faz login, recebe refresh_1
    2. Atacante rouba refresh_1
    3. Operador A usa refresh_1 para renovar -> recebe refresh_2, refresh_1 e revogado
    4. Atacante tenta usar refresh_1 -> detecta que esta revogado -> revoga TODOS os tokens do operador
    5. Operador A precisa fazer login novamente, mas o atacante esta bloqueado

    O hash do refresh token e armazenado (nunca o token raw):

    func hashToken(token string) string {
    h := sha256.Sum256([]byte(token))
    return hex.EncodeToString(h[:])
}

    RBAC — Role-Based Access Control¶

    flowchart TD
    REQ["HTTP Request"] --> MW["Auth Middleware"]
    MW --> JWT_V{"JWT Valido?"}
    JWT_V -->|"Nao"| R401["401 Unauthorized"]
    JWT_V -->|"Sim"| CLAIMS["Extrai Claims
    {operatorID, role}"]
    CLAIMS --> CTX["Claims no Context"]
    CTX --> RBAC{"RequireRole(roles...)"}

    RBAC --> IS_ADMIN{"role == admin?"}
    IS_ADMIN -->|"Sim"| ALLOW["200 Permitido
    Admin tem acesso total"]
    IS_ADMIN -->|"Nao"| ROLE_CHECK{"role in allowed?"}
    ROLE_CHECK -->|"Sim"| ALLOW
    ROLE_CHECK -->|"Nao"| R403["403 Forbidden"]

    Roles¶

    Role Descricao Permissoes
    admin Administrador completo Tudo: CRUD operadores, agents, tasks, listeners, plugins, config
    operator Operador padrao Gerenciar agents, criar tasks, visualizar resultados
    viewer Somente leitura Visualizar agents, tasks e resultados

    Admin sempre tem acesso, independente da lista de roles configurada:

    // pkg/auth/rbac.go
func RequireRole(roles ...string) func(http.Handler) http.Handler {
    allowed := make(map[string]bool, len(roles))
    for _, r := range roles { allowed[r] = true }
    allowed[RoleAdmin] = true // admin sempre permitido

    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            claims := ClaimsFromContext(r.Context())
            if claims == nil {
                http.Error(w, "Forbidden: no credentials", 403)
                return
            }
            if !allowed[claims.Role] {
                http.Error(w, "Forbidden: insufficient role", 403)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}

    Claims no Context¶

    Claims sao propagadas via context.Context de Go, evitando globals:

    func ContextWithClaims(ctx context.Context, claims *Claims) context.Context {
    return context.WithValue(ctx, claimsKey, claims)
}

func ClaimsFromContext(ctx context.Context) *Claims {
    claims, _ := ctx.Value(claimsKey).(*Claims)
    return claims
}

    Rate Limiting e Lockout¶

    O rate limiter protege contra brute force no endpoint de login:

    // pkg/auth/ratelimit.go
type RateLimitConfig struct {
    MaxAttempts int           // max tentativas por IP na janela
    Window      time.Duration // duracao da janela
    LockoutMax  int           // falhas consecutivas para lockout
    LockoutDur  time.Duration // duracao do lockout
}

type RateLimiter struct {
    config RateLimitConfig
    ips    map[string]*ipRecord
    now    func() time.Time // injectable para testes
}

    Comportamento¶

    1. Rate limit por janela: Cada IP pode fazer no maximo MaxAttempts tentativas dentro de Window
    2. Lockout por falhas consecutivas: Apos LockoutMax falhas consecutivas, o IP e bloqueado por LockoutDur
    3. Reset em sucesso: Login bem sucedido reseta o contador de falhas
    func (rl *RateLimiter) Allow(ip string) bool {
    rec := rl.getOrCreate(ip)

    // Lockout ativo?
    if !rec.lockedUntil.IsZero() && now.Before(rec.lockedUntil) {
        return false
    }

    // Prune tentativas fora da janela
    pruned := rec.attempts[:0]
    for _, t := range rec.attempts {
        if t.After(cutoff) { pruned = append(pruned, t) }
    }
    rec.attempts = pruned

    // Verificar limite
    if len(rec.attempts) >= rl.config.MaxAttempts {
        return false
    }

    rec.attempts = append(rec.attempts, now)
    return true
}

func (rl *RateLimiter) RecordFailure(ip string) {
    rec := rl.getOrCreate(ip)
    rec.failures++
    if rec.failures >= rl.config.LockoutMax {
        rec.lockedUntil = rl.now().Add(rl.config.LockoutDur)
    }
}

    O rate limiter tambem funciona como HTTP middleware para proteger qualquer endpoint:

    func (rl *RateLimiter) Middleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if !rl.Allow(r.RemoteAddr) {
            http.Error(w, "Too Many Requests", 429)
            return
        }
        next.ServeHTTP(w, r)
    })
}

    IP Allowlist¶

    O server suporta uma allowlist de IPs que podem acessar a API. Configurada via config.yaml:

    auth:
  ip_allowlist:
    - "10.0.0.0/24"
    - "192.168.1.100"

    Requests de IPs fora da allowlist sao rejeitados antes de qualquer autenticacao.

    TOFU — Trust On First Use¶

    O client CLI implementa TOFU para validar certificados TLS do teamserver, similar ao modelo do SSH (known_hosts).

    // cmd/client/tofu.go
type tofuStore struct {
    path string // ~/.wolfgang/known_servers
}

    Fluxo¶

    1. Primeira conexao: Client exibe fingerprint SHA-256 do certificado e armazena em ~/.wolfgang/known_servers
    2. Conexoes subsequentes: Client verifica que o fingerprint corresponde ao armazenado
    3. Mismatch: Aborta com aviso de possivel MITM
    func (t *tofuStore) verify(host string, state tls.ConnectionState) error {
    fp := certFingerprint(state)
    stored, _ := t.lookup(host)

    if stored == "" {
        // Primeira conexao — pin
        t.pin(host, fp)
        return nil
    }

    if stored != fp {
        return fmt.Errorf(
            "certificate mismatch for %s!\n"+
            "Expected: %s\nGot: %s\n"+
            "This may indicate a MITM attack.",
            host, stored, fp)
    }
    return nil
}

    O formato do arquivo known_servers:

    10.0.0.1:8443 a1b2c3d4e5f6...
192.168.1.100:8443 f0e1d2c3b4a5...

    TLS — Modos de Configuracao¶

    O teamserver suporta tres modos TLS:

    Auto (Self-signed)¶

    // pkg/api/tls.go
func setupAutoTLS() (*tls.Config, string, error) {
    // Gera CA + server cert em ~/.wolfgang/tls/
    // CA: ECDSA P-256, 10 anos
    // Server: ECDSA P-256, 1 ano, assinado pela CA
}

    Custom (User-provided)¶

    server:
  tls:
    mode: "custom"
    cert_file: "/path/to/cert.pem"
    key_file: "/path/to/key.pem"

    mTLS (Mutual TLS)¶

    server:
  tls:
    mode: "mtls"
    cert_file: "/path/to/server-cert.pem"
    key_file: "/path/to/server-key.pem"
    ca_file: "/path/to/client-ca.pem"

    Em modo mTLS, o server exige certificado de cliente assinado pela CA especificada:

    func setupMTLS(cfg config.TLSConfig) (*tls.Config, string, error) {
    tlsCfg := &tls.Config{
        Certificates: []tls.Certificate{cert},
        ClientAuth:   tls.RequireAndVerifyClientCert,
        ClientCAs:    caPool,
        MinVersion:   tls.VersionTLS12,
    }
}

    Bcrypt — Hashing de Senhas¶

    Senhas de operadores sao armazenadas com bcrypt (cost default = 10):

    func HashPassword(password string) (string, error) {
    hash, _ := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)
    return string(hash), nil
}

    Bcrypt e deliberadamente lento (~100ms por hash com cost 10), tornando brute force impraticavel mesmo se o banco de dados for comprometido.
