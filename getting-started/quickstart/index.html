    Quickstart¶

    Este guia cobre a compilacao do projeto, inicializacao do servidor, e conexao do client. Ao final, voce tera o teamserver rodando e um operador autenticado.

    1. Clonar e Compilar¶

    git clone https://github.com/user/wolfgang.git
cd wolfgang
make build

    O comando make build executa go build ./..., compilando os tres binarios:

    • wolfgang-server -- teamserver
    • wolfgang-client -- operator CLI
    • wolfgang-implant -- agent (normalmente gerado via generate, nao usado diretamente)

    Para instalar os binarios no $GOPATH/bin:

    go install ./cmd/server
go install ./cmd/client

    Verificando a compilacao

    Confirme que os binarios foram gerados: 

    ./wolfgang-server --help
./wolfgang-client --help

    2. Inicializar o Servidor¶

    O comando init cria toda a infraestrutura necessaria para o teamserver:

    wolfgang-server init

    O processo interativo vai pedir:

    [+] Directory ready: /home/user/.wolfgang

Admin username: admin
Admin password: ********

[+] Initialization complete. Run 'wolfgang-server start' to begin.

    O que o init cria¶

    O comando inicializa o diretorio ~/.wolfgang/ com a seguinte estrutura:

    ~/.wolfgang/
  config.yaml      # Configuracao do servidor
  wolfgang.db       # Banco SQLite (operators, agents, tasks, logs)
  server.key        # Chave X25519 persistente do servidor
  tls/              # Certificados TLS auto-gerados
    cert.pem
    key.pem

    Sequencia de Boot¶

    flowchart TD
    INIT["wolfgang-server init"] --> DIR["Cria ~/.wolfgang/"]
    DIR --> CFG["Gera config.yaml
    (bind: 0.0.0.0, port: 8443)"]
    CFG --> TLS["Gera certificados TLS
    (modo auto: self-signed)"]
    TLS --> DB["Inicializa SQLite
    (schema migrations)"]
    DB --> KEY["Gera keypair X25519
    (server.key)"]
    KEY --> OP["Cria operador admin
    (bcrypt hash)"]
    OP --> DONE["Pronto para start"]

    style INIT fill:#0f3460,stroke:#e94560,color:#fff
    style DONE fill:#0f3460,stroke:#16213e,color:#fff

    Configuracao Padrao¶

    O arquivo config.yaml gerado usa valores padrao seguros:

    server:
  bind: "0.0.0.0"
  port: 8443
  tls:
    mode: "auto"

auth:
  jwt_ttl: 1h0m0s
  refresh_ttl: 24h0m0s
  rate_limit: 5
  lockout_max: 10
  lockout_dur: 15m0s

storage:
  path: "/home/user/.wolfgang/wolfgang.db"

plugins:
  dir: "plugins"
  verify_hash: true

listeners:
  - type: https
    bind_addr: "0.0.0.0"
    port: 443

implant:
  sleep: 1m0s
  jitter: 10
    Parametro Descricao
    server.port Porta da API REST do teamserver (8443)
    server.tls.mode auto (self-signed), custom (seus certs), ou mtls (mutual TLS)
    auth.rate_limit Max tentativas de login por janela de tempo
    auth.lockout_max Tentativas antes de lockout
    auth.lockout_dur Duracao do lockout (15 minutos)
    listeners[].port Porta do listener para implants (443)
    implant.sleep Intervalo padrao de check-in (60s)
    implant.jitter Variacao aleatoria no sleep (10%)

    3. Iniciar o Servidor¶

    wolfgang-server start

    Voce pode especificar um config alternativo com --config:

    wolfgang-server start --config /path/to/config.yaml

    A saida do servidor mostra cada etapa da inicializacao:

    [+] Config loaded: /home/user/.wolfgang/config.yaml
[+] Database opened: /home/user/.wolfgang/wolfgang.db
[+] TLS ready, fingerprint: a1b2c3d4e5f6...
[+] Server public key: 0123456789abcdef...
[+] Agent manager ready
[+] Implant builder ready, source: /home/user/wolfgang
[+] Task manager ready
[+] Listener configured: https on 0.0.0.0:443
[+] Wolfgang teamserver starting on https://0.0.0.0:8443

    O que acontece no start¶

    1. Config: Carrega config.yaml e merge com defaults
    2. Database: Abre SQLite em WAL mode, executa migrations pendentes, limpa staging tokens expirados
    3. TLS: Carrega ou gera certificados, calcula fingerprint
    4. Auth: Inicializa JWT service e rate limiter
    5. Crypto: Carrega keypair X25519 persistente (key pinning para implants)
    6. Agent Manager: Restaura sessoes de agents do banco de dados
    7. Plugins: Escaneia diretorio de plugins, verifica hashes
    8. Task Manager: Inicia goroutine de expiracao de tasks (TTL: 1 hora, sweep a cada 1 minuto)
    9. Listeners: Auto-inicia listeners configurados no YAML
    10. API Router: Monta rotas REST, Swagger, WebSocket hub
    11. HTTPS Server: Sobe na porta 8443 com TLS

    Portas privilegiadas

    Se o listener usa porta 443 (padrao), o servidor precisa de privilegios de root ou CAP_NET_BIND_SERVICE no Linux: 

    sudo setcap 'cap_net_bind_service=+ep' $(which wolfgang-server)

    4. Conectar o Client¶

    Em outro terminal, conecte-se ao teamserver:

    wolfgang-client connect -s https://10.0.0.1:8443 -u admin -p 'senha'

    Ou execute sem flags para o modo interativo:

    TOFU (Trust On First Use)¶

    Na primeira conexao, o client valida o fingerprint TLS do servidor. Se aceito, o fingerprint e salvo localmente em ~/.wolfgang/tofu.json e verificado em conexoes futuras. Isso impede ataques MITM mesmo com certificados self-signed.

    Comandos Disponiveis no Root¶

    Apos conectar, voce esta no contexto root do REPL. Os comandos disponiveis:

    Comando Descricao
    listeners Listar listeners ativos
    listeners create Criar novo listener interativamente
    listeners stop <id> Parar um listener
    agents Listar agents conectados
    use <agent_id> Selecionar agent para interagir
    operators Listar operadores
    operators create Criar novo operador
    generate Gerar implant interativamente
    plugins Listar plugins carregados
    modules Listar modulos disponiveis no servidor
    logs [n] Ver ultimas N entradas do audit log
    help Mostrar ajuda
    exit Sair

    5. Criar um Operador Adicional¶

    No REPL do client:

    wolfgang >> operators create
Username: pentester
Password: ********
Role (admin/operator/viewer): operator
[+] Operator created: a1b2c3d4 (pentester, role=operator)

    Os roles disponiveis sao:

    Role Permissoes
    admin Acesso total: criar operadores, gerenciar listeners, todas as operacoes
    operator Operacoes com agents, tasks, e generate (sem gerenciamento de operadores)
    viewer Somente leitura: listar agents, tasks, e logs

    Verificacao¶

    Para confirmar que tudo esta funcionando:

    wolfgang >> listeners
ID        TYPE   BIND      PORT  PROFILE  STATUS
a1b2c3d4  https  0.0.0.0   443   default  running

wolfgang >> operators
ID        USERNAME   ROLE     CREATED
a1b2c3d4  admin      admin    2026-03-14 10:00:00
b2c3d4e5  pentester  operator 2026-03-14 10:05:00

wolfgang >> agents
No agents connected.

    O listener HTTPS esta rodando e aguardando conexoes de implants. Na proxima etapa, vamos gerar um implant e registrar o primeiro agent.

    Troubleshooting¶

    Problema Solucao
    address already in use Outra instancia ou servico usando a porta. Mude server.port ou listeners[].port no config
    permission denied (porta 443) Use sudo ou setcap para portas privilegiadas
    TLS handshake error no client Certificado mudou -- delete ~/.wolfgang/tofu.json para re-pin
    authentication failed Verifique username/password. Apos 10 tentativas, lockout de 15 minutos
    WebSocket connection failed Nao-critico -- o client funciona sem WebSocket (sem eventos real-time)
