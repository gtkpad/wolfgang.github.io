    Port Forwarding¶

    O que é¶

    Port forwarding (encaminhamento de porta) cria um relay TCP entre uma porta local no agente e um host:port remoto na rede do alvo. É o building block mais fundamental de network pivoting: o operador se conecta a uma porta local no implant, e o tráfego é transparentemente redirecionado para o destino.

    Diferente do SOCKS proxy (que permite roteamento dinâmico para qualquer destino), port forwarding é estático: cada regra mapeia exatamente uma porta local para exatamente um destino remoto.

    Por que funciona¶

    Em redes segmentadas, o operador frequentemente não tem conectividade direta com alvos internos. O implant comprometido, por estar dentro da rede, tem acesso que o operador não tem.

    O port forwarding resolve isso tornando o implant um relay:

    Operador → [internet] → Implant:4444 → [rede interna] → Alvo:3389

    O operador conecta em implant:4444 e o tráfego é encaminhado para alvo:3389 (RDP, neste exemplo). Do ponto de vista do alvo, a conexão vem do implant -- não do operador.

    Casos de uso comuns:

    • RDP (3389): Acesso gráfico a servidores internos
    • SMB (445): Acesso a file shares
    • SSH (22): Acesso a servidores Linux internos
    • Databases (1433, 3306, 5432): Acesso a bancos de dados
    • Web admin (8080, 8443): Painéis de administração internos

    Implementação no Wolfgang¶

    O módulo lateral.portfwd (pkg/module/lateral/portfwd.go) é stateful: mantém uma coleção de forwards ativos com lifecycle de start/stop/list.

    Estrutura principal¶

    type PortFwdModule struct {
    mu        sync.Mutex
    forwards  map[string]*forwardState
    connGroup sync.WaitGroup
}

type forwardState struct {
    listener net.Listener
    target   string
    stopCh   chan struct{}
    conns    []net.Conn
    connMu   sync.Mutex
}

    Start: Criar um forward¶

    func (m *PortFwdModule) start(args module.ModuleArgs) (module.ModuleResult, error) {
    target := args.Parsed["target"]     // host:port destino
    listenPort := args.Parsed["listen_port"]
    listenAddr := defaultListenAddress  // 127.0.0.1

    bindAddr := net.JoinHostPort(listenAddr, listenPort)

    ln, err := net.Listen("tcp", bindAddr)
    // ...

    state := &forwardState{
        listener: ln,
        target:   target,
        stopCh:   make(chan struct{}),
    }
    m.forwards[actualAddr] = state

    go m.acceptLoop(state, actualAddr)
    // ...
}

    Proxy bidirecional¶

    Para cada conexão aceita, o módulo abre uma conexão para o target e copia dados em ambas as direções simultaneamente:

    func (m *PortFwdModule) proxyConn(state *forwardState, clientConn net.Conn) {
    defer clientConn.Close()

    targetConn, err := net.DialTimeout("tcp", state.target, defaultDialTimeout)
    if err != nil {
        return
    }
    defer targetConn.Close()

    done := make(chan struct{}, 2)

    // client -> target
    go func() {
        io.Copy(targetConn, clientConn)
        if tc, ok := targetConn.(*net.TCPConn); ok {
            tc.CloseWrite()
        }
        done <- struct{}{}
    }()

    // target -> client
    go func() {
        io.Copy(clientConn, targetConn)
        if cc, ok := clientConn.(*net.TCPConn); ok {
            cc.CloseWrite()
        }
        done <- struct{}{}
    }()

    // Espera ambas as direções ou sinal de stop
    select {
    case <-state.stopCh:
        return
    case <-done:
        select {
        case <-state.stopCh:
        case <-done:
        }
    }
}

    Detalhes de implementação¶

    • Half-close: Quando uma direção termina, CloseWrite() sinaliza fim de dados sem fechar a conexão inteira. Isso permite que o lado remoto termine de enviar.
    • Stop channel: state.stopCh permite shutdown graceful de todas as conexões quando o forward é parado.
    • Connection tracking: Todas as conexões ativas são rastreadas em state.conns para cleanup no stop.

    Ação Descrição Parâmetros

    stop Para um forward id ou listen_port

    Considerações Operacionais¶

    Endereço de escuta¶

    Por padrão, o forward escuta em 127.0.0.1 (apenas conexões locais). Isso é intencional: o forward é acessado via C2 channel, não diretamente pela rede.

    Para cenários onde outros implants na rede precisam acessar o forward, o operador pode especificar listen_addr=0.0.0.0.

    Múltiplos forwards¶

    O módulo suporta múltiplos forwards simultâneos. Cada um é independente e pode ser parado individualmente:

    portfwd start listen_port=4444 target=dc01:445
portfwd start listen_port=5555 target=db01:1433
portfwd start listen_port=6666 target=web01:8443
portfwd list
portfwd stop listen_port=4444

    Shutdown¶

    StopAll() encerra todos os forwards ativos e é chamado durante o shutdown do implant para cleanup graceful.

    Detecção¶

    IOC Descrição
    Processo com múltiplos listeners TCP Implant abrindo portas locais
    Conexões TCP do implant para hosts internos Padrão de relay/proxy
    Tráfego TCP bidirecional de longa duração Sessões interativas (RDP, SSH) via relay
    Netstat/ss mostrando LISTEN em portas incomuns Portas locais abertas pelo implant

    Mitigações:

    • Network segmentation: Microsegmentação limita o alcance do pivoting
    • Host-based firewall: Bloquear portas de escuta não-autorizadas
    • NTA (Network Traffic Analysis): Detectar padrões de relay em tráfego interno
