    Windows Registry Run Keys¶

    O que é¶

    Run keys são entradas no registro do Windows que especificam programas a serem executados automaticamente durante o logon do usuário. São o mecanismo de auto-start mais antigo e mais direto do Windows, usado legitimamente por aplicações como antivírus, drivers e utilities de sistema.

    O Wolfgang usa Run keys para garantir que o implant seja re-executado após cada reboot ou logon, sem necessidade de interação do usuário.

    Por que funciona¶

    O Windows processa Run keys como parte do fluxo de inicialização do shell (explorer.exe). Quando um usuário faz logon:

    1. winlogon.exe autentica o usuário
    2. userinit.exe inicializa o perfil
    3. O shell (explorer.exe) é carregado
    4. Explorer enumera todas as entradas em Run e RunOnce e as executa

    O processamento é sequencial e incondicional -- não há validação de assinatura ou whitelist. Qualquer executável referenciado será iniciado.

    Existem quatro localizações principais:

    Chave Escopo Privilégio
    HKCU\Software\Microsoft\Windows\CurrentVersion\Run Usuário atual User
    HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce Usuário atual (uma vez) User
    HKLM\Software\Microsoft\Windows\CurrentVersion\Run Todos os usuários Admin
    HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnce Todos os usuários (uma vez) Admin

    O Wolfgang usa HKCU\...\Run por padrão, que não requer privilégio elevado.

    Implementação no Wolfgang¶

    O módulo de persistência por registry (pkg/module/persist/registry_windows.go) é direto -- uma única chamada de API:

    func platformWriteRunKey(name, exePath string) error {
    key, _, err := registry.CreateKey(
        registry.CURRENT_USER,
        `Software\Microsoft\Windows\CurrentVersion\Run`,
        registry.SET_VALUE,
    )
    if err != nil {
        return fmt.Errorf("open run key: %w", err)
    }
    defer key.Close()

    if err := key.SetStringValue(name, exePath); err != nil {
        return fmt.Errorf("set run value %q: %w", name, err)
    }
    return nil
}

    Parâmetros¶

    • name: Nome do valor no registro. O Wolfgang usa nomes que se misturam com software legítimo (ex: WindowsUpdateCheck, OneDriveSync).
    • exePath: Caminho completo do implant no disco.

    Exemplo de entrada resultante¶

    [HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run]
"WindowsUpdateCheck"="C:\\Users\\user\\AppData\\Local\\Temp\\updater.exe"

    Considerações Operacionais¶

    Vantagens¶

    • Simplicidade: Uma linha de código. Sem dependências externas.
    • Sem privilégio elevado: HKCU é gravável por qualquer usuário.
    • Confiabilidade: Funciona em todas as versões do Windows (XP+).
    • Reversibilidade: Remover a entrada desativa a persistência.

    Desvantagens¶

    • Alta visibilidade: Ferramentas como Autoruns, Sysmon e EDRs monitoram Run keys ativamente.
    • Vinculada ao usuário: HKCU só funciona para o usuário que criou a entrada. Se outro usuário fizer logon, o implant não inicia.
    • Requer binário em disco: O caminho precisa apontar para um executável persistente no filesystem.

    Comparação com Scheduled Tasks¶

    Aspecto Registry Run Key Scheduled Task
    Privilégio mínimo User Admin (para SYSTEM)
    Trigger Logon apenas Logon, horário, evento
    Persistência cross-user Apenas com HKLM Sim
    Complexidade Mínima Moderada
    Detecção Alta Alta

    Detecção¶

    IOC Descrição
    Sysmon Event ID 13 (Registry Value Set) Criação/modificação de valores em Run keys
    Sysmon Event ID 12 (Registry Object) Criação de chaves em Run
    Autoruns entries de publishers desconhecidos Executáveis não assinados em Run keys
    Registry value apontando para %TEMP% ou %APPDATA% Caminho incomum para auto-start

    Mitigações:

    • AppLocker / WDAC: Restringir quais executáveis podem rodar, independente de Run keys
    • Monitoramento de Run keys: Alertar sobre novas entradas em HKCU e HKLM Run
    • Assinatura de código obrigatória: Impedir execução de binários não assinados
