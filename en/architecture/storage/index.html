    Armazenamento¶

    O Wolfgang usa SQLite como unico backend de persistencia. Toda informacao do teamserver — operadores, agents, tasks, listeners, tokens, exfiltracoes, logs e modulos — vive em um unico arquivo .db.

    Configuracao do SQLite¶

    A inicializacao do banco aplica duas PRAGMAs criticas antes de qualquer operacao:

    // pkg/storage/storage.go
func New(dbPath string) (*Store, error) {
    db, _ := sql.Open("sqlite", dbPath)

    // WAL mode: leituras concorrentes sem bloquear escritas
    db.Exec("PRAGMA journal_mode=WAL")

    // Foreign keys: integridade referencial entre tabelas
    db.Exec("PRAGMA foreign_keys=ON")

    s := &Store{db: db}
    s.migrate()
    return s, nil
}

    WAL Mode (Write-Ahead Logging)¶

    No modo WAL, escritas vao para um arquivo separado (*.db-wal) e leituras continuam no banco principal. Isso significa:

    • Multiplas goroutines podem ler simultaneamente enquanto uma escreve
    • Leituras nunca bloqueiam escritas e vice-versa
    • Performance muito melhor que o default (rollback journal) para o padrao de uso do C2 (muitas leituras, poucas escritas)

    Foreign Keys¶

    Foreign keys estao habilitadas para garantir integridade referencial:

    • Tasks referenciam agents(id) e operators(id)
    • Exfil chunks referenciam exfiltrations(id)
    • Refresh tokens referenciam operators(id)

    Tentativas de criar tasks para agents inexistentes falham com erro de FK.

    Tabelas em Detalhe¶

    operators¶

    Armazena operadores (usuarios humanos) que interagem com o teamserver.

    Coluna Tipo Descricao
    id TEXT PK UUID hex de 32 chars (16 bytes random)
    username TEXT UNIQUE Nome de usuario (unico)
    password_hash TEXT Hash bcrypt da senha
    role TEXT "admin", "operator" ou "viewer"
    created_at DATETIME Timestamp de criacao

    agents¶

    Representa implants registrados com o teamserver.

    Coluna Tipo Descricao
    id TEXT PK UUID hex de 32 chars
    hostname TEXT Hostname do target
    os TEXT "windows", "linux", "darwin"
    arch TEXT "amd64", "arm64", "386"
    username TEXT Usuario do processo do implant
    internal_ip TEXT IP interno do target
    external_ip TEXT IP externo do target
    last_seen DATETIME Ultimo check-in bem sucedido
    sleep INTEGER Intervalo de check-in (ms)
    jitter INTEGER Porcentagem de jitter (0-100)
    transport TEXT Transport primario ("https", "dns", "ws")
    status TEXT "active", "dead", "disconnected"
    session_key TEXT Session key hex-encoded (persistida para sobreviver restart)
    tags TEXT JSON array de tags (ex: '["dc","prod"]')

    tasks¶

    Cada task e uma instrucao enviada a um agent.

    Coluna Tipo Descricao
    id TEXT PK UUID hex
    agent_id TEXT FK Referencia agents(id)
    operator_id TEXT FK Referencia operators(id) (quem criou)
    module TEXT Nome do modulo a executar
    args TEXT Argumentos em JSON string
    status TEXT "pending", "completed", "failed", "expired", "cancelled"
    result TEXT Output retornado pelo modulo
    created_at DATETIME Quando a task foi criada
    completed_at DATETIME Quando o resultado chegou (NULL se pendente)

    Lifecycle de status:

    pending -> dispatched (agent buscou no check-in)
        -> completed  (resultado recebido)
        -> failed     (modulo retornou erro)
        -> expired    (Task Manager expirou por timeout)
        -> cancelled  (operador cancelou)

    listeners¶

    Configuracao de transport listeners.

    Coluna Tipo Descricao
    id TEXT PK UUID hex
    type TEXT "https", "dns", "ws", ou plugin type
    bind_addr TEXT Endereco de bind (ex: "0.0.0.0")
    port INTEGER Porta de escuta
    profile TEXT Nome do malleable profile (HTTPS only)
    status TEXT "running", "stopped"

    logs (Audit Log)¶

    Log de auditoria de todas as acoes.

    Coluna Tipo Descricao
    id INTEGER PK Auto-increment
    operator_id TEXT ID do operador (ou "system")
    action TEXT Tipo de acao (ex: "agent.register", "shell.end", "plugin.load")
    details TEXT JSON com detalhes da acao
    timestamp DATETIME Timestamp do evento

    refresh_tokens¶

    Tokens de refresh para rotacao de JWT.

    Coluna Tipo Descricao
    token_hash TEXT PK SHA-256 hex do token (nunca o token raw)
    operator_id TEXT FK Referencia operators(id)
    expires_at DATETIME Expiracao do token
    revoked INTEGER 0 = ativo, 1 = revogado
    created_at DATETIME Quando foi gerado

    staging_tokens¶

    Tokens one-time para entrega de staged payloads.

    Coluna Tipo Descricao
    token TEXT PK Token unico (URL-safe)
    encrypted_payload BLOB Payload do implant cifrado com AES-256-GCM
    created_at DATETIME Timestamp de criacao
    used INTEGER 0 = nao usado, 1 = consumido

    Tokens expiram apos 24 horas e sao limpos na inicializacao do server:

    store.DeleteExpiredStagingTokens(24 * time.Hour)

    exfiltrations + exfil_chunks¶

    Exfiltracao chunked de arquivos grandes.

    exfiltrations (sessao de exfil):

    Coluna Tipo Descricao
    id TEXT PK UUID hex da sessao
    agent_id TEXT FK Agent que originou
    filename TEXT Nome do arquivo original
    total_chunks INTEGER Total de chunks esperados
    received_chunks INTEGER Chunks recebidos ate agora
    total_bytes INTEGER Tamanho total do arquivo
    sha256_expected TEXT Hash esperado (enviado pelo agent)
    sha256_actual TEXT Hash calculado apos reassembly
    status TEXT "in_progress", "completed"

    exfil_chunks (dados de cada chunk):

    Coluna Tipo Descricao
    session_id TEXT FK Referencia exfiltrations(id)
    chunk_index INTEGER Indice ordenado do chunk
    data BLOB Dados binarios do chunk
    received_at DATETIME Timestamp de recebimento

    PK composta: (session_id, chunk_index) — garante que cada chunk e unico por sessao.

    A insercao de chunk e feita em transacao para manter received_chunks consistente:

    func (s *Store) AddExfilChunk(sessionID string, chunkIndex int, data []byte) error {
    tx, _ := s.db.Begin()
    defer tx.Rollback()

    tx.Exec("INSERT INTO exfil_chunks (session_id, chunk_index, data) VALUES (?, ?, ?)", ...)
    tx.Exec("UPDATE exfiltrations SET received_chunks = received_chunks + 1 WHERE id = ?", ...)

    return tx.Commit()
}

    modules¶

    Modulos compilados para dynamic loading.

    Coluna Tipo Descricao
    name TEXT Nome do modulo
    data BLOB Binario compilado
    sha256 TEXT Hash SHA-256 do binario
    version TEXT Versao do modulo
    platform TEXT "windows", "linux", "darwin"
    arch TEXT "amd64", "arm64"

    PK composta: (name, platform, arch) — permite multiplas compilacoes do mesmo modulo.

    schema_version¶

    Controle de versao do schema para migracao incremental.

    Coluna Tipo Descricao
    version INTEGER PK Numero da migracao aplicada

    Sistema de Migrations¶

    O Wolfgang usa um sistema de migracoes sequenciais baseado em DDL statements:

    // pkg/storage/storage.go
var migrations = []string{
    // M1: Operators
    `CREATE TABLE IF NOT EXISTS operators (...)`,
    // M2: Agents
    `CREATE TABLE IF NOT EXISTS agents (...)`,
    // M3: Tasks
    `CREATE TABLE IF NOT EXISTS tasks (...)`,
    // M4: Listeners
    `CREATE TABLE IF NOT EXISTS listeners (...)`,
    // M5: Logs
    `CREATE TABLE IF NOT EXISTS logs (...)`,
    // M6: Refresh tokens
    `CREATE TABLE IF NOT EXISTS refresh_tokens (...)`,
    // M7: Add session_key to agents
    `ALTER TABLE agents ADD COLUMN session_key TEXT NOT NULL DEFAULT ''`,
    // M8: Staging tokens
    `CREATE TABLE IF NOT EXISTS staging_tokens (...)`,
    // M9-M10: Exfiltrations
    `CREATE TABLE IF NOT EXISTS exfiltrations (...)`,
    `CREATE TABLE IF NOT EXISTS exfil_chunks (...)`,
    // M11: Agent tags
    `ALTER TABLE agents ADD COLUMN tags TEXT NOT NULL DEFAULT '[]'`,
    // M12-M13: Modules (recreate with composite PK)
    `DROP TABLE IF EXISTS modules`,
    `CREATE TABLE IF NOT EXISTS modules (... PRIMARY KEY (name, platform, arch))`,
}

    O sistema e forward-only: cada migracao tem um numero sequencial, e o schema_version table rastreia qual foi a ultima aplicada.

    func (s *Store) migrate() error {
    // Cria tabela de controle
    s.db.Exec(`CREATE TABLE IF NOT EXISTS schema_version (version INTEGER PRIMARY KEY)`)

    // Le versao atual
    var currentVersion int
    s.db.QueryRow("SELECT COALESCE(MAX(version), 0) FROM schema_version").Scan(&currentVersion)

    // Aplica apenas migracoes novas
    for i, m := range migrations {
        version := i + 1
        if version <= currentVersion { continue }
        s.db.Exec(m)
        s.db.Exec("INSERT INTO schema_version (version) VALUES (?)", version)
    }
}

    O tratamento de ALTER TABLE duplicado (quando a coluna ja existe) e feito gracefully:

    if strings.Contains(err.Error(), "duplicate column") {
    // Registra versao mesmo assim (idempotente)
    s.db.Exec("INSERT INTO schema_version (version) VALUES (?)", version)
    continue
}

    Padroes de Query¶

    Prepared Statements¶

    Todas as queries usam prepared statements com ? placeholders para prevenir SQL injection:

    s.db.QueryRow("SELECT ... FROM agents WHERE id = ?", id)
s.db.Exec("UPDATE agents SET status = ? WHERE id = ?", status, id)

    ID Generation¶

    IDs sao gerados com 16 bytes de crypto/rand, resultando em 32 hex chars:

    func generateID() (string, error) {
    b := make([]byte, 16) // 128 bits de entropia
    rand.Read(b)
    return hex.EncodeToString(b), nil
}

    Isso equivale a UUID v4 em termos de unicidade (2^128 possibilidades), mas sem os hifens.

    Transacoes¶

    Transacoes sao usadas quando multiplas operacoes precisam ser atomicas. O padrao e Begin() + defer Rollback() + Commit():

    tx, _ := s.db.Begin()
defer tx.Rollback() // safety net — rollback se Commit nao foi chamado

tx.Exec("INSERT INTO exfil_chunks ...")
tx.Exec("UPDATE exfiltrations SET received_chunks = received_chunks + 1 ...")

return tx.Commit() // atomico

    Time Handling¶

    Timestamps do SQLite sao parseados com multiplos formatos para compatibilidade:

    func parseTime(s string) time.Time {
    for _, layout := range []string{
        time.RFC3339,
        "2006-01-02T15:04:05Z",
        "2006-01-02 15:04:05",
    } {
        if t, err := time.Parse(layout, s); err == nil {
            return t
        }
    }
    return time.Time{} // zero value se nenhum formato bate
}

    Backup e Recovery¶

    O SQLite em modo WAL produz tres arquivos:

    Arquivo Conteudo
    wolfgang.db Banco principal
    wolfgang.db-wal Write-Ahead Log (commits pendentes)
    wolfgang.db-shm Shared memory index

    Para backup seguro: copie os tres arquivos enquanto o server esta rodando (WAL garante consistencia de leitura), ou pare o server e copie apenas o .db (o WAL sera merged).
