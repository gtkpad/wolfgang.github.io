    Visao Geral da Arquitetura¶

    Por que Go?¶

    A escolha de Go como linguagem principal do Wolfgang nao foi acidental. Tres propriedades de Go sao criticas para um C2 framework:

    Cross-compilation nativa. Um unico comando GOOS=windows GOARCH=amd64 go build produz um binario Windows a partir de um host Linux. Isso elimina a necessidade de VMs ou toolchains por plataforma. O pipeline de geracao de implants (pkg/generate) usa exatamente isso, injetando configuracao via -ldflags -X.

    Binario estatico. Go compila para um unico executavel sem dependencias de runtime (sem .NET, sem Python, sem libc se usando CGO_ENABLED=0). O implant roda em qualquer target sem pre-requisitos. A excecao e o SQLite via modernc.org/sqlite, que e uma traducao pura de C para Go — zero CGO.

    Goroutines para concorrencia. O teamserver gerencia simultaneamente: API REST, WebSocket hub, multiplos transport listeners (HTTPS, DNS, WS), task manager e plugin manager. Goroutines tornam isso trivial sem threads OS pesadas.

    Trade-off aceito: binarios Go sao maiores que C/C++ (tipicamente 5-15MB). O Wolfgang mitiga isso com garble para obfuscacao e strip de simbolos. Para cenarios onde tamanho importa, o stager (stage 0) e minimo e baixa o implant completo em runtime.

    Por que SQLite?¶

    O Wolfgang usa SQLite em modo WAL (Write-Ahead Logging) como unico backend de persistencia:

    Zero dependencias externas. Nenhum PostgreSQL ou MySQL para instalar e configurar. O teamserver abre um arquivo .db e esta pronto. Isso e critico para deploy rapido em engagements.

    Escala suficiente. Um C2 tipicamente gerencia dezenas a centenas de agents, nao milhoes. SQLite suporta esse volume com folga. O modo WAL permite leituras concorrentes sem bloquear escritas.

    Embedded. A biblioteca modernc.org/sqlite e pure Go — sem CGO, sem .so ou .dll em runtime. Isso mantem o binario self-contained.

    // pkg/storage/storage.go — inicializacao
db, err := sql.Open("sqlite", dbPath)
// WAL mode: leituras nao bloqueiam escritas
db.Exec("PRAGMA journal_mode=WAL")
// Integridade referencial entre tabelas
db.Exec("PRAGMA foreign_keys=ON")

    Trade-off aceito: SQLite nao suporta acesso de rede (multiplos teamservers compartilhando um DB). Se multiplos operadores precisam de acesso simultaneo, todos conectam ao mesmo processo wolfgang-server via API — o que e o design intencional.

    Por que Monolito Modular + Plugins?¶

    O Wolfgang segue o padrao modular monolith: todos os componentes core (API, agent manager, task manager, crypto, transports) vivem no mesmo processo. Ao mesmo tempo, funcionalidade extensivel usa Hashicorp go-plugin para isolamento de processo.

    Monolito para o core:

    • Um unico binario wolfgang-server — deploy simples
    • Sem latencia de rede entre componentes internos
    • Shared memory para agent sessions e replay windows
    • Um unico processo para monitorar e operar

    Plugins para extensibilidade:

    • Cada plugin roda como processo separado via gRPC
    • Crash de um plugin nao derruba o server
    • Plugins podem ser escritos em qualquer linguagem que suporte gRPC
    • Verificacao SHA256 antes do carregamento (supply chain security)
    • Auto-mTLS entre server e plugin process

    Os 5 Binarios¶

    O Wolfgang compila para 5 binarios distintos, cada um com responsabilidade clara:

    wolfgang-server (Teamserver)¶

    O coracao do framework. Responsavel por:

    • API REST (Chi router) para operadores autenticados
    • Transport listeners (HTTPS, DNS, WebSocket) para agents
    • Agent Manager — sessoes criptograficas, replay windows, key exchange
    • Task Manager — lifecycle de tasks (pending -> dispatched -> completed/expired)
    • Plugin Manager — discovery, verificacao e lifecycle de plugins go-plugin
    • Generate Pipeline — compilacao de implants customizados com configuracao embutida
    • Audit Logger — log de todas as acoes de operadores e eventos de sistema
    cmd/server/main.go
├── init     — Cria config, TLS certs, admin operator
├── start    — Inicia o teamserver completo
└── generate — Compila implant customizado

    wolfgang-client (CLI do Operador)¶

    Interface de linha de comando interativa (REPL) para operadores:

    • Autenticacao via JWT (login com usuario/senha)
    • TOFU (Trust-On-First-Use) para certificados TLS do server
    • WebSocket para eventos real-time (novo agent, task concluida)
    • Comandos para gerenciar agents, tasks, listeners e plugins

    wolfgang-implant (Agent)¶

    O binario deployado no target. Execucao completamente autonoma:

    • Evasion chain automatica no boot (AMSI, ETW, sandbox detection)
    • Transport chain com fallback automatico entre protocolos
    • Module registry com filtragem por plataforma (OS-specific)
    • Dynamic module loading — modulos sob demanda do server
    • Reconnector com exponential backoff, domain rotation e kill switch
    • Sleep encryption — cifra memoria durante idle (Windows)

    Toda configuracao e embutida em build time via -ldflags:

    // cmd/implant/config.go — variaveis injetadas em build time
var (
    buildServerPublicKey = "" // X25519 pubkey (key pinning)
    buildServerURL       = "" // primary callback
    buildTransport       = "" // "https", "dns", "ws"
    buildSleep           = "" // check-in interval (ms)
    buildCallbackURLs    = "" // fallback URLs
    buildTransportPrio   = "" // transport priority list
    buildAutoAMSI        = "" // auto AMSI bypass
    buildDynamicModules  = "" // on-demand module loading
)

    wolfgang-stager (Stage 0 Dropper)¶

    Binario minimo (~2-3KB de logica) que:

    1. Faz HTTP GET para URL embutida via ldflags
    2. Recebe payload cifrado com AES-256-GCM
    3. Decifra em memoria
    4. Executa o implant completo via platformExec()

    O stager nao tem criptografia de protocolo, ECDH ou modulos. Existe apenas para ser pequeno e baixar o implant real.

    implant-sim (Simulador de Implant)¶

    Ferramenta de teste que simula N agents simultaneos:

    • Executa o handshake ECDH completo com o server
    • Envia check-ins com o protocolo envelope real
    • Responde a tasks com output simulado
    • Configura latencia artificial e taxa de falha
    • Suporta multiplos agents concorrentes (--count)

    Essencial para testar o teamserver sem deployar implants reais.

    Fluxo Tipico¶

    1. Operador conecta ao teamserver via wolfgang-client (HTTPS + JWT auth)
    2. Teamserver gerencia listeners em portas configuraveis
    3. Redirectors recebem trafego dos implants e encaminham ao teamserver
    4. Implants usam transport chain com fallback automatico:
      • Primario: HTTPS com malleable profile (parece trafego Google)
      • Fallback 1: DNS via subdomains base32
      • Fallback 2: DoH via resolvers publicos
    5. Named Pipe permite encadeamento: implant C conecta ao implant A que faz relay upstream

    Consideracoes de Rede¶

    • O teamserver nunca e exposto diretamente a rede alvo
    • Redirectors sao descartaveis e substituiveis
    • Domain fronting usa CDNs legitimas (o TLS SNI mostra o CDN, o Host header aponta o C2)
    • DNS transport funciona mesmo em redes muito restritivas (UDP 53 quase sempre esta aberto)
    • Named Pipe permite pivot entre segmentos de rede sem trafego externo

    Trade-offs Conscientes¶

    Decisao Vantagem Custo
    Go em vez de C/Rust Cross-compile trivial, dev velocity Binarios maiores (5-15MB)
    SQLite em vez de PostgreSQL Zero deps, deploy instantaneo Sem acesso multi-processo
    Monolito em vez de microservices Latencia zero, deploy simples Scale vertical apenas
    go-plugin em vez de shared libs Crash isolation, language agnostic Overhead gRPC por chamada
    Envelope proprio em vez de protobuf Header fixo 26B, sem schema negotiation Sem backward compat automatico
    Nonce derivado em vez de random Economia de 12B por mensagem Requires seq sync
