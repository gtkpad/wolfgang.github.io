    Plugin SDK¶

    O Wolfgang suporta extensão via plugins externos que rodam como processos separados comunicando-se com o server via gRPC. O SDK é baseado no hashicorp/go-plugin e suporta dois tipos de plugin: Listener e Module.

    Arquitetura de Plugins¶

    flowchart LR
    subgraph "Server Process"
        PM[Plugin Manager]
        GRPCC[gRPC Client]
    end

    subgraph "Plugin Process"
        GRPCS[gRPC Server]
        SDK[pluginsdk]
        IMPL[Plugin Implementation]
    end

    PM -->|Discover & Launch| GRPCC
    GRPCC <-->|gRPC over stdio| GRPCS
    GRPCS --> SDK
    SDK --> IMPL

    Características:

    • Plugins são binários independentes -- falhas no plugin não derrubam o server
    • Comunicação via gRPC over stdio (sem portas expostas)
    • Magic cookie para handshake: WOLFGANG_PLUGIN=wolfgang-v1
    • Protocol version: 1
    • SDK version: 1.0.0

    Interfaces do SDK¶

    O SDK (pkg/pluginsdk/sdk.go) define as interfaces que plugins implementam:

    Listener Plugin¶

    type Listener interface {
    Start(cfg ListenerConfig) (ListenerStatus, error)
    Stop(id string) error
}

type ListenerConfig struct {
    ID      string            `json:"id"`
    Address string            `json:"address"`
    Options map[string]string `json:"options,omitempty"`
}

type ListenerStatus struct {
    ID      string
    Running bool
    Address string
}

    Module Plugin¶

    type Module interface {
    Name() string
    Execute(args ModuleArgs) ModuleResult
}

type ModuleArgs struct {
    Module  string            `json:"module"`
    AgentID string            `json:"agent_id"`
    Args    map[string]string `json:"args,omitempty"`
}

type ModuleResult struct {
    Success bool   `json:"success"`
    Output  string `json:"output,omitempty"`
    Error   string `json:"error,omitempty"`
}

    ModuleProvider (multi-module)¶

    type ModuleProvider interface {
    Execute(module, agentID string, args map[string]string) (ModuleResult, error)
    List() []string
}

    Exemplo Completo: Plugin de Listener¶

    Vamos criar um listener plugin que aceita conexões em um protocolo customizado.

    Estrutura¶

    plugins/example-custom/
├── go.mod
├── go.sum
└── main.go

    go.mod¶

    module example-custom

go 1.25.0

require github.com/user/wolfgang v0.0.0

replace github.com/user/wolfgang => ../../

    main.go¶

    package main

import (
    "fmt"
    "log"
    "net"
    "sync"

    "github.com/user/wolfgang/pkg/pluginsdk"
)

func main() {
    listener := &customListener{
        listeners: make(map[string]net.Listener),
    }
    pluginsdk.ServeListener("example-custom", "0.1.0", listener)
}

type customListener struct {
    mu        sync.Mutex
    listeners map[string]net.Listener
}

func (c *customListener) Start(cfg pluginsdk.ListenerConfig) (pluginsdk.ListenerStatus, error) {
    c.mu.Lock()
    defer c.mu.Unlock()

    if _, exists := c.listeners[cfg.ID]; exists {
        return pluginsdk.ListenerStatus{}, fmt.Errorf("listener %s already running", cfg.ID)
    }

    addr := cfg.Address
    if addr == "" {
        addr = ":9999"
    }

    ln, err := net.Listen("tcp", addr)
    if err != nil {
        return pluginsdk.ListenerStatus{}, fmt.Errorf("bind failed: %w", err)
    }

    c.listeners[cfg.ID] = ln
    log.Printf("[custom] Listening on %s (id=%s)", ln.Addr(), cfg.ID)

    go c.acceptLoop(cfg.ID, ln)

    return pluginsdk.ListenerStatus{
        ID:      cfg.ID,
        Running: true,
        Address: ln.Addr().String(),
    }, nil
}

func (c *customListener) Stop(id string) error {
    c.mu.Lock()
    defer c.mu.Unlock()

    ln, ok := c.listeners[id]
    if !ok {
        return fmt.Errorf("listener %s not found", id)
    }

    delete(c.listeners, id)
    log.Printf("[custom] Stopped listener %s", id)
    return ln.Close()
}

func (c *customListener) acceptLoop(id string, ln net.Listener) {
    for {
        conn, err := ln.Accept()
        if err != nil {
            return
        }
        log.Printf("[custom] Connection from %s on %s", conn.RemoteAddr(), id)
        go c.handleConn(conn)
    }
}

func (c *customListener) handleConn(conn net.Conn) {
    defer conn.Close()
    // Implementar protocolo customizado aqui
}

# Copiar para o diretório de plugins

    Exemplo Completo: Plugin de Module¶

    main.go¶

    package main

import (
    "fmt"
    "os"
    "runtime"

    "github.com/user/wolfgang/pkg/pluginsdk"
)

func main() {
    modules := []pluginsdk.Module{
        &envModule{},
        &uptimeModule{},
    }
    pluginsdk.ServeModule("example-recon", "0.1.0", modules)
}

type envModule struct{}

func (m *envModule) Name() string { return "plugin.env" }

func (m *envModule) Execute(args pluginsdk.ModuleArgs) pluginsdk.ModuleResult {
    env := fmt.Sprintf("PATH=%s\nHOME=%s\nGOOS=%s",
        os.Getenv("PATH"), os.Getenv("HOME"), runtime.GOOS)

    return pluginsdk.ModuleResult{
        Success: true,
        Output:  env,
    }
}

type uptimeModule struct{}

func (m *uptimeModule) Name() string { return "plugin.uptime" }

func (m *uptimeModule) Execute(args pluginsdk.ModuleArgs) pluginsdk.ModuleResult {
    return pluginsdk.ModuleResult{
        Success: true,
        Output:  "[uptime data]",
    }
}

    Plugin de Referência: example-smb¶

    O projeto inclui um plugin de listener SMB (plugins/example-smb/) que demonstra:

    • Implementação mínima do protocolo SMB2 para disfarce de tráfego
    • Uso de pluginsdk.ServeListener() para registrar o plugin
    • Gerenciamento de múltiplos listeners via map com mutex
    • Handshake SMB2: Negotiate -> Session Setup -> Read/Write

    Pontos relevantes do código:

    func main() {
    listener := &smbListener{
        listeners: make(map[string]net.Listener),
    }
    pluginsdk.ServeListener("example-smb", "0.1.0", listener)
}

    O ServeListener cuida de todo o boilerplate gRPC: registra as 3 interfaces (info, listener, module), configura o handshake, e inicia o server gRPC.

    SDK C para Dynamic Modules¶

    Além de plugins Go via gRPC, o Wolfgang suporta modules dinâmicos em C carregados reflectively no processo do implant. O SDK C está em sdk/module.h:

    typedef struct {
    const char* args;      // JSON arguments
    int32_t     args_len;
    char*       out;       // output buffer (allocated by caller)
    int32_t     out_cap;
    int32_t     out_len;   // bytes written by module
} ModuleContext;

    Cada module C exporta 3 funções:

    Função Propósito
    module_init() Inicialização (chamado uma vez após load)
    module_execute(ModuleContext* ctx) Execução principal
    module_cleanup() Cleanup antes de unload

    Exemplo em sdk/example/hello.c:

    MODULE_EXPORT int32_t module_execute(ModuleContext* ctx) {
    const char* msg = "{\"output\":\"hello from reflective module\"}";
    int32_t len = (int32_t)strlen(msg);
    if (len > ctx->out_cap) {
        len = ctx->out_cap;
    }
    memcpy(ctx->out, msg, len);
    ctx->out_len = len;
    return 0;
}

    Build com mingw para Windows:

    cd sdk
make  # Compila hello.dll via x86_64-w64-mingw32-gcc

    Importante: Modules C nunca devem ser compilados com Go. Go DLLs contém seu próprio runtime e causam crash quando carregados reflectively em um processo Go.

    Ciclo de Vida do Plugin¶

    1. Discovery: O server escaneia ~/.wolfgang/plugins/ em busca de binários executáveis
    2. Launch: Cada binário é executado como processo separado
    3. Handshake: Verificação do magic cookie WOLFGANG_PLUGIN=wolfgang-v1
    4. Info: Server consulta GetInfo() para tipo, nome, versão, capabilities
    5. Operation: Server invoca métodos via gRPC conforme necessário
    6. Shutdown: Server envia sinal de término ao processo do plugin
