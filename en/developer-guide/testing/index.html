    Testes¶

    O Wolfgang usa o package testing padrão do Go com table-driven tests, mocks manuais e um simulador de implant dedicado para testes E2E.

    Visão Geral¶

    go test ./...                                    # Todos os testes unitários
go test -tags integration ./test/integration/    # Testes E2E
go test -short ./...                             # Pular testes lentos (compilação)
go test -race ./...                              # Detectar race conditions
go test -v ./pkg/module/...                      # Verbose em package específico

    O projeto tem testes distribuídos em todas as camadas:

    Camada Arquivos de teste O que testam
    pkg/module/ module_test.go Registry, Executor, platform filtering
    pkg/module/recon/ sysinfo_test.go, keylogger_test.go, etc. Execução de modules individuais
    pkg/module/evasion/ amsi_etw_test.go, inject_test.go, etc. Evasion techniques
    pkg/transport/ envelope_test.go, replay_test.go Wire format, replay protection
    pkg/transport/http/ listener_test.go, profile_test.go HTTPS listener, malleable profiles
    pkg/crypto/ aead_test.go, keys_test.go, session_test.go Primitivas criptográficas
    pkg/auth/ jwt_test.go, rbac_test.go, ratelimit_test.go Autenticação e autorização
    pkg/storage/ storage_test.go, agents_test.go, tasks_test.go SQLite CRUD
    pkg/generate/ generate_test.go, polymorph_test.go Build system, polimorfismo
    pkg/pluginsdk/ sdk_test.go, grpc_test.go Plugin interfaces e gRPC wrappers
    test/integration/ e2e_test.go Fluxo completo server-side

    Padrões de Teste¶

    Table-Driven Tests¶

    O padrão predominante no projeto. Cada cenário é uma entrada na tabela:

    func TestMatchesPlatform(t *testing.T) {
    tests := []struct {
        name      string
        supported []string
        want      bool
    }{
        {"empty means all", nil, true},
        {"empty slice means all", []string{}, true},
        {"current OS matches", []string{runtime.GOOS}, true},
        {"multiple with current", []string{"plan9", runtime.GOOS, "js"}, true},
        {"no match", []string{"plan9", "js"}, false},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            if got := matchesPlatform(tt.supported); got != tt.want {
                t.Errorf("matchesPlatform(%v) = %v, want %v", tt.supported, got, tt.want)
            }
        })
    }
}

    Regras para table-driven tests no projeto:

    • Campo name descritivo no formato "condição" ou "input esperado"
    • Um t.Run() por entrada
    • Assertions com t.Errorf (não t.Fatal) para que todos os cenários rodem
    • wantErr bool para cenários de erro

    Mock Modules¶

    Mocks são structs simples no mesmo package de teste:

    type mockModule struct {
    name      string
    platforms []string
    execFn    func(ctx context.Context, args ModuleArgs) (ModuleResult, error)
}

func (m *mockModule) Name() string { return m.name }
func (m *mockModule) OS() []string { return m.platforms }
func (m *mockModule) Execute(ctx context.Context, args ModuleArgs) (ModuleResult, error) {
    if m.execFn != nil {
        return m.execFn(ctx, args)
    }
    return ModuleResult{Output: "mock: " + m.name}, nil
}

    O projeto prefere mocks manuais sobre frameworks de mock. Isso mantém os testes legíveis e sem dependências externas.

    Padrão de Teste para Modules¶

    Todo module segue o mesmo template de testes:

    func TestMyModule_Name(t *testing.T) {
    m := &MyModule{}
    if m.Name() != "expected.name" {
        t.Errorf("Name() = %q, want %q", m.Name(), "expected.name")
    }
}

func TestMyModule_OS(t *testing.T) {
    m := &MyModule{}
    // nil = all platforms, ou []string{"windows"} para específico
}

func TestMyModule_Execute(t *testing.T) {
    tests := []struct {
        name    string
        args    module.ModuleArgs
        wantErr bool
    }{
        {"normal execution", module.ModuleArgs{Raw: `{}`}, false},
        {"with args", module.ModuleArgs{Raw: `{"key":"value"}`}, false},
        {"invalid args", module.ModuleArgs{Raw: `{"bad":"input"}`}, true},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            m := &MyModule{}
            result, err := m.Execute(context.Background(), tt.args)
            if (err != nil) != tt.wantErr {
                t.Errorf("Execute() error = %v, wantErr %v", err, tt.wantErr)
            }
            if !tt.wantErr && result.Output == "" {
                t.Error("expected non-empty output")
            }
        })
    }
}

func TestMyModule_CancelledContext(t *testing.T) {
    ctx, cancel := context.WithCancel(context.Background())
    cancel()

    m := &MyModule{}
    _, err := m.Execute(ctx, module.ModuleArgs{})
    if err == nil {
        t.Error("expected error for cancelled context")
    }
}

    Concurrency Tests¶

    Para packages com acesso concorrente (como Registry):

    func TestRegistryConcurrentAccess(t *testing.T) {
    reg := NewRegistry()
    for i := 0; i < 10; i++ {
        name := "mod-" + string(rune('a'+i))
        reg.RegisterModule(&mockModule{name: name, platforms: nil})
    }

    done := make(chan bool, 20)
    for i := 0; i < 10; i++ {
        go func() {
            reg.ListModules()
            done <- true
        }()
        go func() {
            reg.GetModule("mod-a")
            done <- true
        }()
    }
    for i := 0; i < 20; i++ {
        <-done
    }
}

    Sempre rode com -race para detectar data races:

    go test -race ./pkg/module/...

    Testes de Integração (E2E)¶

    Os testes E2E ficam em test/integration/ e usam a build tag integration:

    //go:build integration

    Executar:

    go test -tags integration ./test/integration/

    Padrão dos Testes E2E¶

    Os testes E2E configuram um server completo em memória usando httptest.Server:

    func setupServer(t *testing.T) (*httptest.Server, *storage.Store, *auth.JWTService) {
    t.Helper()
    store, err := storage.New(":memory:")
    if err != nil {
        t.Fatalf("create store: %v", err)
    }
    t.Cleanup(func() { store.Close() })

    jwtSvc := auth.NewJWTServiceWithSecret([]byte("test-secret-key-32-bytes-long!!1"))
    authSvc := auth.NewAuthService(jwtSvc, store)

    router := api.NewRouter(api.ServerDeps{
        Store:       store,
        JWTService:  jwtSvc,
        AuthService: authSvc,
    })

    ts := httptest.NewServer(router)
    t.Cleanup(ts.Close)
    return ts, store, jwtSvc
}

    Cenários cobertos:

    Teste Fluxo
    TestE2E_AgentLifecycle Login -> list agents -> register -> dispatch task -> cancel task -> delete agent
    TestE2E_AuthFlow Login com senha errada -> login correto -> acesso com token -> refresh -> token expirado
    TestE2E_RBAC Admin vs operator permissions em diferentes endpoints
    TestE2E_MultipleTasksOrdering Múltiplas tasks retornadas na ordem correta
    TestE2E_AgentTagging CRUD de tags em agents

    Implant Simulator (cmd/implant-sim)¶

    O implant-sim é um binário que simula o comportamento de implants reais para testes do server sem precisar de deploys reais.

    Uso¶

    # Um agent simulado
go run ./cmd/implant-sim -server https://127.0.0.1:8443

# 10 agents simultâneos com latência de 2s e 20% de falha
go run ./cmd/implant-sim \
    -server https://127.0.0.1:8443 \
    -count 10 \
    -latency 2s \
    -fail-rate 0.2

# Customizar check-in interval
go run ./cmd/implant-sim \
    -server https://127.0.0.1:8443 \
    -sleep 10000 \
    -jitter 20

    Flags¶

    Flag Default Descrição
    -server https://127.0.0.1:8443 URL do server C2
    -transport https Tipo de transport
    -count 1 Número de agents simultâneos
    -latency 0 Latência artificial nas respostas de tasks
    -fail-rate 0 Taxa de falha aleatória (0.0 a 1.0)
    -sleep 5000 Intervalo de check-in em ms
    -jitter 10 Jitter em porcentagem (0-100)

    Comportamento¶

    O simulador executa o fluxo completo:

    1. Registration: ECDH key exchange real com o server (X25519)
    2. Check-in loop: Envelopes encriptados com AES-256-GCM
    3. Task execution: Respostas simuladas por module type
    4. Jitter: Distribuição normal gaussiana com clamp

    Respostas simuladas por module:

    Module Output simulado
    shell sim$ <args>\n[simulated shell output]
    recon Hostname, OS, user, IP fictícios
    fileops [simulated fileops: <args> completed]
    persist [simulated persistence: <args> installed]
    lateral [simulated lateral movement: <args>]
    evasion [simulated evasion: <args> applied]

    Cenários de Teste com implant-sim¶

    # Teste de carga: 50 agents com check-in a cada segundo
go run ./cmd/implant-sim -count 50 -sleep 1000

# Teste de resiliência: alta taxa de falha
go run ./cmd/implant-sim -count 5 -fail-rate 0.5

# Teste de latência: simular agents lentos
go run ./cmd/implant-sim -count 3 -latency 10s

# Teste de jitter: verificar distribuição
go run ./cmd/implant-sim -sleep 5000 -jitter 50

    Testes de Build¶

    Testes do package generate incluem compilação real do implant:

    func TestBuild_success(t *testing.T) {
    if testing.Short() {
        t.Skip("skipping compilation test in short mode")
    }
    // ... compila implant real e verifica o binário
}

    Use -short para pular esses testes em CI rápido:

    go test -short ./pkg/generate/...

    Estratégia de Mocking¶

    O projeto evita frameworks de mock e usa interfaces + structs de teste:

    Abordagem Quando usar
    Mock manual (struct) Dependências simples com 1-3 métodos
    Fake (in-memory) Storage (":memory:" SQLite), HTTP (httptest.Server)
    Build tag Código platform-specific (_windows_test.go)
    -short flag Pular testes que compilam binários reais
    integration tag Testes E2E que precisam de server completo

    Exemplo: Fake Storage¶

    store, err := storage.New(":memory:")
if err != nil {
    t.Fatalf("create store: %v", err)
}
t.Cleanup(func() { store.Close() })

    SQLite in-memory é usado em praticamente todos os testes que precisam de persistência.

    Convenções¶

    1. Nomes: TestFuncName_Cenario ou TestStruct_Method_Cenario
    2. t.Helper(): Sempre em funções auxiliares de teste
    3. t.Cleanup(): Para cleanup de recursos (fechar DB, parar servers)
    4. t.Parallel(): Usado quando testes são independentes entre si
    5. Subtests: t.Run("cenário", func(t *testing.T) {...}) para agrupamento
    6. Sem testify: Package testing padrão em todo o projeto
