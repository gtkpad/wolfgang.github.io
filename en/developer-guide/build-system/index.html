    Sistema de Build¶

    O Wolfgang usa go build com ldflags para embutir configuração nos binários do implant. O build system suporta cross-compilation, obfuscação via garble, transformações polimórficas e empacotamento via Docker.

    Targets do Makefile¶

    make build     # Compila todos os binários (go build ./...)
make test      # Roda todos os testes (go test ./...)
make swagger   # Gera documentação Swagger da API

    O Makefile é intencionalmente simples. A compilação do implant com configuração customizada é feita programaticamente pelo server via pkg/generate.

    Build Pipeline¶

    flowchart TD
    REQ[Generate Request
    OS, Arch, Transport, Callback,
    Sleep, Jitter, Profile, Flags]
    VAL{Validate}
    REQ --> VAL
    VAL -->|Inválido| ERR[Retorna erro]
    VAL -->|Válido| POLY{Polymorphic?}

    POLY -->|Sim| JUNK[Gerar junk_*.go
    3-10 funções aleatórias]
    POLY -->|Não| LDFLAGS

    JUNK --> LDFLAGS[Construir -ldflags
    -s -w -X main.buildServerURL=...
    -X main.buildTransport=...
    -X main.buildSleep=...]

    LDFLAGS --> GARBLE{UseGarble &&
    garble available?}

    GARBLE -->|Sim| GBUILD[garble -literals -tiny
    -seed=random build
    -trimpath -ldflags=...]
    GARBLE -->|Não| GOBUILD[go build
    -trimpath -ldflags=...]

    GBUILD --> GCFLAGS[gcflags aleatório
    vazio / -N / -l / -N -l]
    GOBUILD --> GCFLAGS

    GCFLAGS --> ENV[GOOS + GOARCH + CGO_ENABLED=0]
    ENV --> COMPILE[Compilação]
    COMPILE --> CLEANUP{Polymorphic?}
    CLEANUP -->|Sim| REMOVE[Remove junk_*.go]
    CLEANUP -->|Não| RESULT
    REMOVE --> RESULT[Result: path, filename, size, OS, arch]

    Request de Build¶

    A struct Request em pkg/generate/generate.go define todos os parâmetros:

    Campo Tipo Default Descrição
    OS string runtime.GOOS Target OS: linux, windows, darwin
    Arch string runtime.GOARCH Target arch: amd64, arm64, 386
    Transport string https Transport primário: https, dns, ws, doh
    Callback []string -- URLs de callback (1a = primária, demais = fallback)
    Sleep int 5000 Intervalo de check-in em ms
    Jitter int -- Porcentagem de jitter (0-100)
    Profile string -- Nome do malleable profile
    Format string auto Formato: exe, elf, macho
    ServerPublicKey string -- Chave pública X25519 hex-encoded
    TransportPriority []string -- Ordem de fallback de transports
    FrontingDomain string -- Domínio CDN para domain fronting
    ActualHost string -- Host header real
    UseGarble bool false Usar garble para obfuscação
    Polymorphic bool false Aplicar transformações polimórficas
    AutoAMSI bool false Auto-patch AMSI no startup
    AutoETW bool false Auto-patch ETW no startup
    AutoSandbox bool false Detecção de sandbox
    SandboxThreshold int -- Mínimo de checks para executar
    SleepEncrypt bool false Encriptar heap durante sleep
    DynamicModules bool false Carregamento de modules on-demand
    DNSDomain string -- Zona DNS autoritativa
    DNSResolvers string -- Resolvers DNS customizados

    Cross-Compilation¶

    O Wolfgang compila para 3 sistemas operacionais e 3 arquiteturas:

    OS Arquiteturas Formato
    linux amd64, arm64, 386 ELF
    windows amd64, arm64, 386 EXE
    darwin amd64, arm64 Mach-O

    Compilação manual:

    # Linux AMD64
GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" -o implant_linux_amd64 ./cmd/implant

# Windows AMD64
GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" -o implant_windows_amd64.exe ./cmd/implant

# Darwin ARM64 (Apple Silicon)
GOOS=darwin GOARCH=arm64 CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" -o implant_darwin_arm64 ./cmd/implant

    CGO_ENABLED=0 garante binários estáticos sem dependência de libc.

    LDFlags: Configuração Embarcada¶

    A configuração do implant é injetada via -X ldflags em tempo de compilação. O server monta a string de flags automaticamente:

    -s -w                                  # Strip debug info + DWARF
-X main.buildServerURL=https://c2:443  # URL primária
-X main.buildTransport=https           # Transport
-X main.buildSleep=5000                # Check-in interval
-X main.buildJitter=10                 # Jitter %
-X main.buildProfileName=google        # Malleable profile
-X main.buildServerPublicKey=abcdef... # X25519 public key
-X main.buildCallbackURLs=url1,url2    # Fallback URLs
-X main.buildTransportPrio=https,dns   # Transport fallback order
-X main.buildAutoAMSI=true             # Auto-bypass flags
-X main.buildAutoETW=true

    Quando Polymorphic=true, adiciona -buildid= para remover o build ID.

    Garble: Obfuscação de Strings¶

    Garble é usado opcionalmente para dificultar análise estática:

    # Instalar
go install mvdan.cc/garble@latest

# Build com garble (feito automaticamente quando UseGarble=true)
garble -literals -tiny -seed=random build -trimpath -ldflags="..." -o implant ./cmd/implant

    Flags do garble:

    Flag Efeito
    -literals Obfusca string literals no binário
    -tiny Remove informações extras para menor tamanho
    -seed=random Seed aleatória para diferentes hashes a cada build

    O Builder verifica se garble está disponível via exec.LookPath("garble"). Se não estiver instalado, usa go build normal.

    Junk Functions¶

    O sistema gera 3-10 funções Go aleatórias em um arquivo junk_<hex>.go:

    //go:noinline
func junk_a1b2c3d4() int {
    v0 := 42 + 17
    v1 := 88 ^ 33
    v2 := 15 - 7
    return v0 + v1 ^ v2
}

    Características das junk functions:

    • Pragma //go:noinline impede que o compilador as remova
    • _junkSink global referencia todas as funções via init()
    • Dead Code Elimination é impossível por causa do init()
    • Corpo tem 3-13 linhas com operações aritméticas aleatórias
    • Operadores: +, -, ^, |, &

    GCFlags Aleatórias¶

    O builder escolhe aleatoriamente uma dessas combinações de gcflags:

    Opção Efeito
    (vazio) Otimizações padrão
    -N Desabilita otimizações
    -l Desabilita inlining
    -N -l Desabilita ambos

    Isso produz binários com layouts de código diferentes a cada build.

    Cleanup¶

    Os arquivos junk_*.go são removidos automaticamente após a compilação via defer, independente de sucesso ou falha.

    Build de Stagers¶

    O Builder.BuildStager() compila stagers menores que baixam e executam o implant completo:

    req := &generate.StagerRequest{
    OS:       "windows",
    Arch:     "amd64",
    StageURL: "https://cdn.example.com/stage",
    StageKey: "hex-encoded-aes256-key",
}
result, err := builder.BuildStager(req)

    O stager embarca apenas 2 variáveis via ldflags:

    • buildStageURL: URL para download do implant
    • buildStageKey: Chave AES-256 para decriptar o payload

    Docker¶

    Dockerfile¶

    O build Docker usa multi-stage:

    Stage 1 (builder): golang:1.25-alpine
  - Compila server e client
  - Download de dependências

Stage 2 (runtime): golang:1.25-alpine
  - Copia binários compilados
  - Mantém Go SDK para compilação de implants em runtime
  - Copia source tree e module cache

    O runtime image mantém o Go SDK porque o server precisa compilar implants sob demanda.

    docker-compose.yml¶

    # Primeiro setup
docker compose run --rm wolfgang init

# Iniciar
docker compose up -d

    Portas expostas:

    Porta Serviço
    8443 API de gerenciamento (HTTPS)
    80 HTTP transport
    443 HTTPS transport
    8080 HTTP/WS transport (alternativo)
    8444 WSS (WebSocket over TLS)
    53/udp DNS transport

    Volumes:

    Volume Caminho Conteúdo
    wolfgang-data /root/.wolfgang Config, TLS certs, DB, plugins
    ./build /root/build Binários de implant gerados
    ./plugins /root/.wolfgang/plugins Plugins externos

    Build e Deploy com Docker¶

    # Build da imagem
docker compose build

# Iniciar em background
docker compose up -d

# Ver logs
docker compose logs -f

# Parar
docker compose down

    Testes do Build System¶

    # Testes rápidos (sem compilação real)
go test -short ./pkg/generate/...

# Testes completos (compila implant real)
go test ./pkg/generate/...

# O que é testado:
# - Validação de Request (OS, arch, transport, jitter, format)
# - LDFlags geradas corretamente
# - Resolução automática de formato por OS
# - Erro em diretórios read-only
# - Erro quando cmd/implant não existe
# - Build real com verificação de tamanho > 0
# - Geração de junk functions (unicidade, sintaxe válida)
# - randomHex, randomInt, randomChoice
