    Gerenciamento de Agents¶

    Agents sao as instancias do implant executando nos alvos. O Wolfgang gerencia o ciclo de vida completo -- do registro ate a remocao.

    Maquina de estados¶

    stateDiagram-v2
    [*] --> registered : ECDH Key Exchange
    registered --> active : Primeiro check-in
    active --> active : Check-in dentro de 3x sleep
    active --> dormant : Sem check-in por 3x-10x sleep
    dormant --> active : Check-in recebido
    dormant --> lost : Sem check-in por 10x+ sleep
    lost --> active : Check-in recebido
    active --> [*] : Kill / Self-destruct
    dormant --> [*] : Remove
    lost --> [*] : Remove

    Status dos agents¶

    O status e derivado automaticamente com base no intervalo de sleep e o tempo desde o ultimo check-in (last_seen):

    Status Condicao Significado
    active elapsed <= 3 * sleep Agent respondendo normalmente
    dormant 3 * sleep < elapsed <= 10 * sleep Agent possivelmente em delay ou rede lenta
    lost elapsed > 10 * sleep Agent provavelmente offline ou bloqueado

    A funcao DeriveStatus calcula o status:

    switch {
case elapsed <= 3*interval:
    return StatusActive   // "active"
case elapsed <= 10*interval:
    return StatusDormant  // "dormant"
default:
    return StatusLost     // "lost"
}

    Se o sleep nao estiver definido, assume 1 segundo como fallback.

    Registro de agent¶

    O registro acontece quando um novo implant conecta pela primeira vez:

    1. Agent gera um par de chaves X25519 efemero
    2. Envia metadata + public key para o listener (/register no HTTPS, message register no WS)
    3. Server realiza ECDH key exchange com o keypair do server
    4. Shared secret e derivado para session key (HKDF com agent ID como context)
    5. Agent recebe: agent ID + server public key
    6. Session key e persistida no banco (sobrevive restart do server)

    Metadata do registro¶

    Campo Descricao
    hostname Nome do host
    os Sistema operacional
    arch Arquitetura
    username Usuario executando o implant
    internal_ip IP interno
    external_ip IP externo (visto pelo listener)
    transport Transport utilizado
    sleep Intervalo de check-in (ms)
    jitter Percentual de jitter
    public_key Chave publica X25519 (hex)

    Restauracao de sessoes¶

    Na inicializacao, o server chama RestoreFromDB() para restaurar session keys de agents previamente registrados. Isso permite que agents continuem operando apos restart do teamserver sem necessidade de re-registro.

    Check-in¶

    O check-in e o heartbeat do agent:

    1. Agent envia um envelope criptografado com seu ID e sequence number
    2. Server verifica a session key e a replay window
    3. last_seen e atualizado para datetime('now')
    4. Status e recalculado com base no novo last_seen
    5. Tasks pendentes sao retornadas na resposta

    O check-in ocorre a cada sleep milissegundos, com variacao aleatoria de ate jitter%.

    Replay protection¶

    Cada mensagem carrega um sequence number monotonicamente crescente. A ReplayWindow no server rejeita messages com seq numbers ja vistos, prevenindo replay attacks.

    Sleep e Jitter¶

    Parametro Descricao Padrao
    sleep Intervalo base entre check-ins (ms) 5000 (implant) / 60000 (config)
    jitter Variacao percentual aleatoria 10%

    Exemplo com sleep=10000ms e jitter=20%:

    • Minimo: 10000 - 20% = 8000ms
    • Maximo: 10000 + 20% = 12000ms

    Valores sao definidos em dois niveis:

    1. Build time -- embutidos no implant via ldflags
    2. Config padrao -- seção implant do config.yaml (usado como default para novos implants)

    Listando agents¶

    wolfgang>> agents
ID        HOSTNAME    OS       USER      LAST SEEN            TRANSPORT  STATUS
a1b2c3d4  WORKST01    windows  admin     2026-03-14 10:30:00  https      active
b2c3d4e5  srv-linux   linux    root      2026-03-14 10:28:00  dns        dormant
c3d4e5f6  DESKTOP-X   windows  user1     2026-03-13 08:15:00  ws         lost

    A lista e ordenada por last_seen descendente (mais recente primeiro).

    Filtro por status¶

    A API suporta filtro por status via query parameter:

    GET /api/v1/agents?status=active

    Selecionando um agent¶

    wolfgang>> use a1b2c3d4

    O use aceita IDs parciais. O client consulta a lista de agents e resolve o prefixo:

    • Unico match: seleciona automaticamente
    • Multiplos matches: retorna erro de ambiguidade
    • Nenhum match: retorna erro

    Apos selecionar, o prompt muda para o contexto do agent e todos os comandos de operacao ficam disponiveis.

    Tags¶

    Agents podem ser taggeados para organizacao:

    PUT /api/v1/agents/{id}/tags
Body: ["dc", "high-value", "persistence"]

    Tags sao armazenadas como JSON array na coluna tags da tabela agents.

    Removendo agents¶

    Para remover um agent do banco de dados:

    DELETE /api/v1/agents/{id}

    Isso remove o registro do agent mas nao afeta o implant em execucao. Para terminar o implant, use selfdestruct antes de remover.

    Self-destruct¶

    O comando selfdestruct envia uma task especial que:

    1. Executa o modulo persist.cleanup (remove toda persistencia)
    2. Remove artefatos no disco
    3. Termina o processo do implant
    wolfgang [a1b2c3d4]>> selfdestruct
[!] WARNING: This will permanently terminate the implant. Confirm? (yes/no): yes
[*] Sending self-destruct command...
[+] Self-destruct dispatched. Agent will terminate.

    O agent e automaticamente deselected apos o dispatch.

    Persistencia de estado no server¶

    O estado do agent e armazenado em dois lugares:

    Dado Storage Persistencia
    Metadata (hostname, OS, etc.) SQLite Permanente
    Session key SQLite + memoria Permanente (restaurado no restart)
    Replay window Memoria Reseta no restart
    Server sequence counter Memoria Reseta no restart

    Na pratica, isso significa que apos restart do server:

    • Agents continuam autenticados (session key preservada)
    • Replay protection e reinicializada (aceita qualquer seq temporariamente)
    • Server sequence counters recomecam de 0
