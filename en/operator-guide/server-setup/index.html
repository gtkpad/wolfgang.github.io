    Configuracao do Servidor¶

    O teamserver e o nucleo do Wolfgang -- gerencia autenticacao de operadores, listeners, agents e task dispatch. Esta pagina documenta toda a estrutura de configuracao e as opcoes de tuning.

    Estrutura completa do config.yaml¶

    O arquivo de configuracao utiliza YAML. Se o arquivo nao existir, valores padrao sao aplicados automaticamente. O caminho padrao e config.yaml no diretorio de trabalho.

    # === Server ===
server:
  bind: "0.0.0.0"        # Interface de escuta da API
  port: 8443              # Porta HTTPS da API
  tls:
    mode: "auto"          # "auto" | "custom" | "mtls"
    cert_file: ""         # Caminho do certificado (custom/mtls)
    key_file: ""          # Caminho da chave privada (custom/mtls)
    ca_file: ""           # Caminho do CA (apenas mtls)

# === Autenticacao ===
auth:
  jwt_ttl: 1h             # Tempo de vida do access token
  refresh_ttl: 24h        # Tempo de vida do refresh token
  rate_limit: 5           # Max tentativas de login por IP/janela
  ip_allowlist: []         # IPs permitidos (vazio = todos)
  lockout_max: 10          # Falhas consecutivas antes do lockout
  lockout_dur: 15m         # Duracao do lockout

# === Storage ===
storage:
  path: "~/.wolfgang/wolfgang.db"  # Caminho do banco SQLite

# === Plugins ===
plugins:
  dir: "plugins"           # Diretorio de binarios de plugin
  verify_hash: true        # Verificar SHA256 antes de carregar

# === Listeners pre-configurados ===
listeners:
  - type: https
    bind_addr: "0.0.0.0"
    port: 443
    profile: ""            # Arquivo de malleable profile

# === Implant defaults ===
implant:
  sleep: 60s               # Intervalo de check-in padrao
  jitter: 10               # Percentual de jitter (0-100)

    Modos de TLS¶

    Auto (padrao)¶

    O modo auto gera uma CA self-signed e um certificado de servidor automaticamente. Os certificados sao armazenados em ~/.wolfgang/tls/:

    • ca-key.pem / ca-cert.pem -- CA interna
    • server-key.pem / server-cert.pem -- certificado do servidor

    Certificados sao gerados com ECDSA P-256, validade de 1 ano (servidor) e 10 anos (CA). O client utiliza TOFU (Trust On First Use) para fixar o fingerprint SHA-256 na primeira conexao.

    server:
  tls:
    mode: "auto"

    Custom¶

    Para usar certificados proprios (Let's Encrypt, CA corporativa, etc.):

    server:
  tls:
    mode: "custom"
    cert_file: "/etc/wolfgang/server.crt"
    key_file: "/etc/wolfgang/server.key"

    Ambos cert_file e key_file sao obrigatorios neste modo. O TLS minimo e 1.2.

    mTLS (Mutual TLS)¶

    O modo mtls exige que os clients apresentem um certificado valido assinado pela CA especificada. Util para restringir acesso ao teamserver apenas a operadores com certificado.

    server:
  tls:
    mode: "mtls"
    cert_file: "/etc/wolfgang/server.crt"
    key_file: "/etc/wolfgang/server.key"
    ca_file: "/etc/wolfgang/operator-ca.crt"

    Os tres campos sao obrigatorios. A policy e RequireAndVerifyClientCert -- conexoes sem certificado valido sao rejeitadas no handshake TLS.

    Multi-operador e RBAC¶

    O Wolfgang suporta multiplos operadores com tres roles:

    Role Permissoes
    admin Todas as operacoes, incluindo criar/remover operadores
    operator Gerenciar listeners, agents, tasks e modulos
    viewer Somente leitura -- listar agents, tasks e logs

    Criar operadores¶

    Via CLI interativo:

    wolfgang>> operators create
Username: analyst1
Password: ********
Role (admin/operator/viewer): operator
[+] Operator created: a1b2c3d4 (analyst1, role=operator)

    Cada operador recebe um par de tokens JWT (access + refresh). O access token tem TTL configuravel (jwt_ttl, padrao 1h) e o client faz refresh automatico.

    Seguranca: Rate Limiting e Lockout¶

    O rate limiter opera por IP com sliding window:

    auth:
  rate_limit: 5        # Max 5 tentativas por janela
  lockout_max: 10      # Lockout apos 10 falhas consecutivas
  lockout_dur: 15m     # Bloqueio por 15 minutos

    Comportamento:

    1. Cada IP tem um contador de tentativas dentro da janela
    2. Se rate_limit e excedido, retorna HTTP 429 (Too Many Requests)
    3. Falhas de autenticacao incrementam o contador de falhas consecutivas
    4. Ao atingir lockout_max, o IP e bloqueado por lockout_dur
    5. Login bem-sucedido reseta o contador de falhas
    6. Apos o lockout expirar, os contadores sao zerados

    IP Allowlist¶

    Para restringir acesso ao teamserver a IPs especificos:

    auth:
  ip_allowlist:
    - "10.0.0.50"
    - "10.0.0.51"
    - "192.168.1.100"

    Quando a lista esta vazia (padrao), todos os IPs sao permitidos. Quando populada, requisicoes de IPs fora da lista recebem HTTP 403 (Forbidden). O middleware IPAllowlist extrai o IP de RemoteAddr, suportando conexoes diretas e atraves de proxies.

    Storage¶

    O Wolfgang usa SQLite com modo WAL (Write-Ahead Logging) para performance de leitura concorrente. O schema e versionado e migracoes rodam automaticamente no startup.

    storage:
  path: "~/.wolfgang/wolfgang.db"

    Tabelas principais¶

    Tabela Funcao
    operators Credenciais e roles dos operadores
    agents Metadata dos implants conectados
    tasks Fila de tasks e resultados
    listeners Listeners configurados
    logs Audit trail de operacoes
    refresh_tokens Tokens de refresh ativos
    staging_tokens Tokens para staging de implants
    exfiltrations Sessoes de exfiltracao
    exfil_chunks Chunks de dados exfiltrados
    modules Binarios de modulos dinamicos

    Pragmas SQLite¶

    • journal_mode=WAL -- melhor concorrencia de leitura
    • foreign_keys=ON -- integridade referencial

    Backup¶

    O banco SQLite e um arquivo unico. Para backup:

    # Backup com sqlite3 (garante consistencia WAL)
sqlite3 ~/.wolfgang/wolfgang.db ".backup /backup/wolfgang-$(date +%Y%m%d).db"

# Ou copiar com checkpoint WAL primeiro
sqlite3 ~/.wolfgang/wolfgang.db "PRAGMA wal_checkpoint(TRUNCATE);"
cp ~/.wolfgang/wolfgang.db /backup/

    Tuning¶

    Timeouts e conexoes¶

    Valores recomendados para engagements:

    Parametro Valor Notas
    server.port 8443 Porta da API (operadores)
    auth.jwt_ttl 1h Reduzir para ambientes mais restritivos
    auth.refresh_ttl 24h Sessao maxima sem re-autenticacao
    implant.sleep 60s Intervalo padrao para novos implants
    implant.jitter 10 10% de variacao no sleep

    Consideracoes de performance¶

    • Para engagements com muitos agents (>50), use wal_checkpoint periodico
    • O rate limiter usa memoria proporcional ao numero de IPs unicos
    • Listeners DNS abrem tanto UDP quanto TCP na mesma porta

    Iniciando o server¶

    # Com config padrao
./wolfgang-server

# Com config customizado
./wolfgang-server -config /etc/wolfgang/config.yaml

# Primeiro operador (admin)
./wolfgang-server -init-admin

    Na primeira execucao com -init-admin, o server cria o operador admin e exibe as credenciais. Guarde-as em local seguro.
