    ETW Bypass¶

    O que é¶

    ETW (Event Tracing for Windows) é a infraestrutura de telemetria nativa do Windows. Praticamente todo componente do sistema -- kernel, CLR, PowerShell, networking -- emite eventos via ETW. EDRs dependem pesadamente dessas traces para detectar atividade maliciosa em tempo real.

    O ETW bypass do Wolfgang neutraliza a emissão de eventos patchando EtwEventWrite em ntdll.dll, fazendo com que a função retorne STATUS_SUCCESS sem emitir nenhum evento.

    Por que funciona¶

    A arquitetura ETW é baseada em três componentes:

    Provider ──emit──► Session ──consume──► Consumer
 (CLR, etc)         (ETW Logger)        (EDR, Defender, PerfMon)
    1. Providers registram-se via EventRegister e emitem eventos via EtwEventWrite
    2. Sessions (controladas via StartTrace) recebem eventos dos providers habilitados
    3. Consumers (EDRs, Event Viewer, PerfMon) processam os eventos

    O ponto crítico: todos os providers no processo chamam EtwEventWrite da cópia de ntdll.dll carregada no mesmo espaço de endereçamento. Patchando essa única função, toda a telemetria do processo é silenciada -- incluindo:

    • .NET CLR events (Assembly loading, JIT compilation)
    • PowerShell Script Block Logging
    • Windows Defender AMSI events
    • Thread creation/injection events
    • Networking events

    A função EtwEventWrite vive em ntdll.dll, que é mapeada em userland. Como o implant tem acesso read-write ao próprio espaço de memória, pode modificar a função diretamente.

    Implementação no Wolfgang¶

    Strategy: Memory Patching¶

    O patch sobrescreve o início de EtwEventWrite com instruções que retornam STATUS_SUCCESS (0) imediatamente:

    // xor eax, eax; ret — retorna 0 (STATUS_SUCCESS)
etwPatch := []byte{0x33, 0xC0, 0xC3}

original, err := e.patchFunc("ntdll.dll", "EtwEventWrite", etwPatch)

    O patch é compacto (3 bytes) e faz a função retornar sucesso imediatamente. Callers não detectam falha porque o NTSTATUS indica sucesso -- eles simplesmente acreditam que o evento foi emitido.

    Strategy: Hardware Breakpoints (HWBP)¶

    func ETWHandler(ei *ExceptionInfo) {
    ei.SetRax(0) // STATUS_SUCCESS
    ei.SkipFunction()
}

func InstallETWBypass(m *Manager) (int, error) {
    addr, _ := ResolveFunction("ntdll.dll", "EtwEventWrite")
    return m.SetBreakpoint(addr, ETWHandler)
}

    O handler HWBP seta RAX = 0 (STATUS_SUCCESS) e pula a função inteira via SkipFunction(), que ajusta RIP para o return address e incrementa RSP.

    Seleção de estratégia¶

    Idêntica ao AMSI bypass -- auto tenta HWBP primeiro:

    func (e *ETWBypass) applyBypass(strategy string) (module.ModuleResult, error) {
    switch strategy {
    case "auto":
        result, err := e.executeHWBP()
        if err != nil {
            return e.patch()
        }
        return result, nil
    }
}

    Note over ETW: Retorna sem executar
    nenhuma instrucao original

    Impacto na Telemetria¶

    A tabela abaixo mostra o que é silenciado quando EtwEventWrite é neutralizado:

    Provider ETW GUID Eventos Silenciados
    Microsoft-Windows-DotNETRuntime {E13C0D23-...} Assembly load, JIT, GC
    Microsoft-Windows-PowerShell {A0C1853B-...} Script block logging
    Microsoft-Antimalware-Scan-Interface {2A576B87-...} AMSI scan results
    Microsoft-Windows-Kernel-Process {22FB2CD6-...} Process/thread creation
    Microsoft-Windows-Threat-Intelligence {F4E1897C-...} Memory allocation, injection

    O bypass de ETW é particularmente poderoso porque atua como multiplicador: ele desabilita toda a telemetria baseada em ETW do processo de uma só vez, complementando o AMSI bypass que atua apenas na camada de scanning.

    Detecção¶

    Indicadores¶

    IOC Descrição
    Prologue de EtwEventWrite modificado Integrity check dos primeiros bytes
    VirtualProtect em ntdll.dll Tentativa de tornar código gravável
    Ausência de eventos ETW do processo Gap na telemetria (ausência como sinal)
    Debug registers apontando para EtwEventWrite Inspeção via GetThreadContext

    Contramedidas dos defensores¶

    • Kernel-mode ETW consumers: Operam fora do alcance do patch em userland (Microsoft-Windows-Threat-Intelligence provider roda no kernel)
    • Syscall-level monitoring: Hooks em SSDT/callbacks no kernel não são afetados
    • Periodic integrity checks: Scanning do prologue de funções críticas
    • ETW session monitoring: Detectar processos que param de emitir eventos

    Por que o bypass ainda é efetivo¶

    A maioria dos EDRs depende do usermode ETW como fonte primária de telemetria. Mesmo EDRs que têm componentes kernel-mode usam ETW extensivamente porque é a forma mais rica de obter contexto sobre operações .NET, PowerShell e comportamento de aplicações.
