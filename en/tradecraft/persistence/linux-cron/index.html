    Linux Cron¶

    O que é¶

    Cron é o agendador de tarefas padrão de sistemas Unix/Linux. Permite agendar comandos para execução periódica com granularidade de minutos. Para persistência, o Wolfgang adiciona uma entrada no crontab do usuário que re-executa o implant periodicamente.

    Cron é universal: está presente em praticamente toda distribuição Linux, de servidores bare-metal a containers Docker. É o equivalente Linux mais direto das Scheduled Tasks do Windows.

    Por que funciona¶

    O daemon crond roda como root e verifica periodicamente os crontabs de todos os usuários. Quando encontra uma entrada cujo schedule corresponde ao horário atual, executa o comando no contexto do usuário dono do crontab.

    Existem dois níveis de crontab:

    Tipo Localização Privilégio Escopo
    User crontab /var/spool/cron/crontabs/<user> User Apenas o próprio usuário
    System crontab /etc/crontab, /etc/cron.d/ Root Todos os usuários

    Variantes adicionais incluem diretórios de drop-in:

    • /etc/cron.daily/ -- Scripts executados diariamente
    • /etc/cron.hourly/ -- Scripts executados a cada hora
    • /etc/cron.weekly/ -- Scripts executados semanalmente
    • /etc/cron.monthly/ -- Scripts executados mensalmente

    Esses diretórios são processados pelo anacron, que garante execução mesmo se o sistema estava desligado no horário agendado (útil para laptops e servidores com manutenção).

    O Wolfgang usa user crontab por padrão, que não requer root.

    Implementação no Wolfgang¶

    O módulo (pkg/module/persist/cron_linux.go) adiciona uma entrada ao crontab existente do usuário de forma idempotente:

    func platformInstallCron(schedule, command string) error {
    // Lê crontab existente
    out, err := exec.Command("crontab", "-l").CombinedOutput()
    existing := ""
    if err == nil {
        existing = string(out)
    }
    // crontab -l retorna erro se não existe; trata como vazio

    entry := schedule + " " + command
    // Idempotência: verifica se a entrada já existe
    if strings.Contains(existing, entry) {
        return nil
    }

    // Adiciona nova entrada
    newCrontab := strings.TrimRight(existing, "\n")
    if newCrontab != "" {
        newCrontab += "\n"
    }
    newCrontab += entry + "\n"

    cmd := exec.Command("crontab", "-")
    cmd.Stdin = strings.NewReader(newCrontab)
    output, err := cmd.CombinedOutput()
    if err != nil {
        return fmt.Errorf("crontab install failed: %s: %w", string(output), err)
    }
    return nil
}

    Detalhes de implementação¶

    Idempotência: O módulo verifica se a entrada já existe antes de adicionar, prevenindo duplicatas em execuções repetidas. Isso é feito com strings.Contains no crontab existente.

    Preservação: O crontab existente é preservado -- a nova entrada é appendada, não substituída. Isso evita interferir com cron jobs legítimos do usuário.

    Pipe via stdin: O novo crontab é instalado via crontab - (leitura de stdin), que é o método padrão e portável entre distribuições.

    Formato cron¶

    O schedule segue o formato padrão de 5 campos:

    ┌───────────── minuto (0 - 59)
│ ┌───────────── hora (0 - 23)
│ │ ┌───────────── dia do mês (1 - 31)
│ │ │ ┌───────────── mês (1 - 12)
│ │ │ │ ┌───────────── dia da semana (0 - 7, 0 e 7 = domingo)
│ │ │ │ │
* * * * * comando

    Exemplos comuns para persistência:

    Schedule Significado
    */5 * * * * A cada 5 minutos
    0 * * * * A cada hora (no minuto 0)
    @reboot No boot do sistema
    0 9 * * 1-5 Às 09:00, dias úteis

    Considerações Operacionais¶

    Vantagens¶

    • Universal: Disponível em toda distribuição Linux
    • Sem root: User crontab funciona com qualquer conta
    • Idempotente: Múltiplas instalações não criam duplicatas
    • Persistente: Sobrevive reboots (o crontab é um arquivo, não memória)

    Desvantagens¶

    • Visível: crontab -l mostra todas as entradas imediatamente
    • Logging: Execuções ficam em /var/log/cron ou /var/log/syslog
    • Intervalo mínimo: 1 minuto (sem subsecond scheduling)
    • PATH limitado: Cron jobs rodam com PATH mínimo; usar caminhos absolutos

    User vs System crontab¶

    Aspecto User crontab System crontab
    Acesso Qualquer usuário Root
    Sobrevive a delete do user Não Sim
    Visível para admin Sim (crontab -l -u user) Sim (/etc/cron.d/)
    Formato 5 campos + comando 6 campos (inclui user) + comando

    Cleanup¶

    # Remover entrada específica
crontab -l | grep -v "implant_command" | crontab -

# Ou remover todo o crontab do user
crontab -r

    Detecção¶

    IOC Descrição
    Modificação de crontab (crontab -) Instalação de novo cron job
    Entries com caminhos em /tmp, /dev/shm Caminhos incomuns para cron
    Alta frequência (*/1 ou */5) Schedules agressivos para persistência
    Logs em /var/log/cron Execuções do job registradas
    auditd rule em /var/spool/cron/ Monitorar alterações nos crontabs

    Mitigações:

    • cron.allow / cron.deny: Restringir quais usuários podem usar cron
    • Auditd: Monitorar escritas em /var/spool/cron/crontabs/
    • Baseline de cron jobs: Alertar sobre novos jobs não documentados
