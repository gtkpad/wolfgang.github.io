    Plugins¶

    O sistema de plugins do Wolfgang permite estender o framework com novos listeners, modulos ou ambos, sem modificar o codigo fonte. Plugins sao binarios standalone que se comunicam com o teamserver via gRPC com mTLS automatico.

    Tipos de plugin¶

    Tipo Constante Descricao
    Listener listener Adiciona novos tipos de listener
    Module module Adiciona novos modulos para agents
    Full full Listener + Module combinados

    Configuracao¶

    No config.yaml:

    plugins:
  dir: "plugins"          # Diretorio onde os binarios de plugin estao
  verify_hash: true       # Verificar SHA256 antes de carregar

    Diretorio de plugins¶

    O diretorio padrao e plugins/ relativo ao working directory do server. Todos os arquivos (nao-diretorios) neste diretorio sao tratados como binarios de plugin.

    Verificacao de hash¶

    Quando verify_hash: true (padrao), apenas plugins cujo SHA-256 esta na whitelist sao carregados. Plugins com hash desconhecido sao rejeitados:

    [!] Plugin load failed: custom-listener: hash a3f2b1c8... not in whitelist

    Ciclo de vida¶

    flowchart TD
    SCAN[Manager.Scan] --> HASH{Calcular SHA-256}
    HASH --> VER{verify_hash?}
    VER -->|Sim| WL{Hash na whitelist?}
    VER -->|Nao| LOAD
    WL -->|Sim| LOAD[Iniciar processo plugin]
    WL -->|Nao| REJECT[Rejeitar]
    LOAD --> GRPC[Conectar via gRPC + AutoMTLS]
    GRPC --> INFO[Dispensar InfoProvider]
    INFO --> META[Obter metadata: nome, versao, tipo]
    META --> TYPE{Tipo?}
    TYPE -->|listener / full| DL[Dispensar Listener proxy]
    TYPE -->|module / full| DM[Dispensar Module proxy]
    DL --> REG[Registrar no Manager]
    DM --> REG
    REG --> AUDIT[Audit: plugin.load]
    REG --> READY[Plugin ativo]

    Descoberta¶

    No startup ou quando Reload() e chamado, o Manager.Scan() percorre o diretorio de plugins:

    1. Lista todas as entries do diretorio
    2. Ignora subdiretorios
    3. Para cada arquivo, tenta carregar como plugin
    4. Erros sao coletados mas nao impedem o carregamento dos demais
    func (m *Manager) Scan() []error

    Verificacao¶

    A verificacao de integridade usa SHA-256:

    func hashFile(path string) (string, error) {
    f, err := os.Open(path)
    h := sha256.New()
    io.Copy(h, f)
    return hex.EncodeToString(h.Sum(nil)), nil
}

    Configurando a whitelist¶

    A whitelist e configurada via Manager.SetWhitelist() com uma lista de hashes SHA-256 hex:

    manager.SetWhitelist([]string{
    "a3f2b1c8d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0",
    "b4c3d2e1f0a9b8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2a1b0c9d8e7f6a5b4",
})

    Para obter o hash de um plugin:

    sha256sum plugins/my-custom-listener

    Carregamento¶

    O carregamento usa hashicorp/go-plugin com protocolo gRPC e AutoMTLS:

    client := goplugin.NewClient(&goplugin.ClientConfig{
    HandshakeConfig:  pluginsdk.Handshake,
    Plugins:          pluginsdk.PluginMap,
    Cmd:              exec.Command(path),
    AllowedProtocols: []goplugin.Protocol{goplugin.ProtocolGRPC},
    AutoMTLS:         true,
})
    • Handshake -- verifica compatibilidade de protocolo entre server e plugin
    • AutoMTLS -- gera certificados efemeros automaticamente para criptografar a comunicacao
    • Processo isolado -- cada plugin roda como processo separado

    Dispensacao de interfaces¶

    Apos conexao, o server dispensa as interfaces necessarias:

    1. "info" (obrigatorio) -- InfoProvider.GetInfo() retorna metadata
    2. "listener" (se tipo listener/full) -- proxy para operacoes de listener
    3. "module" (se tipo module/full) -- proxy para operacoes de modulo

    Metadata do plugin¶

    type PluginInfo struct {
    Name          string     // Identificador unico
    Version       string     // Versao semantica
    PluginType    PluginType // "listener", "module", ou "full"
    Path          string     // Caminho do binario
    Hash          string     // SHA-256 do binario
    ListenerTypes []string   // Tipos de listener fornecidos
    ModuleNames   []string   // Nomes de modulos fornecidos
}

    Gerenciamento¶

    Listar plugins¶

    wolfgang>> plugins
NAME             VERSION  STATUS
custom-listener  1.0.0    running
extra-recon      2.1.0    running

    A API retorna informacoes de todos os plugins carregados via Manager.List().

    Reload¶

    wolfgang>> plugins reload

    O reload:

    1. Para todos os plugins em execucao (StopAll())
    2. Rescanneia o diretorio (Scan())
    3. Carrega todos os plugins novamente

    Parar plugins¶

    Na shutdown do server, StopAll() e chamado:

    1. Coleta nomes de todos os plugins ativos
    2. Para cada um, chama client.Kill() para terminar o processo
    3. Emite evento de audit plugin.unload

    Tipos de plugin: Listener¶

    Plugins de listener adicionam novos tipos de transport. Apos carregamento, o tipo do listener fica disponivel para criacao via API e CLI.

    wolfgang>> listeners create
Listener type (http/https/dns/ws/custom): custom

    O Manager.GetListenerPlugin(listenerType) retorna o proxy para operacoes:

    • ListenerTypes() -- lista todos os tipos fornecidos por plugins
    • GetListenerPlugin(type) -- busca o plugin que fornece o tipo

    Tipos de plugin: Module¶

    Plugins de modulo adicionam novos modulos que agents podem executar:

    • ModuleNames() -- lista todos os modulos fornecidos por plugins
    • GetModulePlugin(name) -- busca o plugin que fornece o modulo

    Auditoria¶

    Eventos de plugin sao registrados no audit log:

    Evento Detalhes
    plugin.load name, version, plugin_type, path, hash
    plugin.unload name, plugin_type

    Prevencao de duplicatas¶

    O Manager rejeita plugins com nome duplicado:

    [!] Plugin load failed: custom-listener: plugin "custom-listener" already loaded

    Tambem rejeita plugins com nome vazio ou tipo invalido.

    Tolerancia a falhas¶

    • Se um plugin entra em panic durante o carregamento, o defer recover() captura e retorna erro
    • Plugins que falham no carregamento nao afetam os demais
    • Se a conexao gRPC falhar, o client e morto (Kill()) e o erro reportado
    • Scan() retorna uma lista de todos os erros encontrados

    Desenvolvendo plugins¶

    Para criar um plugin, implemente as interfaces do pluginsdk:

    1. InfoProvider -- metadata obrigatoria
    2. Listener -- para plugins de listener
    3. ModuleProvider -- para plugins de modulo

    O plugin e compilado como binario Go standalone e colocado no diretorio de plugins.

    Consulte o Guia do Desenvolvedor para detalhes de implementacao.
