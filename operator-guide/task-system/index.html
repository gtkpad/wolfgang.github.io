    Sistema de Tasks¶

    Tasks sao a unidade fundamental de operacao no Wolfgang. Todo comando executado em um agent -- de shell a screenshot -- e uma task com ciclo de vida rastreavel.

    Maquina de estados¶

    stateDiagram-v2
    [*] --> pending : QueueTask()
    pending --> running : Agent faz check-in e recebe task
    pending --> expired : TTL excedido (sweep)
    pending --> cancelled : Cancel() pelo operador
    running --> completed : Agent retorna resultado com sucesso
    running --> failed : Agent retorna erro
    running --> cancelled : Cancel() + signal no proximo check-in
    completed --> [*]
    failed --> [*]
    expired --> [*]
    cancelled --> [*]

    Ciclo de vida¶

    1. Criacao (pending)¶

    O operador despacha uma task via CLI ou API:

    wolfgang [a1b2c3d4]>> exec whoami
[*] Task queued: f1e2d3c4 (module=shell.exec)

    A task e criada no banco com status pending e atribuida ao agent e operador:

    Campo Descricao
    id UUID gerado automaticamente
    agent_id Agent alvo
    operator_id Operador que criou a task
    module Modulo a executar
    args Argumentos JSON
    status pending
    created_at Timestamp de criacao

    2. Dispatch (running)¶

    No proximo check-in do agent, tasks pendentes sao retornadas como envelope criptografado:

    [
  {
    "task_id": "f1e2d3c4...",
    "module": "shell.exec",
    "args": "{\"command\":\"whoami\"}",
    "agent_id": "a1b2c3d4..."
  }
]

    Tasks sao retornadas em lote -- todas as pendentes de um agent sao enviadas de uma vez.

    3. Execucao¶

    O implant executa o modulo correspondente via Executor.Execute() e retorna o resultado como task response criptografada.

    4. Conclusao¶

    O resultado atualiza a task no banco:

    • status: completed ou failed
    • result: output do modulo (JSON)
    • completed_at: timestamp de finalizacao

    TTL e expiracao¶

    Tasks pendentes tem um TTL (Time-To-Live) configuravel. Um goroutine de sweep roda periodicamente verificando tasks expiradas.

    Parametro Padrao Descricao
    TTL 1 hora Tempo maximo que uma task pode ficar pendente
    SweepInterval 1 minuto Frequencia de verificacao de tasks expiradas

    Tasks que ultrapassam o TTL sao marcadas como expired com result "TTL expired".

    Quando uma task expira?¶

    • Agent esta offline (status lost)
    • Agent com sleep muito longo que excede o TTL
    • Rede bloqueada impedindo check-in

    Cancelamento¶

    Operadores podem cancelar tasks pendentes ou em execucao:

    POST /api/v1/tasks/{id}/cancel

    Task pendente: Marcada como cancelled imediatamente. Nao sera enviada ao agent.

    Task em execucao: Marcada como cancelled no banco e um cancel signal e enfileirado para o agent. No proximo check-in, o DrainCancelSignals() entrega os IDs das tasks canceladas.

    Cancel signals sao scoped por agent -- cada agent recebe apenas os seus. O mapa cancelSignals no TaskManager armazena agentID -> set(taskID).

    Visualizando tasks¶

    Listar tasks de um agent¶

    wolfgang [a1b2c3d4]>> tasks
ID        MODULE       STATUS     CREATED
f1e2d3c4  shell.exec   completed  2026-03-14 10:30:00
a2b3c4d5  recon.sys    pending    2026-03-14 10:31:00
b3c4d5e6  screenshot   failed     2026-03-14 10:29:00

    Detalhes de uma task¶

    GET /api/v1/tasks/{id}

    Retorna todos os campos incluindo result e completed_at.

    Resultado de tasks¶

    O client faz polling automatico quando o comando usa wait=true (maioria dos comandos). O polling verifica a cada 2 segundos com timeout de 5 minutos:

    [*] Task queued: f1e2d3c4 (module=shell.exec)
[+] Task completed:
nt authority\system

    Para tasks que retornam dados binarios (screenshot, LSASS dump, download), o client decodifica o base64 e salva em arquivo local.

    Formato do resultado¶

    Tasks retornam resultados no formato ModuleResult:

    {
  "output": "nt authority\\system",
  "data": {
    "key": "value"
  }
}
    • output -- resultado textual principal
    • data -- dados estruturados opcionais (ex: screenshot base64, dump base64)

    O client extrai e formata automaticamente:

    1. Tenta parsear output como JSON e pretty-print
    2. Senao, exibe como texto
    3. Para campos data com conteudo binario, decodifica e salva em arquivo

    Auditoria¶

    Cada operacao de task gera entradas de audit log:

    Evento Quando
    Task criada POST /api/v1/agents/{id}/tasks
    shell.end Resultado de task shell ou exec recebido
    Task completada Status muda para completed
    Task cancelada Operador chama cancel

    Logs podem ser consultados via logs no REPL.

    Tasks e transport¶

    O dispatch de tasks depende do transport do agent:

    Transport Mecanismo Latencia
    HTTPS Retornadas no check-in GET sleep interval
    DNS Retornadas como TXT records no check-in A sleep interval
    WebSocket Push imediato via PushTasks() ~instantaneo
    Named Pipe Push direto ~instantaneo

    Para WebSocket, tasks podem ser despachadas proativamente sem esperar check-in, tornando operacoes interativas (shell) mais responsivas.

    Fluxo completo¶

    sequenceDiagram
    participant Op as Operador
    participant API as Teamserver API
    participant DB as SQLite
    participant AM as Agent Manager
    participant Agent as Implant

    Op->>API: POST /agents/{id}/tasks
    API->>DB: CreateTask (status=pending)
    API->>Op: Task ID

    Agent->>AM: Check-in (GET)
    AM->>DB: GetPendingTasks
    DB->>AM: [task1, task2]
    AM->>Agent: Encrypted envelope

    Agent->>Agent: Execute module
    Agent->>AM: Task response (POST)
    AM->>DB: UpdateTaskStatus (completed)

    Note over Op: Polling via WebSocket ou REST
    Op->>API: GET /tasks/{id}
    API->>Op: Result
